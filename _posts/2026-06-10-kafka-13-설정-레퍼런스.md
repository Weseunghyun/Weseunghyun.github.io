지금까지 우리는 카프카를 "개념"으로 배웠다.
토픽이 뭔지, 파티션이 어떻게 나뉘는지, 복제는 왜 필요한지, "정확히 한 번"은 어떻게 가능한지.
그런데 실제로 카프카를 띄워서 운영하려고 `properties` 파일을 열면, 거기엔 수십 개의 알 수 없는 설정값이 기다리고 있다.
`acks=all`, `enable.idempotence=true`, `min.insync.replicas=1`, `retention.ms=604800000`… 이게 다 무슨 뜻이고, **기본값은 왜 하필 그 값**이며, 어떤 건 그냥 두면 안 되는지.

이 장은 그 설정값들을 한자리에 모아 정리하는 **레퍼런스 장**이다.
앞 장들에서 이미 의미를 다룬 설정은 표에 다시 담되 설명은 짧게 하고 해당 장으로 넘긴다.
대신 **"이 설정이 왜 중요한가", "기본값을 그대로 두면 무슨 일이 벌어지는가"**를 표 앞뒤에 붙여 외우기 좋게 만든다.
이 장의 모든 수치는 **Apache Kafka 4.1 공식 설정 문서 기준**이다.

> 한 가지 미리 짚자. 설정값엔 "정답"이 없다. 대부분은 **트레이드오프**다 — 한쪽을 좋게 하면 다른 쪽이 나빠진다. 그래서 이 장은 "이 값으로 바꿔라"가 아니라, **"이 값을 움직이면 무엇과 무엇이 교환되는지"**를 이해하는 것이 목표다.

## 13.1 설정은 네 군데에 흩어져 있다 — 누가 무엇을 정하나

먼저 큰 그림.
카프카 설정은 **누가 그 설정을 읽느냐**에 따라 네 묶음으로 나뉜다.
이걸 구분하지 못하면 "같은 이름의 설정이 왜 여러 군데 있지?" 하고 헷갈린다.

![](/assets/img/posts/kafka-13-설정-레퍼런스/Pasted-image-20260612213314.png)

- **Producer 설정**: 데이터를 *보내는* 클라이언트가 들고 있다.
  "몇 개 받아줘야 성공으로 칠까(`acks`)", "얼마나 모아서 한 번에 보낼까(`batch.size`)" 같은, 보내는 행위에 관한 것.
- **Consumer 설정**: 데이터를 *읽는* 클라이언트가 들고 있다.
  "한 번에 몇 건 가져올까(`max.poll.records`)", "어디서부터 읽을까(`auto.offset.reset`)" 같은 것.
- **Topic 설정**: 토픽 하나하나에 붙는다.
  "이 토픽의 데이터를 며칠 보관할까(`retention.ms`)", "복제 몇 개는 받아야 쓰기 성공으로 칠까(`min.insync.replicas`)" 같은 것.
  토픽별로 다르게 줄 수 있고, 안 주면 브로커 기본값을 따른다.
- **Broker 설정**: 브로커 프로세스 전체에 적용되는 바닥값.
  "자동으로 만들어지는 토픽은 복제를 몇 개 둘까(`default.replication.factor`)" 처럼 클러스터 전반의 기본 정책.

> 비유로 말하면(어디까지나 비유다) — Broker 설정은 **회사 전체의 기본 사규**, Topic 설정은 **부서별 세부 규정**(사규를 덮어쓸 수 있음), Producer/Consumer 설정은 **각 직원이 일하는 방식**이다.

여기서 자주 헷갈리는 한 가지: **같은 의미의 설정이 이름만 살짝 바꿔 여러 군데 존재**한다.
예를 들어 "메시지 한 건의 최대 크기"는 토픽엔 `max.message.bytes`, 브로커엔 `message.max.bytes`, 컨슈머엔 `max.partition.fetch.bytes`가 따로 있다.
큰 메시지를 보내려면 이 셋을 **함께** 올려줘야 한다는 게 핵심인데, 이건 뒤에서 다시 본다.

## 13.2 Producer 설정 — "성공"의 기준을 내가 정한다

프로듀서 설정의 절반은 **"무엇을 성공으로 칠 것인가"**에 관한 것이다.
데이터를 보냈을 때, 어디까지 도달해야 "보냈다"고 인정할지.
나머지 절반은 **"얼마나 모아서 보낼까"**라는 처리량 튜닝이다.

가장 중요한 셋부터 보자.

- **`acks`** — 쓰기 응답을 누가 줘야 성공으로 칠지.
  `all`이면 현재 ISR(동기화된 복제본) 전원이 받아야 성공이다.
  `1`이면 리더만, `0`이면 보내고 확인도 안 한다.
  기본값이 `all`이라는 건, 카프카가 **기본적으로 데이터 안전을 처리량보다 우선**한다는 신호다.
  (`acks` 개념은 3장, ISR과의 관계는 7장에서 다뤘다.)
- **`enable.idempotence`** — 멱등성.
  켜면 같은 메시지가 네트워크 재시도로 두 번 도착해도 카프카가 중복을 걸러낸다.
  기본값이 `true`인데, 이걸 켜면 카프카가 자동으로 `acks=all`·`retries>0`·`max.in.flight≤5`를 강제한다.
  즉 **기본 상태가 이미 "중복 없이 안전하게"**로 맞춰져 있다.
  (멱등성·정확히 한 번은 8장.)
- **`max.in.flight.requests.per.connection`** — 응답을 기다리지 않고 동시에 날려둘 수 있는 요청 수.
  기본 `5`.
  멱등성이 켜진 상태에선 **5 이하라야 순서 보장과 중복 제거가 성립**한다.

이 셋이 왜 묶여 있는지가 포인트다.
"재시도하면 중복이 생길 수 있다 → 그래서 멱등성으로 중복을 거른다 → 그러려면 동시 요청을 5개 이하로 묶어야 순서가 안 꼬인다." 기본값들이 이 사슬을 이미 안전하게 채워놓은 상태다.

### Producer 전체 표

| 설정 | 기본값 | 의미 / 어디서 다뤘나 |
|---|---|---|
| `acks` | **all** | 성공 기준. all = 현재 ISR 전원 (→ 3장, 7장) |
| `enable.idempotence` | **true** | 멱등성. 켜면 acks=all·retries>0·max.in.flight≤5 강제 (→ 8장) |
| `retries` | **2147483647** | 재시도 횟수. 사실상 무한 — 실질 상한은 `delivery.timeout.ms` |
| `max.in.flight.requests.per.connection` | **5** | 멱등성 시 ≤5라야 순서·중복제거 보장 |
| `batch.size` | **16384** (16KiB) | 한 배치에 모을 최대 크기 (→ 9장) |
| `linger.ms` | **5** | 배치를 채우려 기다리는 시간 (지연 ↔ 처리량) |
| `buffer.memory` | **33554432** (32MiB) | 프로듀서가 쓰는 버퍼 총량 |
| `compression.type` | **none** | 배치 압축. none·gzip·snappy·lz4·zstd |
| `max.request.size` | **1048576** (1MiB) | 요청 하나의 최대 크기 |
| `delivery.timeout.ms` | **120000** (2분) | send() 후 성공/실패 보고까지 **전체 상한**. ≥ request.timeout.ms + linger.ms |
| `request.timeout.ms` | **30000** (30초) | 단일 요청 응답을 기다리는 시간 |
| `retry.backoff.ms` | **100** | 재시도 간격. 지수적으로 늘어 최대 1000ms |
| `transaction.timeout.ms` | **60000** (1분) | 트랜잭션 타임아웃. 브로커 `transaction.max.timeout.ms`(15분) 이하여야 |
| `max.block.ms` | **60000** (1분) | send()/partitionsFor()가 블록될 최대 시간 |
| `metadata.max.age.ms` | **300000** (5분) | 메타데이터를 강제로 새로 받는 주기 |
| `partitioner.class` | **null** | null = 기본 파티셔너 (key 있으면 murmur2 해시 / 없으면 sticky) (→ 3장) |

표를 다 외울 필요는 없다.
대신 **`retries`가 무한대(2147483647)인데 왜 무한 재시도가 안 되는지**는 알아두면 좋다.
답은 `delivery.timeout.ms`다.
재시도 횟수가 아무리 많아도, send()를 호출한 시점부터 2분(`delivery.timeout.ms` 기본값)이 지나면 카프카는 "이건 실패"라고 보고한다.
즉 **실질적인 상한은 횟수가 아니라 시간**이다.
그래서 "재시도 몇 번?"이 아니라 "최대 몇 초까지 버틸까?"로 생각하는 게 맞다.

### 숫자로 보는 배치 트레이드오프

`batch.size`(16KiB)와 `linger.ms`(5ms)의 관계를 작은 시나리오로 보자.
![](/assets/img/posts/kafka-13-설정-레퍼런스/Pasted-image-20260612213348.png)

```
linger.ms = 0 인 경우 (기다리지 않음):
  메시지 1건 도착 → 즉시 전송 → 1건짜리 요청 → 네트워크 왕복 1번
  메시지 1000건이면 → 요청 1000번 (오버헤드 큼, 지연 최소)

linger.ms = 5 인 경우 (기본, 최대 5ms 기다림):
  5ms 동안 도착한 메시지들을 batch.size(16KiB)까지 모아 1번에 전송
  메시지 1000건이 5ms 안에 모이면 → 요청 몇 번 (오버헤드 작음, 지연 +최대 5ms)
```

`linger.ms`를 올리면 한 번에 많이 보내 **처리량이 오르지만**, 그만큼 메시지가 버퍼에서 기다리니 **지연이 늘어난다.** 정답이 없는 전형적인 트레이드오프다.
기본값 5ms는 "거의 안 기다리지만 아주 약간은 모아본다"는 보수적 절충이다.

## 13.3 Consumer 설정 — "어디까지 읽었나"와 "얼마나 가져올까"

컨슈머 설정도 두 갈래다.
하나는 **오프셋 관리**(어디까지 읽었는지 기록·복구), 다른 하나는 **그룹 멤버십**(컨슈머 그룹 안에서 살아있다고 인정받는 법).
둘 다 앞 장(4장·5장)에서 깊이 다뤘으니 여기선 표 중심으로 정리하고 함정만 짚는다.

오프셋 쪽에서 가장 조심할 기본값은 **`enable.auto.commit=true`**다.
켜져 있으면 컨슈머가 `auto.commit.interval.ms`(기본 5초)마다 "여기까지 읽었다"를 자동으로 기록한다.
편하지만 위험한 면이 있다 — *읽기만 하고 아직 처리(저장·계산)는 안 끝났는데* 오프셋이 먼저 커밋되면, 그 사이 컨슈머가 죽었을 때 처리 못 한 메시지를 "이미 읽었다"고 건너뛸 수 있다.
그래서 정확한 처리가 필요하면 자동 커밋을 끄고 직접 커밋한다(5장).

또 하나, **`auto.offset.reset=latest`**의 함정.
이 설정은 **커밋된 오프셋이 아예 없을 때만** 작동한다.
새 컨슈머 그룹이 처음 붙었을 때, `latest`면 **"지금부터 들어오는 것"만** 읽고 이전 데이터는 통째로 건너뛴다.
"왜 과거 데이터를 안 읽지?" 하는 흔한 당황의 원인이 이것이다.
처음부터 다 읽고 싶으면 `earliest`로 둬야 한다.

### Consumer 전체 표

| 설정 | 기본값 | 의미 / 어디서 다뤘나 |
|---|---|---|
| `enable.auto.commit` | **true** | 오프셋 자동 커밋 (→ 5장) |
| `auto.commit.interval.ms` | **5000** (5초) | 자동 커밋 주기. poll 시점에 확인 |
| `auto.offset.reset` | **latest** | 커밋 없을 때만 적용. latest·earliest·none |
| `session.timeout.ms` | **45000** (45초) | (classic 그룹) 하트비트 끊기면 추방까지 (→ 4장) |
| `heartbeat.interval.ms` | **3000** (3초) | (classic) 하트비트 주기. session의 1/3 이하 권장 |
| `max.poll.interval.ms` | **300000** (5분) | poll 사이 최대 간격. 처리가 느려 넘기면 리밸런스 |
| `partition.assignment.strategy` | **Range, CooperativeSticky** | 파티션 배정 전략 (→ 4장) |
| `group.protocol` | **classic** | consumer = KIP-848 새 프로토콜 |
| `group.remote.assignor` | **null** | (KIP-848) 서버 측 배정자 선택 |
| `group.instance.id` | **null** | static membership(KIP-345). 재시작해도 같은 멤버 |
| `fetch.min.bytes` | **1** | 이만큼 쌓일 때까지 기다렸다 응답 (→ 4장) |
| `fetch.max.bytes` | **52428800** (50MiB) | fetch 응답 크기 목표. 절대 상한은 아님 |
| `fetch.max.wait.ms` | **500** | fetch.min.bytes 못 채워도 이 시간엔 응답 |
| `max.partition.fetch.bytes` | **1048576** (1MiB) | 파티션 하나에서 한 번에 가져올 양 |
| `max.poll.records` | **500** | 한 번 poll에 반환할 레코드 수 |
| `isolation.level` | **read_uncommitted** | read_committed면 커밋된 트랜잭션만 (→ 8장) |
| `client.rack` | **""** | (KIP-392) 같은 랙의 follower에서 읽기 |
| `receive.buffer.bytes` | **65536** (64KiB) | TCP 수신 버퍼. -1 = OS 기본 |
| `check.crcs` | **true** | 레코드 CRC 검증 |

여기서 짚을 미묘한 점 하나.
**`fetch.max.bytes`(50MiB)는 "절대 상한"이 아니다.** 공식 문서가 분명히 해두는데, 한 건의 메시지가 이 값보다 크더라도 컨슈머가 멈추지 않도록 **메시지 하나는 통째로 반환**된다.
즉 이 값은 "대략 이 정도까지만 가져오자"는 *목표치*에 가깝지, 넘으면 자르는 칼날이 아니다.
비슷하게 `max.poll.records`(500)는 **반환 개수만** 제한할 뿐, 네트워크로 가져오는 양 자체를 줄이지는 않는다 — 가져오는 양은 위의 `fetch.*` 설정들이 정한다.

또 하나, **`isolation.level`의 기본값이 `read_uncommitted`**라는 점은 트랜잭션을 쓸 때 꼭 기억해야 한다.
프로듀서가 트랜잭션으로 "정확히 한 번"을 보장해도, 컨슈머가 기본값 그대로면 **아직 커밋되지 않은(중단될 수도 있는) 메시지까지 읽는다.** 트랜잭션의 격리를 받으려면 컨슈머를 `read_committed`로 바꿔줘야 한다(8장).

## 13.4 Topic 설정 — 보관 정책과 내구성의 핵심

토픽 설정은 **"이 데이터를 얼마나, 어떻게 보관할 것인가"**가 중심이다.
카프카가 단순한 메시지 전달이 아니라 **로그(append-only 기록)**라는 정체성을 가장 잘 드러내는 영역이기도 하다.

핵심 축은 둘이다.

**축 1 — 얼마나 오래 보관하나(retention).**
- `retention.ms`의 기본값은 **604800000밀리초 = 정확히 7일**이다.
  즉 카프카는 기본적으로 **데이터를 1주일간 보관**한다.
  이 시간이 지난 메시지는 삭제 대상이 된다.
  `-1`로 주면 시간 기준 무제한.
- `retention.bytes`는 크기 기준 보관.
  기본 `-1`(무제한).
  시간과 크기 둘 다 설정하면 **둘 중 먼저 도달하는 쪽**에서 삭제가 일어난다.

**축 2 — 어떻게 정리하나(cleanup.policy).**
- `cleanup.policy`의 기본값은 **`delete`** — 오래된 것을 시간/크기 기준으로 지운다.
- `compact`로 주면 다르다.
  같은 키의 옛 값을 지우고 **키별 최신 값만 남긴다.** "현재 상태"를 담는 토픽(예: 사용자별 최신 설정)에 쓴다.
  compaction 관련 설정들(`min.cleanable.dirty.ratio`, `delete.retention.ms` 등)이 이 모드에서 의미를 가진다.

여기에 **내구성의 핵심 설정** 하나가 토픽 레벨에 있다 — `min.insync.replicas`.

### `min.insync.replicas`와 `acks=all`의 짝 — 이게 진짜 중요하다

`min.insync.replicas`(약칭 min ISR)는 **"쓰기를 성공으로 치려면 최소 몇 개의 복제본이 동기화돼 있어야 하는가"**다.
기본값은 **`1`**.

이게 왜 중요하냐면, **`acks=all`과 짝을 이뤄야 진짜 내구성이 나오기 때문**이다.

![](/assets/img/posts/kafka-13-설정-레퍼런스/Pasted-image-20260612213459.png)

핵심은 이것이다: **기본값 `min ISR=1`은 `acks=all`을 무력화한다.** "전원이 받아야 성공"인데 전원이 1명(리더)이면 결국 리더만 받으면 끝이니까.
그래서 운영 환경에서 데이터 안전이 중요하면 **`min.insync.replicas=2`로 올리는 것을 권장**한다(복제본은 3개로 두고).
이 둘을 함께 이해하는 것이 카프카 내구성의 핵심이다(개념은 7장에서 다뤘다).

### Topic 전체 표

| 설정 | 기본값 | 의미 / 어디서 다뤘나 |
|---|---|---|
| `cleanup.policy` | **delete** | delete = 오래된 것 삭제 / compact = 키별 최신만 (→ 6장) |
| `retention.ms` | **604800000** (7일) | 시간 기준 보관. -1 = 무제한 |
| `retention.bytes` | **-1** | 크기 기준 보관. -1 = 무제한 |
| `segment.bytes` | **1073741824** (1GiB) | 로그 세그먼트 파일 크기 (→ 6장) |
| `segment.ms` | **604800000** (7일) | 시간 기준 세그먼트 롤링 |
| `segment.index.bytes` | **10485760** (10MiB) | 오프셋 인덱스 최대 크기 |
| `index.interval.bytes` | **4096** (4KiB) | sparse 인덱스 엔트리 간격 (→ 6장) |
| `min.insync.replicas` | **1** | acks=all과 짝. 내구성 위해 **2 권장** (→ 7장) |
| `max.message.bytes` | **1048588** (≈1MiB) | 메시지 한 건 최대 크기 |
| `message.timestamp.type` | **CreateTime** | CreateTime = 생성 시각 / LogAppendTime = 적재 시각 |
| `compression.type` | **producer** | producer = 프로듀서가 압축한 원본 그대로 보관 |
| `min.cleanable.dirty.ratio` | **0.5** | (compact) 이 비율 넘으면 compaction 시작 |
| `delete.retention.ms` | **86400000** (1일) | (compact) tombstone(삭제 표식) 보존 시간 |
| `min.compaction.lag.ms` | **0** | compaction까지 최소 대기 |
| `max.compaction.lag.ms` | **9223372036854775807** | Long.MAX = 사실상 무한 |
| `flush.messages` / `flush.ms` | **9223372036854775807** | Long.MAX = fsync 강제 안 함 (OS·복제에 의존) |
| `unclean.leader.election.enable` | **false** | 뒤처진 복제본을 리더로 승격 금지 (→ 7장) |

여기서 의외로 중요한 기본값이 **`flush.messages` / `flush.ms`**다.
둘 다 Long.MAX(사실상 무한)로 설정돼 있다.
무슨 뜻이냐면, **카프카는 메시지가 들어올 때마다 디스크에 강제로 fsync(즉시 영구 기록)하지 않는다.** 대신 OS의 페이지 캐시에 맡겨뒀다가 OS가 알아서 디스크로 내려보내게 한다.
"그럼 안전하지 않은 거 아닌가?" 싶지만, 카프카는 **디스크 fsync 대신 복제(replication)로 안전을 보장**하는 설계다 — 여러 복제본에 동시에 있으면 한 노드의 디스크가 늦게 써져도 데이터가 살아있다.
카프카가 빠른 이유의 한 축이 바로 이것이다(9장).

또 `unclean.leader.election.enable=false`라는 기본값도 의미가 깊다.
"unclean(불결한) 리더 선출"이란 **ISR에 없던, 즉 최신 데이터를 못 따라잡은 뒤처진 복제본을 억지로 리더로 올리는 것**이다.
기본값이 `false`라는 건, 카프카가 **"데이터를 잃느니 차라리 잠깐 쓰기를 멈추겠다"**는 쪽을 택했다는 뜻이다(가용성보다 일관성 우선).
이걸 `true`로 바꾸면 가용성은 오르지만 데이터 유실 위험이 생긴다(7장).

## 13.5 Broker 설정 — 그냥 두면 위험한 기본값들

브로커 설정 중 가장 많이 회자되는 건 **"기본값을 그대로 운영에 쓰면 안 되는 것들"**이다.
카프카의 브로커 기본값 일부는 **단일 노드 개발 환경 기준**으로 맞춰져 있어서, 운영 클러스터에선 명시적으로 바꿔줘야 한다.

가장 조심할 둘:

- **`default.replication.factor=1`** — 자동으로 만들어지는 토픽의 복제본 수가 기본 **1개**다.
  즉 복제 없이 단 하나의 복제본만 둔다.
  그 브로커가 죽으면 그 토픽 데이터는 사라진다.
  운영에선 보통 **3**으로 둔다.
- **`num.partitions=1`** — 자동 생성 토픽의 파티션이 기본 **1개**다.
  파티션이 1개면 그 토픽은 컨슈머를 아무리 늘려도 동시에 하나만 읽을 수 있다(4장에서 본 "파티션 수가 병렬성의 상한").
  처리량을 미리 고려해 명시 설정이 필요하다.

이 둘이 합쳐지면 무서운 시나리오가 된다 — **누군가 토픽 이름을 오타 내서 메시지를 보냈는데**, 자동 생성이 켜져 있으면 그 오타 토픽이 **복제 1개·파티션 1개**로 조용히 생겨버린다.
그래서 운영에선 자동 생성 자체를 끄거나, 최소한 이 기본값들을 안전하게 올려둔다.

또 하나 알아둘 브로커 설정은 **`replica.lag.time.max.ms=30000`(30초)**다.
이건 ISR 멤버십을 정하는 시계다 — **follower(복제본)가 30초 안에 리더를 따라잡지 못하면 ISR에서 제거**된다.
7장에서 본 "동기화된 복제본 집합"의 경계를 정하는 게 바로 이 값이다.
너무 짧으면 잠깐 느려진 복제본도 자꾸 빠지고, 너무 길면 한참 뒤처진 복제본을 계속 "동기화됨"으로 인정해 위험해진다.

### Broker 전체 표

| 설정 | 기본값 | 의미 / 어디서 다뤘나 |
|---|---|---|
| `default.replication.factor` | **1** | 🔴 자동생성 토픽 복제본 수. 운영 시 **3 권장** |
| `num.partitions` | **1** | 자동생성 토픽 파티션 수 (→ 4장) |
| `min.insync.replicas` | **1** | 클러스터 기본 min ISR. 내구성 위해 **2 권장** |
| `replica.lag.time.max.ms` | **30000** (30초) | 🔑 이 시간 내 못 따라잡은 follower는 ISR에서 제거 (→ 7장) |
| `unclean.leader.election.enable` | **false** | 뒤처진 복제본 리더 승격 금지 (가용성↔일관성) |
| `auto.leader.rebalance.enable` | **true** | preferred 리더 자동 복구 (→ 7장) |
| `leader.imbalance.check.interval.seconds` | **300** (5분) | 리더 쏠림 점검 주기 |
| `offsets.topic.replication.factor` | **3** | `__consumer_offsets` 토픽 복제본 수 |
| `offsets.topic.num.partitions` | **50** | `__consumer_offsets` 파티션 수 |
| `offsets.retention.minutes` | **10080** (7일) | 커밋된 오프셋 만료 시간 |
| `transaction.state.log.replication.factor` | **3** | `__transaction_state` 토픽 복제본 수 |
| `transaction.state.log.min.isr` | **2** | 트랜잭션 상태 로그 min ISR |
| `message.max.bytes` | **1048588** (≈1MiB) | 브로커 레벨 메시지 최대 크기 |
| `replica.fetch.max.bytes` | **1048576** (1MiB) | follower가 파티션당 fetch하는 최대 |
| `log.segment.bytes` | **1073741824** (1GiB) | 세그먼트 파일 크기 (브로커 기본) |
| `log.retention.ms` | **null** | null이면 hours/minutes 설정을 사용 |
| `num.replica.fetchers` | **1** | follower fetch 스레드 수 |

흥미로운 대비가 보인다.
사용자가 만드는 일반 토픽의 기본 복제본은 1인데, **카프카가 내부적으로 쓰는 토픽**(`__consumer_offsets`, `__transaction_state`)의 복제본 기본값은 **3**이다.
컨슈머가 "어디까지 읽었나"를 저장하는 `__consumer_offsets`, 트랜잭션 상태를 저장하는 `__transaction_state`는 잃으면 큰일이니, 카프카 스스로는 기본부터 3중 복제로 안전하게 둔다.
**"중요한 데이터는 복제를 많이"**라는 원칙을 카프카가 자기 내부 토픽으로 직접 보여주는 셈이다.

## 13.6 큰 메시지를 보내려면 — 여러 설정을 함께 올려야 한다

13.1에서 예고한 "같은 의미의 설정이 여러 군데"의 대표 사례를 정리하자.
기본보다 큰 메시지(예: 5MiB)를 카프카로 흘려보내려면, 데이터가 지나가는 **모든 길목의 한도**를 함께 올려야 한다.
한 군데라도 막히면 거기서 거부된다.

![](/assets/img/posts/kafka-13-설정-레퍼런스/Pasted-image-20260612213517.png)

각 한도의 역할:
- **`max.request.size`**(프로듀서, 기본 1MiB): 프로듀서가 한 요청에 담을 수 있는 크기.
  작으면 보내는 단계에서 막힌다.
- **`message.max.bytes`**(브로커) / **`max.message.bytes`**(토픽, 기본 ≈1MiB): 브로커/토픽이 받아 저장할 수 있는 한 건의 크기.
- **`replica.fetch.max.bytes`**(브로커, 기본 1MiB): follower가 리더에서 복제해 올 때의 한도.
  이게 작으면 **큰 메시지가 복제되지 못해** ISR이 무너질 수 있다.
- **`max.partition.fetch.bytes`**(컨슈머, 기본 1MiB): 컨슈머가 읽어갈 때의 파티션당 한도.

이 사례가 보여주는 교훈은 명확하다 — 카프카에서 설정 하나만 바꿔서 해결되는 일은 드물다.
**데이터가 흐르는 경로 전체를 머릿속에 그리고, 그 경로의 모든 관문을 함께 손봐야** 한다.
(참고로 큰 메시지를 자주 보낸다면 애초에 원본은 외부 스토리지에 두고 카프카엔 참조만 보내는 패턴도 있지만, 그건 이 노트의 범위 밖이다.)
