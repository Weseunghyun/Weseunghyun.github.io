1장에서 쿠버네티스가 "원하는 상태를 유지하는 선언형 시스템"이라는 걸 봤다.
그럼 이제 묻자.
**그 일을 실제로 해내는 부품들은 무엇이고, 어떻게 협력하는가?**

쿠버네티스 한 덩어리를 **클러스터(cluster)** 라고 부른다.
클러스터는 여러 대의 서버 묶음이다.
이 장에서 그 안을 해부한다.

처음엔 부품 이름이 많아 어지러울 수 있다.
하지만 큰 그림은 단순하다 — **두뇌 한 묶음과 일꾼 여러 명.**

## 2.1 큰 그림 — 두뇌(control plane)와 일꾼(node)

클러스터는 두 종류의 서버로 나뉜다.

| 구분 | 이름 | 역할 |
|---|---|---|
| 두뇌 | **control plane**(컨트롤 플레인) | 결정한다 — 무엇을 어디에 띄울지, 상태를 어떻게 맞출지 |
| 일꾼 | **node**(노드) | 실행한다 — 실제로 컨테이너(Pod)가 도는 서버 |

비유하면 식당이다.
control plane은 **주방장과 주문 관리 시스템**이다 (무엇을 만들지 결정·지휘).
node는 **실제 조리대**다 (음식이 실제로 만들어지는 곳).
주방장이 직접 모든 요리를 하지 않듯, control plane도 컨테이너를 직접 띄우지 않는다 — node에게 시킨다.

![](/assets/img/posts/kubernetes-02-클러스터-해부/k8s-cluster-architecture.svg)

이제 각각의 안을 들여다보자.

## 2.2 control plane — 클러스터의 두뇌

control plane 안에는 네 개의 핵심 부품이 있다.

### ① kube-apiserver — 모든 것의 단일 관문

클러스터의 **모든 소통은 이 API 서버를 거친다.**
우리가 `kubectl`로 명령을 보낼 때도, 다른 부품들이 서로 이야기할 때도, 전부 API 서버하고만 대화한다.

비유하면 회사의 **단일 접수처**다.
어떤 요청이든 반드시 접수처를 통해야 하고, 거기서 신원 확인(인증)과 검토(검증)를 거친다.
모든 길목이 한 곳이라 보안·감사를 한 군데서 통제할 수 있다.

### ② etcd — 클러스터의 기억

**etcd**는 클러스터의 **모든 상태를 저장하는 데이터베이스**다.
어떤 앱이 있고, 몇 개여야 하고, 어떤 설정인지 — 1장에서 말한 "원하는 상태"가 전부 여기 저장된다.

비유하면 회사의 **마스터 장부**다.
"우리 회사의 모든 사실"이 여기 적혀 있다.

🔴 중요한 두 가지.
- etcd에 **직접 접근하는 건 API 서버뿐**이다.
  다른 부품은 etcd를 직접 안 본다 (모든 길은 접수처를 통한다).
- **etcd가 곧 클러스터의 기억**이다.
  이게 깨지면 클러스터는 "자기가 무엇을 해야 하는지"를 잃는다.
  그래서 etcd 백업이 재해 복구의 핵심이다(8장).

(etcd는 여러 대로 복제해 안정성을 높이는데, 이때 "과반의 동의"로 데이터를 정하는 합의 알고리즘을 쓴다.
홀수 대로 구성하는 이유다.)

### ③ kube-scheduler — 어디에 둘지 정하는 자

새 Pod(컨테이너 묶음)가 생기면, **어느 node에 둘지**를 결정하는 게 스케줄러다.
결정만 하고, 실제 실행은 안 한다 (그건 node의 일).

어떻게 정할까?
두 단계다.
1. **필터링**: 가능한 node를 추린다 (자원이 충분한가?
   특별한 제약을 통과하는가?).
2. **스코어링**: 추려진 것 중 가장 좋은 node를 고른다 (가장 여유 있는 곳 등).

비유하면 호텔 **객실 배정 담당**이다.
손님(Pod)이 오면, 빈 방 중에서(필터링) 가장 적합한 방(스코어링)을 골라 배정한다.

### ④ controller-manager — 조정 루프가 사는 곳

1장에서 본 그 reconciliation 루프, 즉 "현재를 원하는 상태로 맞추는" 컨트롤러들이 모여 있는 곳이다.

예를 들어 "Pod 3개 유지" 컨트롤러는 끊임없이 현재 개수를 세고, 부족하면 새로 만든다.
node가 죽었는지 감시하는 컨트롤러도 여기 있다.

🔴 **이 조정 루프가 쿠버네티스의 심장**이다.
1장에서 본 자가치유가 전부 여기서 나온다.

## 2.3 node — 실제로 일하는 서버

이제 일꾼 쪽이다.
실제 컨테이너(Pod)는 node에서 돈다.
각 node에는 세 가지가 있다.

### ① kubelet — node의 대리인

각 node에서 도는 에이전트다.
API 서버로부터 "이 node에 이 Pod를 띄워라"를 받아, 실제로 컨테이너를 띄우고 상태를 보고한다.

비유하면 각 지점의 **현장 매니저**다.
본사(control plane)의 지시를 받아 현장에서 실제로 실행하고, 현장 상황을 본사에 보고한다.

### ② 컨테이너 런타임 — 진짜로 컨테이너를 띄우는 자

kubelet은 컨테이너를 직접 만들지 않는다.
**컨테이너 런타임**(주로 containerd)에게 시킨다.
이 런타임이 그 아래 runc를 통해 namespace·cgroup을 만들어 컨테이너를 띄운다 (Docker 시리즈 3장 그대로).

🔴 여기서 1장의 그림이 완성된다.
**쿠버네티스의 맨 밑바닥은 결국 Docker 시리즈에서 본 "리눅스 컨테이너 그 자체"** 다.
K8s는 그 위에 얹힌 오케스트레이션 층일 뿐이다.

(kubelet과 런타임은 **CRI**라는 표준 규격으로 대화한다.
이 표준 덕에 containerd든 다른 런타임이든 갈아끼울 수 있다.
"쿠버네티스가 도커를 버렸다"는 말의 진짜 의미가 바로 이것 — dockerd라는 중간층을 빼고 containerd를 직접 쓴다는 뜻이다.)

### ③ kube-proxy — 트래픽을 나누는 자

각 node에서 네트워크 트래픽을 적절한 Pod로 보내는 규칙을 관리한다.
이건 4장(Service와 네트워킹)에서 본격적으로 다룬다.

## 2.4 다 합쳐서 — `kubectl apply` 한 번의 여정

부품을 다 봤으니, 우리가 매니페스트를 제출했을 때 무슨 일이 벌어지는지 따라가 보자.

1. 내가 `kubectl apply -f web.yaml`로 "web 앱 3개"를 제출한다.
2. **API 서버**가 받아 신원·내용을 검증하고, **etcd**에 "원하는 상태: web 3개"를 저장한다.
3. **컨트롤러**가 본다 — "web이 3개여야 하는데 0개네." Pod 3개를 만들라고 한다.
4. **스케줄러**가 그 3개의 Pod를 어느 node에 둘지 정한다.
5. 해당 node의 **kubelet**이 자기에게 배정된 Pod를 보고, **컨테이너 런타임**에게 시켜 실제로 띄운다.
6. 이후로도 **컨트롤러**가 끊임없이 감시한다 — 하나가 죽으면 4~5번을 반복해 다시 3개로.

![](/assets/img/posts/kubernetes-02-클러스터-해부/k8s-request-journey.svg)

이 흐름 하나에 쿠버네티스의 모든 핵심이 들어 있다.
선언(2) → 조정(3) → 배치(4) → 실행(5) → 유지(6).

## 2.5 직접 다 운영해야 하나? — 관리형 쿠버네티스

여기까지 보면 느낄 것이다.
이 control plane을 직접 설치하고 안정적으로 운영하는 건 **만만치 않다.**
etcd 백업, API 서버 이중화, 버전 업그레이드…

그래서 실무 대부분은 **관리형 쿠버네티스**를 쓴다.
- AWS의 **EKS**, 구글의 **GKE**, 애저의 **AKS**.
- 이들은 **control plane을 클라우드가 대신 운영**해준다.
  사용자는 node와 그 위의 앱에만 집중한다.

처음 배울 땐 control plane 구조를 알아야 하지만(지금 배운 것), 실제 운영에선 그 복잡함의 상당 부분을 클라우드가 가져간다.

## 2.6 정리

이 장을 한 문장으로 줄이면 이렇다.

> 클러스터는 **두뇌(control plane)** 와 **일꾼(node)** 으로 나뉜다 — 두뇌의 **API 서버**가 단일 관문, **etcd**가 모든 상태의 기억, **스케줄러**가 배치 결정, **컨트롤러**가 조정 루프이고, 일꾼의 **kubelet**이 런타임을 시켜 실제 컨테이너를 띄운다.

이제 클러스터의 뼈대를 안다.
다음 장에서는 그 위에서 우리가 실제로 다루는 대상 — **Pod와 워크로드**로 들어간다.
"배포의 단위는 컨테이너가 아니다"라는 의외의 사실에서 시작한다.
