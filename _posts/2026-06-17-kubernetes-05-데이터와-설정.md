3장에서 Pod는 일회용이라고 했다.
죽으면 새 Pod가 새로 뜬다.
여기서 자연스러운 걱정이 생긴다.
**그럼 그 안에 쌓인 데이터는?** 데이터베이스 Pod가 죽으면 데이터도 같이 사라지나?

또 하나.
같은 이미지를 개발·운영에서 똑같이 쓴다면(Docker 시리즈), **환경마다 다른 설정(접속 주소, 비밀번호)은 어떻게 넣어주지?**

이 장은 이 두 질문에 답한다 — **데이터 영속성**과 **설정 주입**.

## 5.1 문제 — 일회용 Pod와 사라지는 데이터

Docker 시리즈 7장에서 본 그 문제가, 쿠버네티스에선 더 커진다.
컨테이너의 쓰기 공간은 컨테이너와 함께 사라진다.
Pod도 마찬가지 — Pod가 죽으면 그 안 데이터도 사라진다.

웹 서버라면 괜찮다 (상태가 없으니까).
하지만 데이터베이스라면 재앙이다.

그래서 데이터는 Pod **바깥**, Pod가 죽어도 살아남는 곳에 둬야 한다.
쿠버네티스는 이걸 위해 영리한 분리를 한다.

## 5.2 🔴 PVC와 PV — "신청서"와 "실제 창고"의 분리

쿠버네티스는 저장소를 **쓰는 쪽**과 **주는 쪽**으로 나눈다.

| 개념 | 역할 | 비유 |
|---|---|---|
| **PVC**(PersistentVolumeClaim) | "이만큼, 이런 방식으로 저장소를 달라"는 **요청** | 창고 임대 **신청서** |
| **PV**(PersistentVolume) | 실제 저장 공간 | 실제 **창고** |
| **StorageClass** | 신청서가 오면 창고를 **자동 발급**하는 틀 | 창고 자동 발급기 |

왜 이렇게 나눴을까?
앱(Pod)을 만드는 사람은 "10GB짜리 저장소가 필요해"라고 **신청(PVC)** 만 하면 된다.
그게 실제로 AWS의 디스크인지, 구글의 디스크인지, 사내 스토리지인지는 **몰라도 된다.**

**StorageClass**가 있으면, 신청(PVC)이 들어올 때 거기에 맞는 실제 저장소(PV)를 **자동으로 만들어준다**(동적 프로비저닝).

이 분리의 힘은 **이식성**이다.
같은 앱 매니페스트가 AWS에서도, 구글에서도, 사내 서버에서도 그대로 돈다.
저장소의 실제 구현 차이는 환경별 StorageClass가 흡수하니까.

![](/assets/img/posts/kubernetes-05-데이터와-설정/k8s-pvc-pv-storageclass.svg)

(클라우드·벤더의 실제 스토리지를 쿠버네티스에 연결하는 표준 규격을 **CSI**라고 한다.
2장의 CRI, 4장의 CNI와 같은 "인터페이스로 분리" 패턴이다 — 쿠버네티스는 이 패턴을 곳곳에서 반복한다.)

## 5.3 상태 앱과의 연결 — StatefulSet + PVC

3장에서 StatefulSet이 "각 Pod에 전용 저장소를 보장한다"고 했는데, 그 전용 저장소가 바로 이 PVC다.

StatefulSet은 Pod마다 자동으로 PVC를 만들어준다.
그래서 `db-0` Pod가 죽고 새로 떠도, **항상 자기 PVC(자기 데이터)에 다시 연결**된다.
"2번 DB는 2번의 데이터를 그대로 이어받는다"는 3장의 요구가 이렇게 실현된다.

🔴 다시 한번 — 그래서 상태 앱(DB)을 K8s에 올리는 건 StatefulSet + PVC가 한 묶음이다.
그리고 이게 까다롭기 때문에(데이터 안전·백업·복구), 실무에선 DB를 관리형 서비스에 맡기는 경우도 많다(3장).

## 5.4 설정은 이미지에서 분리한다 — ConfigMap

이제 두 번째 질문, 설정이다.

Docker 시리즈에서 강조했듯, 이미지는 **어느 환경에서나 똑같아야** 한다.
그런데 접속 주소나 기능 플래그 같은 건 환경마다 다르다 (개발 DB 주소 ≠ 운영 DB 주소).

그래서 이런 설정을 **이미지 안에 박지 않고**, 바깥에서 주입한다.
그 그릇이 **ConfigMap**이다.

ConfigMap에 설정을 담아두고, Pod에 두 가지 방식으로 넣어준다.
- **환경 변수**로 주입하거나,
- **파일**로 마운트해서 컨테이너 안에 보이게.

이렇게 하면 똑같은 이미지에 환경별 ConfigMap만 갈아끼워, 개발·운영을 같은 이미지로 돌릴 수 있다.

## 5.5 🔴 Secret — 그리고 흔한 오해

비밀번호, API 키, 인증서 같은 **민감한 값**은 ConfigMap 대신 **Secret**에 담는다.
용도는 ConfigMap과 비슷하지만(환경변수·파일로 주입), 민감 정보 전용으로 구분해 다룬다.

여기서 아주 흔한, 그리고 위험한 오해 하나를 반드시 짚어야 한다.

> 🔴 **Secret은 "암호화"가 아니다. 그냥 base64로 인코딩된 것뿐이다.**

base64는 암호화가 아니라 **누구나 되돌릴 수 있는 인코딩**이다.
`echo <값> | base64 -d` 한 줄이면 원문이 나온다.
즉 Secret에 비밀번호를 넣었다고 그게 "보호"되는 게 아니다.

진짜로 보호하려면 추가 조치가 필요하다.
- etcd에 저장될 때 **암호화(encryption at rest)** 를 켜고,
- 누가 Secret을 읽을 수 있는지 **권한(RBAC)** 을 조이고,
- 더 민감하면 외부 비밀 관리 시스템(Vault·클라우드 KMS)과 연동한다.

"Secret에 넣었으니 안전하다"고 믿는 것이 사고의 시작이다.
이 사실을 아는 것만으로도 한 수 위다.

## 5.6 매니페스트로 보기

데이터(PVC)와 설정(ConfigMap/Secret)을 함께 보자.

```yaml
# 설정 (일반)
apiVersion: v1
kind: ConfigMap
metadata: { name: web-config }
data:
  LOG_LEVEL: "info"
  FEATURE_X: "true"
---
# 비밀값 (base64 — 암호화 아님!)
apiVersion: v1
kind: Secret
metadata: { name: web-secret }
type: Opaque
data:
  DB_PASSWORD: c2VjcmV0    # "secret"을 base64 인코딩한 것
---
# 저장소 신청
apiVersion: v1
kind: PersistentVolumeClaim
metadata: { name: data }
spec:
  accessModes: ["ReadWriteOnce"]    # 한 node에서 읽기쓰기
  resources:
    requests:
      storage: 10Gi                 # 10GB 요청
```

그리고 Pod에서 이렇게 가져다 쓴다.

```yaml
    spec:
      containers:
        - name: web
          image: myapp:1.2
          envFrom:
            - configMapRef: { name: web-config }   # 설정 주입
            - secretRef:    { name: web-secret }   # 비밀 주입
          volumeMounts:
            - name: data
              mountPath: /var/lib/data             # 저장소 연결
      volumes:
        - name: data
          persistentVolumeClaim: { claimName: data }
```

읽어보면 — "이 Pod는 web-config의 설정과 web-secret의 비밀을 환경변수로 받고, `data` PVC를 `/var/lib/data`에 연결한다."

(위 `ReadWriteOnce`라는 단어를 기억해두자.
"한 node에서만 마운트 가능"이라는 뜻인데, 이걸 여러 node에서 동시에 쓰려다 막히는 게 흔한 함정이다 — 8장.)

## 5.7 정리

이 장을 한 문장으로 줄이면 이렇다.

> Pod는 일회용이라 데이터는 **PVC(신청)/PV(실제)/StorageClass(자동발급)** 로 Pod 바깥에 영속화하고(상태 앱은 StatefulSet과 한 묶음), 환경별 설정은 **ConfigMap**, 민감값은 **Secret**으로 이미지와 분리해 주입하되 — **Secret은 암호화가 아니라 base64일 뿐**임을 잊지 말아야 한다.

이제 앱을 띄우고, 연결하고, 데이터와 설정까지 갖췄다.
다음 질문은 "자원"이다.
컨테이너에 CPU·메모리를 어떻게 나눠주고, 부하가 몰리면 어떻게 자동으로 늘릴까?
6장에서 **스케줄링·자원·오토스케일링**으로 간다.
