PromQL을 배우기 전에, 메트릭에 어떤 종류가 있는지부터 잡아야 한다.
타입을 잘못 고르면 쿼리가 이상해지고, 분위수(p99)를 못 구하거나 카디널리티가 터진다.

## 2.1 네 가지 타입

| 타입 | 의미 | 노출 형태 |
|---|---|---|
| **Counter**(카운터) | 증가만 하는 값(재시작하면 0) | `x_total` |
| **Gauge**(게이지) | 오르내리는 순간값 | `x` |
| **Histogram**(히스토그램) | 값의 분포를 구간별로 | `x_bucket{le}` + `x_sum` + `x_count` |
| **Summary**(서머리) | 클라이언트가 분위수 직접 계산 | `x{quantile}` + `x_sum` + `x_count` |

- **Counter**는 "누적해서 세는 것"이다 — 총 요청 수, 총 에러 수.
  절대값보다 **증가 속도**가 중요해서 PromQL에서 `rate()`를 씌워 본다(5장).
- **Gauge**는 "지금 이 순간의 값"이다 — 현재 메모리, 큐 길이, 동시 접속 수.
  올라가기도 내려가기도 한다.
- **Histogram**은 "분포"다 — 요청 지연이 0.1초 이하 몇 건, 0.5초 이하 몇 건… 이런 식으로 구간(bucket)별로 센다.
  이걸로 p99 같은 분위수를 서버에서 계산한다.
- **Summary**는 분위수를 클라이언트가 미리 계산한다.
  집계가 안 되는 한계가 있어 대부분 histogram을 권장한다.

## 2.2 Histogram vs Summary — 왜 histogram인가

둘 다 분포를 다루는데 결정적 차이가 있다.

histogram은 bucket(구간별 누적 카운트)을 그대로 노출하고, 분위수는 **서버에서 `histogram_quantile()`로 계산**한다.
그래서 여러 서버의 데이터를 합쳐서 전체 p99를 구할 수 있다(집계 가능).

summary는 각 인스턴스가 미리 자기 분위수를 계산해 노출한다.
그래서 여러 인스턴스의 p99를 합쳐 전체 p99를 구할 수 없다(분위수는 평균낼 수 없다).

대부분의 경우 답은 histogram이다.
분위수 계산 원리와 함정은 5장에서 다룬다.

## 2.3 🔴 native histogram (2026)

기존 histogram에는 약점이 있다.
bucket 경계를 **미리 고정**해야 하고(0.1초·0.5초·1초…), bucket마다 별도 시계열이 생겨 카디널리티를 늘린다.

이걸 푼 게 **native histogram**이다.

- bucket이 **별도 시계열이 아니라 단일 sample 안**에 다해상도로 들어간다.
  `le` 시계열이 사라진다.
- 해상도가 동적이라 같은 비용으로 훨씬 정밀하다.
- **v3.8(2025-12)부터 정식(stable) 기능**이다(그 전엔 실험적).
  scrape에서 켜는 옵션이 필요하고, 향후 버전에서 기본 활성 예정이다.

요점은 "분포를 더 싸고 정밀하게 다루는 새 방식이 표준이 됐다"는 것이다.

## 2.4 메트릭은 어떻게 노출되나

타깃은 `/metrics`에 텍스트로 메트릭을 내건다.

```
# HELP http_requests_total Total HTTP requests.
# TYPE http_requests_total counter
http_requests_total{method="post",code="200"} 1027
```

`# HELP`와 `# TYPE`은 메타데이터고, 나머지 줄이 실제 값이다.
이 텍스트 포맷의 표준화 버전이 OpenMetrics인데, trace로 점프하는 exemplar(메트릭 점에 trace_id를 매다는 것)도 여기서 노출된다.
신호 연결의 그 exemplar다(관측가능성 시리즈 2장).

## 2.5 이름 규약

작은 것 같지만 지키면 두고두고 편한 규칙이 있다.

- counter는 `_total` 접미사.
- 단위는 복수형으로 — `_seconds`·`_bytes`.
- **base unit 사용** — 밀리초가 아니라 seconds, 메가바이트가 아니라 bytes.
- 콜론(`:`)은 규칙(recording rule) 전용이라 직접 계측에선 쓰지 않는다.

2026년부터는 UTF-8 이름이 허용돼서, OpenTelemetry의 점(.) 들어간 이름(`http.server.duration`)도 변환 없이 저장할 수 있다.

## 2.6 정리

- **Counter**(증가만)·**Gauge**(순간값)·**Histogram**(분포)·**Summary**(클라 분위수).
- 분위수는 **histogram + `histogram_quantile`** — 집계가 되기 때문(summary는 안 됨).
- **native histogram**(v3.8 stable): bucket을 단일 sample에 — 싸고 정밀, 카디널리티↓.
- 이름은 `_total`·base unit 규약.
