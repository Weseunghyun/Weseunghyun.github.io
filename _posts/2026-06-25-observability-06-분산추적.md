0장의 그 새벽으로 돌아가자.
"느린데 어디가 느린지 모르겠다."
이 질문에 정면으로 답하는 신호가 **분산추적(distributed tracing)** 이다.

## 6.1 trace와 span

분산추적의 핵심 개념은 두 가지다.

- **Trace(트레이스)**: 하나의 요청이 시스템 전체를 거치는 **전체 경로**.
- **Span(스팬)**: 그 경로 안의 **작업 한 단위**.
  이름, 시작·종료 시각, 상태 등을 담는다.

span들은 부모-자식 관계로 이어져 **트리**를 이룬다.
예를 들어 결제 요청 하나의 트레이스는 이렇게 생겼다.

![](/assets/img/posts/observability-06-분산추적/o11y-trace-tree.svg)

이 그림을 보면 한눈에 안다.
전체 요청이 800ms 걸렸는데, 그중 750ms가 결제 서비스의 DB 호출에서 났다.
범인이 어디인지가 보인다.
로그만으로는 절대 못 얻는 그림이다.

## 6.2 어떻게 흩어진 span을 하나로 묶나

여기서 의문이 생긴다.
요청이 서비스 A를 거쳐 B, C로 넘어가는데, A에서 만든 span과 C에서 만든 span이 **같은 요청**의 것이라는 걸 어떻게 알까.

답은 **공통 ID를 함께 들고 다니는 것**이다.
모든 span은 다음을 공유한다.

- **Trace ID**(16바이트): 이 요청 전체의 고유 번호.
  A·B·C 모두 같은 값.
- **Span ID**(8바이트): 이 작업 한 단위의 번호.
- 부모 Span ID: "나를 누가 불렀는지".

이 정보 묶음을 **span context**라고 한다.
서비스 A가 B를 호출할 때, 이 context를 HTTP 헤더에 실어 보낸다.
B는 그 헤더를 읽어 "아, 나는 이 trace의 일부고, 부모는 A의 그 span이구나"를 안다.
이렇게 context가 서비스 사이를 타고 흐르는 것을 **context propagation(컨텍스트 전파)** 이라고 한다.

## 6.3 W3C Trace Context — 표준 헤더

이 "헤더에 실어 보내기"가 회사마다 제각각이면 서로 추적이 안 끊긴다.
그래서 표준이 생겼다.
**W3C Trace Context**, 그중 `traceparent`라는 헤더다.

생긴 모양은 이렇다.

```
traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01
```

`-`로 구분된 네 부분이 차례로 이렇다.

| 부분 | 값 | 뜻 |
|---|---|---|
| `00` | 버전 | 헤더 형식 버전 |
| `4bf9…4736` | trace ID (16바이트) | 요청 전체의 고유 번호 |
| `00f0…02b7` | 부모 span ID (8바이트) | 나를 부른 작업 |
| `01` | 플래그 | 샘플링 여부 등 |

이 표준 덕에 서로 다른 회사가 만든 라이브러리끼리도 추적이 끊기지 않고 이어진다.
(참고로 이 표준은 단계가 있다 — Level 1은 2020년에 정식 권고가 됐고, Level 2는 2026년 6월 기준 아직 후보 단계다.
"언제 표준이 됐다"고 단정할 때 주의가 필요한 지점이다.)

이 개념의 뿌리는 구글이 2010년에 낸 **Dapper**라는 기술 보고서다.
trace와 span이라는 어휘, 그리고 "성능에 부담을 거의 안 주면서 운영 환경에서 상시 추적한다"는 모델이 여기서 나왔다.

## 6.4 🔴 Sampling — 다 저장할 수는 없다

추적에는 피할 수 없는 비용 문제가 있다.
모든 요청의 모든 span을 다 저장하면 너무 비싸다.
초당 수만 요청이 들어오는 서비스라면 더더욱.

그래서 일부만 저장하는 **샘플링(sampling)** 을 한다.
대부분의 요청이 정상이라면, 그중 일부만 저장해도 시스템을 이해하는 데 충분하다는 발상이다.
(단, 아무거나 막 버리면 안 된다 — 통계적으로 대표성 있게 골라야 한다.)

샘플링에는 두 방식이 있는데, 트레이드오프가 정반대다.

| | Head-based (앞단 샘플링) | Tail-based (뒷단 샘플링) |
|---|---|---|
| 결정 시점 | 요청 **시작할 때** | 요청 **끝난 뒤** |
| 장점 | 싸고 단순 | 에러·느린 요청을 **골라서** 보존 |
| 단점 | **에러 트레이스를 놓칠 수 있음** | 메모리·복잡도 큼 |

**Head-based**는 요청이 시작하는 순간 "이건 저장, 이건 버림"을 정한다.
싸고 간단하지만, 치명적인 약점이 있다.
1%만 저장하기로 했는데 하필 그 1%에 안 든 요청이 에러였다면, **정작 보고 싶은 에러 트레이스를 놓친다**.

**Tail-based**는 요청이 다 끝난 뒤에 결정한다.
"이건 에러가 났네, 저장하자", "이건 느렸네, 저장하자"처럼 결과를 보고 고를 수 있다.
보고 싶은 걸 놓치지 않는다.
하지만 대가가 크다.
요청이 끝날 때까지 **모든 span을 메모리에 들고 있어야** 하고, 한 요청의 모든 span이 같은 수집기에 모여야 판단할 수 있어서 구조가 복잡하다.
(이 tail 샘플링의 함정은 10장에서 실제 사고 사례로 다시 본다.)

공식 권고는 둘을 섞는 것이다.
앞단에서 head로 전체 양을 줄이고, 뒷단에서 tail로 에러·지연을 정교하게 잡는다.

## 6.5 정리

- **Trace** = 요청의 전체 경로, **Span** = 그 안의 작업 한 단위(부모-자식 트리).
- 흩어진 span은 **공통 trace ID + context 전파**로 하나로 묶인다.
- 전파의 표준이 **W3C Trace Context(`traceparent` 헤더)**, 뿌리는 구글 Dapper.
- 다 저장 못 하니 **샘플링** — head(싸지만 에러 놓침) vs tail(에러 잡지만 무거움), 보통 혼합.

다음 장에서는 이 추적을 포함해 메트릭·로그까지 한 번에 표준화한 **OpenTelemetry**, 그리고 추적을 저장하는 백엔드(Jaeger·Tempo)를 본다.
