## 1. 한눈 요약표 — 시리즈 1~14장 핵심 명제

각 행은 "이것만은 기억하라"는 한 줄이다.
세부는 오른쪽 장에서 다뤘다.

| 주제 | 핵심 명제 | 다룬 장 |
|---|---|---|
| 정체성 | 컬럼 지향 OLAP DBMS — 대량 집계·분석에 최적, OLTP(단건 트랜잭션·잦은 업데이트)에는 부적합 | 1장 |
| 빠른 이유 | 필요한 컬럼만 읽고, 같은 타입을 모아 압축률을 높여 분석 쿼리 I/O를 크게 줄인다 | 2장 |
| MergeTree | 핵심 엔진 계열 — ORDER BY(정렬 키)로 정렬 저장하고 백그라운드에서 파트를 병합 | 3장 |
| 인덱스 성격 | PRIMARY KEY/ORDER BY는 유니크 제약이 아니라 sparse primary index — granule 단위로 읽을 범위를 좁힌다 | 3·4장 |
| part 단위 쓰기 | 데이터는 part로 쓰고 작은 파트를 병합 — 잦은 소량 INSERT는 'too many parts'를 부르므로 배치 INSERT가 원칙 | 3·8장 |
| 파티셔닝 | PARTITION BY는 데이터 관리(DROP/TTL)용 — 성능 튜닝 1차 수단이 아니고, 고카디널리티 키는 파트 폭증을 부른다 | 4장 |
| 정렬 키 설계 | 카디널리티 낮은 컬럼을 앞에 — WHERE/GROUP BY 패턴에 맞춰야 인덱스 prefix가 효과를 낸다 | 4장 |
| Skipping index | minmax/set/bloom_filter 등 보조 인덱스 — granule을 건너뛰지만 정렬 키만큼 강력하지 않고 데이터 분포에 좌우된다 | 5장 |
| MergeTree 변형 | Replacing/Summing/Aggregating/Collapsing은 **병합 시점**에 동작 — 중복 제거·집계가 즉시 보장되지 않아 FINAL이나 GROUP BY로 보정 | 6장 |
| Materialized View | INSERT 트리거로 동작 — 들어온 블록에만 변환 적용, 기존 데이터를 자동 소급하지 않는다 | 7장 |
| 삽입 패턴 | 대량·비동기 수집은 async_insert·Buffer·외부 배칭(Kafka 엔진 등) — 정확히-한-번 보장과 지연 사이 트레이드오프 | 8장 |
| 분산·복제 | Distributed 엔진 + 샤드/레플리카 — 집계는 샤드 부분 집계 후 코디네이터 최종 병합(two-stage). 복제는 ReplicatedMergeTree + Keeper(ZooKeeper 호환 대체재, 권장) | 9장 |
| JOIN·Dictionary | JOIN은 기본 오른쪽 테이블을 메모리 적재(hash join) — 작은 테이블을 오른쪽에. 자주 쓰는 차원은 Dictionary + dictGet()으로 대체 | 10장 |
| 쿼리 진단 | EXPLAIN, system.query_log, system.parts로 읽은 행/바이트·granule 스킵·메모리를 근거로 튜닝 | 11장 |
| TTL | 컬럼/행/파티션 단위 만료·이동(계층형 스토리지) — 비용 절감과 보존 정책의 핵심 | 12장 |
| 스키마·압축 | LowCardinality, 적절한 코덱(Delta·DoubleDelta·Gorilla·ZSTD 등), 타입 최소화로 압축·성능 개선 | 13장 |
| 안티패턴 | 단건 INSERT 남발 · 고카디널리티 PARTITION BY · FINAL 남용 · 무분별한 대형 JOIN · mutation 남용 — 모두 피한다 | 14장 |

이 표 하나가 머리에 들어오면,
ClickHouse를 처음 접하는 동료에게 30분 안에 큰 그림을 설명할 수 있다.

## 2. 단계별 체크리스트 — 테이블을 설계하고 운영할 때

개념을 아는 것과 손으로 만드는 것은 다르다.
실제로 테이블을 하나 만든다고 할 때, 위에서 아래로 짚어야 할 순서다.

### 설계 단계

- [ ] 이 데이터는 정말 OLAP인가?
  — 단건 조회·잦은 업데이트 위주면 ClickHouse가 답이 아닐 수 있다 (1장)
- [ ] 쿼리 패턴을 먼저 적었는가?
  — 어떤 WHERE, 어떤 GROUP BY가 자주 오는지 (4장)
- [ ] ORDER BY를 그 쿼리 패턴에 맞췄는가?
  — 카디널리티 낮은 컬럼을 앞에 (4장)
- [ ] PARTITION BY는 관리 목적인가?
  — 보통 월/일 단위.
  고카디널리티 키로 파티션을 쪼개지 않았는가 (4장)
- [ ] 타입을 최소화했는가?
  — 불필요하게 큰 정수/문자열 대신 적정 타입, 반복되는 문자열엔 LowCardinality (13장)
- [ ] 시계열/단조 증가 컬럼에 코덱을 고려했는가?
  — Delta·DoubleDelta·Gorilla 등 (13장)

### 집계·뷰 단계

- [ ] 사전 집계가 필요한가?
  — 그렇다면 Materialized View.
  단, 들어온 블록에만 적용되고 기존 데이터는 소급 안 됨을 기억 (7장)
- [ ] 중복 제거/합계가 필요한가?
  — Replacing/Summing 변형은 병합 시점 동작이라 즉시 보장이 아님.
  쿼리에서 FINAL/GROUP BY 보정 (6장)
- [ ] 차원 테이블 JOIN이 반복되는가?
  — Dictionary + dictGet()으로 대체 검토 (10장)

### 삽입 단계

- [ ] 배치로 넣는가?
  — 단건/소량 INSERT 남발은 'too many parts'의 직행 티켓 (3·8장)
- [ ] 클라이언트가 배칭을 못 하면?
  — async_insert 또는 Buffer 테이블, 외부 배칭(Kafka 엔진 등) (8장)
- [ ] 정확히-한-번이 필요한가, 지연을 더 허용해도 되는가?
  — 둘 사이 트레이드오프를 명시적으로 정함 (8장)

### 쿼리·운영 단계

- [ ] 느린 쿼리는 EXPLAIN으로 봤는가?
  — 읽은 행/바이트, granule 스킵 효과를 근거로 (11장)
- [ ] system 테이블을 보는가?
  — query_log, parts로 실제 동작을 확인 (11장)
- [ ] 오래된 데이터 정책이 있는가?
  — TTL로 만료·이동 자동화, 비용 통제 (12장)
- [ ] 복제·분산이 필요한가?
  — ReplicatedMergeTree + Keeper, Distributed 엔진 (9장)
- [ ] 안티패턴을 피했는가?
  — FINAL 남용, 대형 JOIN 무분별, mutation(ALTER UPDATE/DELETE) 남발 (14장)

체크리스트의 순서 자체가 메시지다.
**쿼리 패턴 → 정렬 키 → 삽입 방식 → 진단** 순으로 생각하면,
ClickHouse가 어디서 빨라지고 어디서 무너지는지가 자연스럽게 따라온다.

## 3. 면접 예상 Q&A

질문 형태로 정리하면 같은 개념도 다르게 박힌다.
실제 면접에서 자주 변주되는 것들이다.

**Q1.
ClickHouse는 왜 OLTP에 안 맞나요?**
컬럼 지향 OLAP 엔진이라 대량 스캔·집계에 최적화돼 있다 (1장).
반대로 단건 point lookup은 느리고,
UPDATE/DELETE는 즉시 반영이 아니라 비동기 mutation으로 처리되며,
여러 문장을 묶는 일반적인 ACID 트랜잭션 모델이 없다 (확정 사실 기준).
즉 "한 행을 자주 고치는" 업무에는 구조적으로 불리하다.

**Q2.
왜 그렇게 빠른가요?
한 문장으로.**
쿼리에 필요한 컬럼만 읽고,
같은 타입 데이터를 모아 높은 압축률로 디스크 I/O를 줄이며,
그 위에 벡터화·병렬 실행을 얹기 때문이다 (2장).
숫자로 보면 공식 벤치마크에서 1억 행 집계를 약 92ms,
초당 10억 행 이상으로 처리한다 (공식 문서 기준).

**Q3.
PRIMARY KEY가 유니크 제약이 아니라는 게 무슨 뜻인가요?**
RDB와 가장 헷갈리는 지점이다.
ClickHouse의 PRIMARY KEY/ORDER BY는 중복을 막는 제약이 아니라,
데이터를 정렬해 저장하고 **sparse primary index**를 만든다 (3·4장).
이 인덱스는 모든 행이 아니라 granule(기본 8192행) 단위로 표시를 남겨,
WHERE 조건에 맞는 후보 범위만 읽게 좁혀준다.

**Q4.
granule이 뭔가요?**
정렬된 데이터를 일정 행 수(기본 index_granularity 8192)로 끊은 묶음이다 (4장).
인덱스는 granule마다 한 점씩만 기록하므로 인덱스가 아주 작다.
4장에서 본 실측처럼, 880만 행이 약 1000개 남짓 granule로 줄고,
필터 조건이 정렬 순서와 맞으면 이진 탐색 몇 스텝으로 후보를 좁혀
실제로 읽는 행 수가 극적으로 줄어든다.

**Q5.
'too many parts' 에러는 왜 나고 어떻게 막나요?**
ClickHouse는 데이터를 part 단위로 쓰고 작은 파트를 백그라운드에서 병합한다 (3장).
단건/소량 INSERT를 남발하면 파트가 병합 속도보다 빨리 쌓여 한계에 부딪힌다.
해법은 배치 INSERT가 원칙이고,
클라이언트가 못 모으면 async_insert·Buffer·외부 배칭(Kafka 엔진 등)을 쓴다 (8장).

**Q6.
Materialized View는 기존 데이터도 변환해 주나요?**
아니다.
INSERT 트리거로 동작해 **새로 들어온 블록에만** 변환을 적용한다 (7장).
이미 들어와 있던 과거 데이터는 자동으로 소급되지 않는다.
이 점을 모르면 "뷰가 비어 있다"는 함정에 자주 빠진다.

**Q7.
ReplacingMergeTree를 쓰면 중복이 바로 사라지나요?**
아니다.
Replacing/Summing/Aggregating 같은 변형은 **병합 시점**에 동작한다 (6장).
병합이 일어나기 전에는 중복이 남아 있을 수 있으므로,
쿼리에서 FINAL을 쓰거나 GROUP BY로 보정해야 정확한 결과를 얻는다.
다만 FINAL 남용은 비용이 크니 주의한다 (14장).

**Q8.
큰 테이블 두 개를 JOIN하면 왜 메모리가 터지나요?**
JOIN은 기본적으로 오른쪽 테이블을 메모리에 적재하는 hash join이다 (10장).
그래서 작은 테이블을 오른쪽에 두는 것이 원칙이고,
대형 JOIN은 메모리 한계·분산 처리(GLOBAL JOIN)·Dictionary 대체를 고려한다.
반복되는 차원 조회는 dictGet()으로 바꾸는 게 정석이다.

**Q9.
분산 환경에서 GROUP BY는 어떻게 동작하나요?**
Distributed 엔진 + 샤드/레플리카 구조에서,
각 샤드가 자기 데이터를 부분 집계하고
코디네이터 노드가 그 결과를 최종 병합한다 — 이른바 two-stage aggregation (9장).
복제는 ReplicatedMergeTree와 코디네이션 서비스로 이뤄지며,
ZooKeeper 호환 대체재인 ClickHouse Keeper가 권장된다 (9장).

**Q10.
운영하면서 피해야 할 안티패턴을 꼽는다면?**
다섯 가지를 외워두면 된다 (14장).
단건/소량 INSERT 남발,
고카디널리티 PARTITION BY,
FINAL 남용,
무분별한 대형 JOIN,
mutation(ALTER UPDATE/DELETE) 남용.
이 다섯이 성능·안정성을 가장 자주 무너뜨린다.

## 마무리

처음 이 시리즈를 시작할 때,
ClickHouse는 "그냥 빠른 분석 DB" 정도로 보였을지 모른다.
하지만 15장을 지나온 지금은,
**컬럼 저장 → 정렬 키 → sparse index → part·머지 → 배치 삽입 → 분산·복제** 라는 하나의 일관된 설계 철학으로 읽힌다.
빠른 이유도, 느려지는 이유도, 안티패턴이 안티패턴인 이유도 전부 이 흐름 위에 있다.

다음 16장에서는 ClickHouse AI 기능을 다루고,
17·18장에서는 한 발 물러서서 행기반 RDB와의 기원 비교, 그리고 OLAP·실시간 분석 DB 지형 속 위치를 살핀다.
이 장의 요약표와 체크리스트는,
앞으로 실제 테이블을 만들 때 옆에 펼쳐두는 지도로 쓰면 된다.
