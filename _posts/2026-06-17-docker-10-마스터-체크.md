## 10.1 한 장 요약 — 시리즈 전체를 한 표로

| 주제 | 핵심 한 줄 |
|---|---|
| Docker란 | 앱+환경을 이미지로 묶어 컨테이너로 실행하는 도구 |
| 컨테이너 vs VM | VM은 OS를 새로 깔고, 컨테이너는 **호스트 커널 공유** → 가벼움 |
| 컨테이너의 정체 | 프로세스 + **namespace**(격리) + **cgroup**(제한) + 권한 제한 |
| 런타임 스택 | CLI → dockerd → containerd → shim → **runc** |
| 이미지 | 읽기전용 **레이어** 층층(overlay2) + 컨테이너의 쓰기 판(CoW) |
| 빌드 캐시 | 안 바뀌는 것 위, 자주 바뀌는 것 아래 |
| 네트워킹 | 사용자 정의 네트워크 → 이름 통신, `-p` → 외부 노출 |
| 데이터 | 보존할 건 **볼륨**(컨테이너와 분리) |
| Compose | 여러 컨테이너를 한 파일로, healthcheck로 준비 대기 |
| 함정 | PID1·OOM·캐시·방화벽 우회·컨테이너 탈출 |

## 10.2 스스로 점검 — 개념 체크리스트

각 항목을 "남에게 설명할 수 있는가"로 자문해보자.

- [ ] 컨테이너가 VM보다 가벼운 이유를 "커널 공유"로 설명할 수 있다.
- [ ] namespace와 cgroup의 차이를 "격리 vs 제한"으로 구분할 수 있다.
- [ ] 컨테이너 안 첫 프로세스가 PID 1이고, 그게 왜 함정인지 안다.
- [ ] `docker run` 뒤에 dockerd·containerd·runc·shim이 무슨 일을 하는지 안다.
- [ ] 데몬을 재시작해도 컨테이너가 안 죽는 이유(shim)를 안다.
- [ ] 이미지가 레이어로 나뉘어 얻는 이득(공유·빠른 시작)을 안다.
- [ ] copy-on-write와 "컨테이너 지우면 데이터 사라짐"을 연결할 수 있다.
- [ ] Dockerfile에서 의존성을 소스보다 위에 두는 이유를 안다.
- [ ] 멀티스테이지 빌드가 이미지를 줄이는 원리를 안다.
- [ ] 사용자 정의 네트워크라야 컨테이너 이름으로 통신되는 걸 안다.
- [ ] `-p`가 방화벽을 우회할 수 있다는 걸 안다.
- [ ] 보존할 데이터는 볼륨에 둬야 하는 이유를 안다.
- [ ] OOM kill(137)과 런타임 힙 설정의 관계를 안다.
- [ ] `--privileged`·docker.sock 마운트가 왜 위험한지 안다.

## 10.3 운영 체크리스트 — 컨테이너 띄우기 전

실제로 컨테이너를 운영에 올리기 전에 확인할 것들이다.

| 항목 | 확인 |
|---|---|
| 이미지 | 멀티스테이지로 작게, 베이스 가볍게, 태그/digest 고정 |
| 실행 사용자 | 비-root(`USER`), 가능하면 `--read-only` |
| 신호 처리 | `--init` + exec 형태 ENTRYPOINT |
| 자원 한도 | `--memory`·`--cpus` 설정, 런타임 힙과 정렬 |
| 네트워크 | 필요한 포트만 `-p`, 내부용은 `127.0.0.1:` 바인드 |
| 데이터 | 상태/DB는 볼륨, `down -v` 주의 |
| 로그 | 회전 설정(`max-size`/`max-file`) |
| 헬스체크 | `HEALTHCHECK` 정의 |
| 보안 | privileged·docker.sock 마운트 회피 |

## 10.4 면접 Q&A 모음

**Q.
컨테이너와 VM의 차이는?**
VM은 하이퍼바이저 위에 게스트 OS를 통째로 띄우지만, 컨테이너는 호스트 커널을 공유하면서 namespace로 격리하고 cgroup으로 자원을 제한한다.
그래서 컨테이너가 훨씬 가볍고 빠르지만, 커널을 공유하는 만큼 격리 강도는 VM보다 약하다.

**Q.
컨테이너는 어떻게 격리되나?**
리눅스 커널의 namespace가 프로세스·네트워크·파일시스템 등 "보이는 것"을 격리하고, cgroup이 CPU·메모리 등 "쓰는 양"을 제한한다.
거기에 capability·seccomp로 할 수 있는 일을 줄인다.

**Q.
`docker run`을 치면 무슨 일이 일어나나?**
CLI가 dockerd에 요청하고, dockerd가 containerd에 넘기고, containerd가 shim을 통해 runc를 호출한다.
runc가 실제로 namespace와 cgroup을 만들어 프로세스를 띄운 뒤 빠지고, shim이 그 컨테이너의 부모로 남는다.

**Q.
Docker 데몬을 재시작하면 컨테이너가 죽나?**
아니다.
컨테이너의 부모는 데몬이 아니라 shim이라서, 데몬을 재시작·업그레이드해도 실행 중인 컨테이너는 살아 있다.

**Q.
이미지 레이어가 뭐고 왜 좋나?**
이미지는 읽기 전용 레이어의 스택이고, 같은 레이어는 여러 이미지가 공유한다.
컨테이너는 그 위에 얇은 쓰기 판만 얹어 시작하므로 빠르고, 디스크·전송도 절약된다.

**Q.
Dockerfile 캐시를 잘 쓰려면?**
한 레이어가 바뀌면 그 아래 캐시가 다 무효가 되므로, 자주 안 바뀌는 의존성을 위에 두고 자주 바뀌는 소스를 아래에 둔다.

**Q.
컨테이너 데이터를 어떻게 보존하나?**
컨테이너의 쓰기 판은 컨테이너와 함께 사라지므로, 데이터베이스나 상태는 볼륨으로 컨테이너 바깥에 둔다.

**Q.
쿠버네티스가 Docker를 버렸다는데?**
정확히는 dockerd 중간층을 건너뛰고 그 아래 containerd를 직접 쓰기로 한 것이다.
Docker로 빌드한 OCI 이미지는 그대로 잘 돈다.

**Q.
컨테이너 보안에서 조심할 것은?**
컨테이너는 커널을 공유하므로 절대적 격리가 아니다.
`--privileged`·docker.sock 마운트·호스트 루트 마운트를 피하고, 비-root·최소 권한으로 돌리며, 강격리가 필요하면 gVisor·Kata를 쓴다.

## 10.5 여기서 더 나아가려면

이 시리즈는 컨테이너 **하나**와 **한 서버 안의 묶음**(Compose)까지를 다뤘다.
그다음 자연스러운 단계는 이것들이다.

- **쿠버네티스** — 여러 서버에 컨테이너를 배치·확장·복구하는 오케스트레이션.
- **이미지 보안** — 취약점 스캔, 서명, 최소 베이스(distroless).
- **CI/CD** — 빌드·테스트·배포 자동화 파이프라인에 컨테이너 끼우기.

하지만 그 모든 것의 바닥에는 이 시리즈에서 배운 것이 그대로 깔려 있다.
컨테이너는 결국 namespace와 cgroup으로 격리된 프로세스이고, 이미지는 레이어의 층층이라는 사실.
이 원리를 쥐고 있으면, 그 위에 무엇이 올라와도 흔들리지 않는다.
