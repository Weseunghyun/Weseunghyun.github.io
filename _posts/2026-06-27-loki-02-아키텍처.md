Loki는 쓰기와 읽기를 컴포넌트로 나눠 처리한다.
배포 모드는 이 컴포넌트들을 어떻게 묶느냐의 차이다.

## 2.1 컴포넌트

| 컴포넌트 | 역할 |
|---|---|
| **distributor** | 쓰기 입구. 검증·레이블 정규화·rate limit, 해시 링으로 ingester에 분산 |
| **ingester** | 메모리 chunk + WAL, 다 차면 압축해 오브젝트 스토리지로 flush |
| **querier** | LogQL 실행. ingester(최근)+스토리지(과거) 동시 조회·중복 제거 |
| **query-frontend** | 큰 쿼리를 분할·캐시 |
| **query-scheduler** | 테넌트별 큐로 공정하게 분배 |
| **compactor** | 색인 압축 + retention(삭제) (6장) |
| **ruler** | 로그 기반 규칙·알림 평가 (6장) |
| **index-gateway** | 색인 조회 서빙 |

## 2.2 쓰기·읽기 경로

![](/assets/img/posts/loki-02-아키텍처/loki-read-write-path.svg)

**쓰기**는 이렇게 흐른다.

1. **distributor**가 들어온 로그를 검증·정규화하고, `(테넌트 + 스트림 레이블)` 해시로 ingester를 정한다.
2. replication factor(보통 3)만큼 ingester에 복제해 보내고, 과반의 확인을 받으면 성공으로 친다.
3. **ingester**는 스트림별로 메모리 chunk에 쌓다가(동시에 WAL에 기록), 다 차면 압축해 오브젝트 스토리지로 flush한다.

**읽기**는 이렇다.

1. **query-frontend**가 큰 쿼리를 잘게 나누고 캐시를 확인한다.
2. **querier**가 그 조각들을 받아, **ingester(최근 메모리)+오브젝트 스토리지(과거)** 를 함께 조회한다.
3. replication으로 인한 중복은 (타임스탬프+레이블+메시지 기준으로) 제거한다.

## 2.3 배포 모드

| 모드 | 구성 | 규모 |
|---|---|---|
| monolithic | 한 프로세스(`-target=all`) | ~20GB/day |
| simple scalable(SSD) | write·read·backend 3타깃 | ~1TB/day |
| microservices | 컴포넌트별 독립 | >1TB/day |

🔴 다시 강조 — **SSD 모드는 deprecated**(Loki 4.0 전 제거 예정).
Helm 차트 기본 모드도 SSD에서 monolithic으로 바뀌었다.
신규 배포는 monolithic(작게 시작)이나 microservices(크게)를 택한다.

## 2.4 정리

- **쓰기**: distributor(분산) → ingester(메모리+WAL → 스토리지 flush).
- **읽기**: query-frontend(분할) → querier(ingester+스토리지 조회·dedup).
- 배포는 monolithic/SSD(deprecated)/microservices — SSD는 피한다.
