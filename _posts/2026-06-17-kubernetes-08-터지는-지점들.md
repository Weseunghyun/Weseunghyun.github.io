지금까지 쿠버네티스를 만들고 쓰는 법을 배웠다.
이 장은 결이 다르다.
**실제 운영에서 어디가, 왜 터지는가** 를 모은다.

여기 모은 것들은 우연한 버그가 아니라, 앞 장에서 배운 원리들이 **서로 맞물리며 생기는 함정**이다.
그래서 원리를 알면 증상만 보고도 원인이 보인다.
그리고 거의 모든 진단은 같은 두 명령에서 시작한다 — `kubectl describe pod`와 `kubectl logs`(7장).

## 8.1 🔴 CrashLoopBackOff — 가장 자주 보는 그것

Pod 상태에 `CrashLoopBackOff`가 뜨면, **Pod가 떴다 죽었다를 반복**하고 있다는 뜻이다.
쿠버네티스가 재시작 간격을 점점 늘리며("BackOff") 계속 살려보는 중이다.

원인 조합이 몇 가지다.
- **liveness probe가 너무 공격적**(6장): 앱 초기화가 안 끝났는데 "안 살아있네"로 판정 → 재시작 → 또 초기화 중 판정 → 무한 반복.
  해결: `startupProbe`로 초기화를 유예하거나 liveness를 여유 있게.
- **앱 자체가 시작하자마자 죽음**: 설정 오류, 필수 환경변수 누락, 의존 서비스에 접속 실패.

진단:
```bash
kubectl describe pod <이름>      # 이벤트에서 재시작 사유
kubectl logs <이름> --previous   # 🔴 죽기 직전 로그 (왜 죽었는지)
```
`--previous`가 핵심이다 — 이미 죽은 컨테이너의 마지막 유언을 봐야 원인을 안다.

## 8.2 🔴 OOMKilled (종료 코드 137)

Pod가 갑자기 죽고 종료 코드가 **137**이면, 메모리 **limit**을 넘어 강제 종료(OOM kill)당한 것이다(6장, Docker 9장).

그런데 더 교묘한 버전이 있다.
자바(JVM)나 Node.js 같은 런타임이 "내가 쓸 수 있는 메모리"를 판단할 때, **컨테이너의 limit이 아니라 node 전체 메모리**를 기준으로 잡는 경우다.
예: 컨테이너 limit은 512MB인데, 런타임은 node의 16GB를 보고 힘껏 메모리를 잡다가 512MB를 넘는 순간 죽는다.
런타임은 자기가 왜 죽었는지도 모른다.

해결:
- 컨테이너 limit과 런타임의 메모리 설정을 **함께** 맞춘다 (자바·노드 각각의 옵션으로).
- `kubectl describe pod`에서 `OOMKilled: true`인지 확인.
- limit을 무작정 올리지 말 것 — node가 부족해지면 다른 Pod가 QoS 순서로 쫓겨난다(6장).

## 8.3 🔴 Pending — Pod가 안 떠요

Pod가 `Pending` 상태에 머물면, **들어갈 자리를 못 찾았다**는 뜻이다.
2장에서 본 스케줄러가 배치할 node를 못 정한 것이다.

흔한 원인:
- **requests 합 > node 여유**(6장): 요청한 자원을 받아줄 node가 없음.
- **PVC가 안 풀림**(아래 8.4): 필요한 저장소를 못 받아 대기.
- **taint를 견디는 toleration이 없음**(6장): 갈 수 있는 node가 막혀 있음.
- affinity 규칙을 만족하는 node가 없음.

진단은 역시 `kubectl describe pod`.
맨 아래 이벤트에 `FailedScheduling`과 함께 **정확한 이유**가 적혀 있다 ("0/5 nodes available: insufficient memory" 같은 식).

🔴 반대 함정도 있다.
requests를 너무 **낮게** 잡으면 Pending은 안 나지만, node에 Pod가 과밀하게 들어차 실제 부하에서 OOM·경합이 터진다(6장).
Pending과 과밀은 동전의 양면이다.

## 8.4 🔴 ImagePullBackOff & PVC Pending

**ImagePullBackOff** — 이미지를 못 받는 상태.
- 태그 오타, 존재하지 않는 태그.
- **프라이빗 레지스트리 인증 누락**(imagePullSecret을 안 줌).
- `latest` 태그라 어제와 다른(깨진) 이미지가 잡힘 → 버전 고정으로 예방(Docker 4장).

**PVC가 안 풀려 Pod가 Pending** —
- 맞는 StorageClass가 없거나 동적 프로비저닝 실패.
- 🔴 **접근 모드 충돌**: 5장에서 본 `ReadWriteOnce`(한 node에서만 마운트) 볼륨을 여러 node의 Pod가 동시에 쓰려 하면 막힌다.
  여러 곳에서 동시에 쓰려면 `ReadWriteMany`를 지원하는 스토리지가 필요하다.

## 8.5 Secret을 암호화로 착각

5장에서 강조한 그것.
**Secret은 base64일 뿐 암호화가 아니다.**
"비밀번호를 Secret에 넣었으니 안전해"라고 믿는 것 자체가 함정이다.
etcd 저장 암호화(encryption at rest), 접근 권한(RBAC), 외부 비밀 관리 도구로 실제 보호를 더해야 한다.

## 8.6 NetworkPolicy를 썼는데 안 막힌다

4장에서 예고한 함정.
NetworkPolicy로 "이 통신은 막아라"라고 적었는데도 다 통한다면, **CNI 플러그인이 NetworkPolicy를 지원하지 않는** 경우다.
정책은 조용히 무시된다.
"막은 줄 알았는데 안 막힌" 보안 사고의 흔한 원인 — NetworkPolicy를 지원하는 CNI(Calico·Cilium 등)인지 먼저 확인해야 한다.

## 8.7 🔴 etcd 손상 = 클러스터의 기억 상실

가장 무거운 사고다.
2장에서 etcd가 "클러스터의 모든 상태를 담은 기억"이라고 했다.
etcd가 깨지거나, 복제본들이 과반의 동의를 이루지 못하면(quorum 상실), **클러스터는 자기 상태 전체를 잃는다.**
이미 떠 있는 Pod는 잠깐 더 살지 몰라도, 새로운 조정·변경이 불가능해진다.

예방:
- etcd **정기 스냅샷 백업** (재해 복구의 핵심).
- 홀수 대 구성, 분리된 안정적 디스크.
- 관리형 쿠버네티스(EKS·GKE 등)를 쓰면 이 부분을 클라우드가 대신 책임진다(2장) — 많은 팀이 관리형을 택하는 큰 이유다.

## 8.8 함정 한눈에 보기

| 증상 | 원인 | 첫 진단 / 예방 |
|---|---|---|
| CrashLoopBackOff | liveness 과공격·앱 크래시 | `logs --previous` / startupProbe·설정 점검 |
| OOMKilled (137) | 메모리 limit 초과 | limit과 런타임 힙 함께 설정 |
| Pending | requests 과대·PVC·taint | `describe`의 FailedScheduling |
| ImagePullBackOff | 태그/인증 | 태그 고정·imagePullSecret |
| PVC Pending | StorageClass·접근모드 | RWO vs RWX 확인 |
| 정책 안 먹음 | CNI 미지원 | NetworkPolicy 지원 CNI 사용 |
| 클러스터 마비 | etcd 손상 | etcd 백업·관리형 K8s |

## 8.9 정리

이 장을 한 문장으로 줄이면 이렇다.

> 쿠버네티스의 함정 대부분은 새로운 버그가 아니라 **앞에서 배운 원리(probe·requests/limits·PVC·etcd·CNI)가 맞물리며** 생기고, 진단은 거의 항상 **`kubectl describe`(이벤트)와 `kubectl logs --previous`(유언)** 에서 시작된다.
