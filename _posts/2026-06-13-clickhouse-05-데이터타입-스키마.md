4장에서 우리는 ClickHouse가 정렬키(ORDER BY)와 희소 인덱스로 어떻게 빠르게 행을 좁히는지 봤다.
그런데 거기엔 한 가지 전제가 숨어 있었다.
"각 컬럼이 디스크에 얼마나 작게, 얼마나 잘 압축되어 누워 있는가."
이건 결국 **어떤 데이터타입을 골랐느냐**에 달려 있다.

ClickHouse에서 타입 선택은 단순히 "정수냐 문자열이냐"의 문제가 아니다.
같은 데이터라도 타입을 어떻게 고르느냐에 따라 디스크 사용량, 압축률, 그리고 쿼리 속도가 눈에 띄게 달라진다.
이 장은 그래서 "타입 카탈로그"가 아니라 **"타입으로 성능을 설계하는 법"** 이다.

---

## 첫 원리: 컬럼은 곧 파일이고, 타입은 그 파일의 모양이다

ClickHouse는 컬럼 지향 DB다(2장).
한 컬럼은 디스크에 자기만의 파일(들)로 저장되고, 그 파일은 압축되어 누워 있다.
그러니 타입을 고른다는 건 곧 **"이 컬럼 파일이 어떤 바이트 패턴으로 쌓일지"** 를 정하는 일이다.

여기서 두 가지 원칙이 나온다.
첫째, **좁을수록 좋다.** 0~255 사이 값만 들어오는 컬럼을 `UInt32`(4바이트)로 잡으면, 매 행마다 쓸데없는 바이트 3개를 디스크에 깔고 압축·해제하는 셈이다.
`UInt8`(1바이트)이면 충분하다.
둘째, **규칙적일수록 좋다.** 비슷한 값이 모여 있거나 일정한 패턴이면 압축기가 신나게 줄인다.
그래서 4장에서 본 "정렬 순서로 압축률이 3:1에서 39:1까지 벌어진" 현상이 생긴다.

이 두 원칙을 기억하고, 실제 타입들을 하나씩 보자.

## Nullable — 편하지만 거의 항상 손해

많은 RDB 출신 개발자가 무심코 `Nullable(String)`, `Nullable(Int32)`를 쓴다.
"값이 없을 수도 있으니까."
ClickHouse에서 이건 대체로 나쁜 습관이다.

**비유로 먼저.** (이건 이해를 돕는 비유다.
실제 저장 구조는 바로 아래에서 설명한다.)
어떤 컬럼을 "값 노트"와 "이 칸이 비었는지 표시하는 체크리스트 노트", 이렇게 노트 두 권으로 관리한다고 생각해 보자.
값을 읽을 때마다 두 권을 다 펼쳐야 한다.

실제 구조도 거의 그렇다.
`Nullable` 컬럼은 정상 값을 담는 파일과 **별도로 행별 NULL 마스크 파일**(행마다 UInt8로 "이 값이 NULL인가")을 하나 더 둔다.
파일이 하나 더 생기니 저장 공간을 더 쓰고, 읽을 때 한 번 더 건드려야 한다.
공식 문서는 이걸 아주 단호하게 말한다 — "Nullable을 쓰면 거의 항상 성능에 부정적 영향을 준다"(공식 문서 표현 그대로).

그럼 어떻게 하나.
스키마 설계 가이드는 **NULL 대신 기본값(default value)을 쓰라**고 권한다.
"값 없음"을 빈 문자열이나 0, 혹은 의미상 안전한 기본값으로 표현하면 마스크 파일이 사라진다.

```sql
-- 나쁜 예: NULL 마스크 파일이 따로 생긴다
user_id  Nullable(UInt64)

-- 권장: 기본값으로 "없음"을 표현
user_id  UInt64 DEFAULT 0
```

물론 "0과 NULL을 의미상 꼭 구분해야 하는" 도메인이라면 Nullable이 필요할 수 있다.
하지만 그건 예외이고, 기본 태도는 "일단 Nullable을 피한다"가 맞다.

## LowCardinality — 반복되는 문자열의 정석

로그나 이벤트 테이블엔 같은 문자열이 수없이 반복되는 컬럼이 흔하다.
국가 코드, 상태값, 디바이스 타입 같은 것들.
`KR`, `US`, `JP` ... 수천만 행에 같은 값 몇십 종이 계속 나온다.
이런 컬럼을 그냥 `String`으로 두면 같은 글자를 매번 통째로 저장한다.

`LowCardinality(String)`은 여기에 **딕셔너리 인코딩**을 씌운다.

**비유.** (이해용 비유다.)
긴 단어를 매번 쓰는 대신, 단어집을 한 권 만들어 두고 본문엔 "단어집 7번"처럼 번호만 적는 것.
본문이 훨씬 짧아진다.

실제 구조도 그렇다.
LowCardinality 컬럼은 내부적으로 (a) 유니크한 값들의 **딕셔너리**와 (b) 각 행이 딕셔너리 몇 번을 가리키는지를 담은 **인덱스 컬럼**, 두 부분으로 나뉜다(소스 코드 확인).
인덱스의 정수폭(UInt8/16/32/64)은 유니크 값 개수에 맞춰 자동으로 선택된다 — 종류가 적으면 작은 정수로.

한 가지 중요한 디테일이 있다.
이 딕셔너리는 **데이터 파트 단위로 로컬(part-local)** 이다.
즉 같은 `KR`이라도 파트마다 인덱스 번호가 다를 수 있다(소스 확인).
3장에서 본 "part는 불변, 머지로 합쳐진다"는 구조와 맞물리는 설계다.

그럼 아무 문자열에나 다 씌우면 되나?
아니다.
LowCardinality는 **카디널리티(유니크 값 개수)** 가 적당히 낮을 때만 이득이다.

| 유니크 값 개수 | 효과 |
|---|---|
| ~1만 미만 | 읽기·저장 효율이 더 높다 (공식 권장 구간) |
| ~10만 초과 | 일반 타입보다 성능이 나빠질 수 있다 |

10만을 넘기면 왜 나빠지나.
딕셔너리 자체가 커지면서 메모리·캐시 효율이 떨어지고, 딕셔너리를 관리(삽입·조회·정리)하는 비용이 늘고, 인덱스 정수도 UInt32/64로 굵어지기 때문이다(소스 확인).
딕셔너리 최대 크기는 `low_cardinality_max_dictionary_size` 설정으로 제어할 수 있다.

LowCardinality는 String뿐 아니라 FixedString, Date, DateTime, 숫자형(Decimal 제외)에도 쓸 수 있다.
그리고 문자열 범주값이라면 — 다음에 볼 Enum보다 **LowCardinality(String)을 먼저 고려하라**는 게 공식 권고다.
더 유연하면서 효율은 같거나 더 나은 경우가 많기 때문이다.

## Enum — 고정·안정적인 범주값에

Enum은 미리 정해둔 범주값을 정수로 저장하는 타입이다.

`Enum8`은 내부적으로 `Int8`과 똑같이 저장된다 — 최대 256개, 값 범위 [-128, 127].
`Enum16`은 `Int16` — 최대 65,536개, [-32768, 32767].
즉 `'pending'`, `'paid'` 같은 라벨을 쓰지만 디스크에는 작은 정수 하나만 깔린다.
ORDER BY나 GROUP BY에서도 대응하는 정수처럼 동작해서 효율적이다.

```sql
CREATE TABLE orders
(
    order_id  UInt64,
    -- 고정·안정적 범주: Enum8 (최대 256개, [-128,127])
    status    Enum8('pending' = 1, 'paid' = 2, 'shipped' = 3, 'cancelled' = 4)
)
ENGINE = MergeTree
ORDER BY order_id;
-- 주의: 기존 멤버의 숫자값 변경은 예외. 값 추가/삭제는 ALTER로 안전.
```

Enum의 함정은 **변경에 약하다**는 점이다.
이미 정의된 멤버의 숫자 값을 바꾸려 하면 예외가 난다(값을 추가하거나 삭제하는 건 안전하다).
또 Enum 자체는 NULL을 지원하지 않는다(`Nullable(Enum)` 형태로는 가능).

그래서 결론은 이렇다.
**값 집합이 거의 안 바뀌는 안정적 범주**(주문 상태 같은 것)면 Enum이 깔끔하다.
값이 자주 늘어나거나 종류가 많고 유동적이면 LowCardinality(String)이 낫다.

## FixedString — 길이가 정확히 정해진 값

`FixedString(N)`은 정확히 N **바이트**(문자가 아니라 바이트) 고정 길이 문자열이다.
값이 N보다 짧으면 null 바이트로 패딩하고, N을 넘으면 "Too large value" 예외가 난다.
`length()`는 실제 내용과 무관하게 항상 N을 돌려준다.

여기서 핵심은 **길이가 진짜로 고정일 때만 쓴다**는 것.
값 길이가 들쭉날쭉하면 짧은 값마다 패딩으로 자리를 낭비한다.

| 적합한 경우 | 예 |
|---|---|
| IPv6 주소 | `FixedString(16)` |
| 통화 코드 | `USD`, `KRW` → `FixedString(3)` |
| 언어 코드 | `en_US` 같은 고정폭 코드 |
| 고정 길이 해시 | MD5 16바이트, SHA256 32바이트 |

길이가 가변이면 FixedString 대신 LowCardinality(String)을 쓰는 게 낫고, UUID는 뒤에 볼 전용 UUID 타입을 쓰는 게 맞다.

## 숫자·시간·금액 — "딱 맞게 좁게"

베스트프랙티스 페이지가 권하는 타입 선택 원칙을 정리하면 이렇다.

| 원칙 | 이유 |
|---|---|
| String 대신 네이티브 숫자·날짜 타입 | 필터·집계 의미론이 정확해지고 압축에 유리 |
| 데이터 범위에 맞는 가장 좁은 정수폭 | 음수 없고 0~65535면 Int32 대신 UInt16 |
| 가능한 한 굵은 입도의 시간 타입 | 밀리초가 불필요하면 DateTime64 대신 DateTime |
| Nullable 회피 | NULL 마스크 파일 제거 |
| 저카디널리티엔 LowCardinality | 딕셔너리 인코딩으로 절감 |

시간 타입을 조금 더 보자.
`DateTime`은 **초(second)** 정밀도다.
`DateTime64(P)`는 서브초까지 — P=0~9이고 틱은 10^-P초다(3=밀리초, 6=마이크로초, 9=나노초, 기본 3).
내부적으로는 epoch 이후 틱 수를 Int64로 저장하고, 타임존은 컬럼 메타데이터에 저장된다(표시·파싱용이고 행 데이터엔 안 들어간다).

작은 주의점.
일반적으로 DateTime64의 표현 범위는 대략 1900~2299년이지만, **최대 정밀도(P=9, 나노초)** 에서는 상한이 2262-04-11로 내려간다.
나노초 정밀도가 정말 필요한 게 아니라면 굳이 P를 높이지 말고, 밀리초도 필요 없으면 DateTime을 쓰는 게 베스트프랙티스다.

금액 같은 **정확한 소수**에는 `Decimal(P, S)`을 쓴다.
P(정밀도, 1~76)는 총 자릿수, S(스케일, 0~P)는 소수 자릿수다.
P 범위에 따라 내부 저장 정수 타입이 결정된다.

| P 범위 | 내부 타입 |
|---|---|
| 1~9 | Decimal32 |
| 10~18 | Decimal64 |
| 19~38 | Decimal128 |
| 39~76 | Decimal256 |

예를 들어 `Decimal32(4)`는 -99999.9999 ~ 99999.9999를 0.0001 단위로 표현한다.

여기서 성능 주의가 하나 있다.
Decimal32/64는 최신 CPU에서 효율적이지만, **Decimal128/256 연산은 네이티브 128/256비트 정수 지원이 없어 에뮬레이트되며 "상당히 느리다"**(공식 문서 표현).
게다가 Decimal128/256은 오버플로 검사를 건너뛰어, 경고 없이 잘못된 결과가 나올 수 있다(`decimal_check_overflow` 설정으로 제어).
오버플로 시 소수부 초과 자릿수는 반올림이 아니라 **버림**이고, 정수부가 넘치면 예외다.
또 Decimal과 Float를 직접 연산할 수는 없어 명시적 캐스트가 필요하다.

> 참고로 "금액엔 Float 대신 Decimal" 이라는 권고 자체는 널리 통용되지만, 이 장에서 확인한 공식 베스트프랙티스 페이지에 직접 문구로 박혀 있지는 않았다. 여기서는 Decimal의 정밀도 보장과 Float의 부동소수 오차라는 일반 통념 선에서만 말해 둔다.

## 복합 타입 — Map과 Nested

값 하나가 아니라 "값들의 묶음"을 한 행에 담아야 할 때가 있다.

### Map(K, V) — 편하지만 조회는 선형 스캔

`Map(K, V)`는 키-값 쌍을 담는다.
다만 이름과 달리 **진짜 해시맵이 아니다.**
내부적으로 `Array(Tuple(K, V))`로 저장된다.

그래서 `m['key']` 조회는 맵 전체를 훑는다 — 런타임이 맵 크기에 **선형(O(n))** 이다.
키가 몇 개뿐이면 문제없지만, 한 맵에 키가 수천 개씩 들어가면 단일 키 조회가 느려질 수 있다.
(키가 매우 많을 때는 키 해시 기반으로 버킷을 나누는 `with_buckets` 직렬화로 단일 키 조회를 크게 가속할 수 있다고 공식 문서가 안내한다.)

```sql
CREATE TABLE logs
(
    ts    DateTime CODEC(DoubleDelta, ZSTD(1)),
    -- 균일한 값 타입의 유연 키-값. 키가 매우 많으면 with_buckets 직렬화 고려
    attrs Map(LowCardinality(String), String)
)
ENGINE = MergeTree
ORDER BY ts;

-- 조회 예: 단일 키 -> 맵 전체 스캔 발생
SELECT attrs['http_status'] FROM logs;
```

값 타입이 균일하면 Map이 적합하고, 행마다 키별 값 타입까지 제각각인 유연한 스키마라면 JSON 타입을 고려한다.

### Nested — "셀 안의 작은 테이블"

`Nested`는 한 행 안에 작은 테이블을 넣는 느낌의 구조다.

**비유.** (이해용 비유다.)
한 페이지(행) 안에 "이 페이지에서 달성한 목표 목록"이라는 작은 표를 끼워 넣는 것 — 목표 ID 열, 값 열이 나란히.

실제 구현은 그 비유와 가깝다.
Nested는 내부적으로 **길이가 같은 여러 컬럼 배열의 묶음**으로 저장된다(사실상 Array들의 집합).
한 행의 모든 서브컬럼 배열은 길이가 같아야 하고, 안 맞으면 오류가 난다.

```sql
CREATE TABLE page_views
(
    page_id UInt64,
    -- 한 행에 여러 goal — 내부적으로 Goals.id / Goals.value 두 배열(길이 동일)
    Goals Nested
    (
        id    UInt32,
        value Decimal64(2)
    )
)
ENGINE = MergeTree
ORDER BY page_id;

-- 개별 요소 필터: 스칼라 비교 불가 -> ARRAY JOIN으로 펼침
SELECT page_id, Goals.id, Goals.value
FROM page_views
ARRAY JOIN Goals
WHERE Goals.id = 591325;
```

점 표기(`Goals.id`)로 삽입·조회하고, 개별 요소를 필터링할 땐 `ARRAY JOIN`으로 펼친다.
Nested 컬럼은 Array라서 스칼라와 직접 비교가 안 되므로, `has()`·`arrayExists()`·`length()`·`notEmpty()` 같은 배열 함수나 ARRAY JOIN을 쓴다.

## UUID — 16바이트지만 "어디를 보고 정렬하나"가 중요

`UUID`는 16바이트(128비트) 값이다.
연산 면에서는 String이 지원하는 것(min/max/count)만 되고, 합·평균 같은 산술은 안 된다.
생성은 `generateUUIDv4()`를 쓴다.

한 가지 함정이 있다.
UUID는 역사적 설계로 값의 **"두 번째 절반" 기준으로 정렬**된다.
그래서 앞 절반에 타임스탬프가 들어가는 UUIDv7을 기본키에 그대로 쓰면, 정렬이 뒤쪽(카운터 부분)을 따라가게 되어 비효율적이고 불필요한 디스크 접근을 유발할 수 있다.
UUIDv7을 시간순 정렬로 활용하려면 기본키를 이렇게 우회한다.

```sql
-- UUIDv7의 앞부분(타임스탬프)을 기준으로 정렬되게
PRIMARY KEY (UUIDv7ToDateTime(uuid))
-- 또는 시간 입도를 굵게
PRIMARY KEY (toStartOfHour(UUIDv7ToDateTime(uuid)))
```

## 스키마 전체를 보는 두 가지 결정 — ORDER BY와 비정규화

타입을 다 골랐어도, 스키마 설계엔 두 가지 큰 결정이 남는다.

### ORDER BY 컬럼 순서

4장에서 ORDER BY가 희소 인덱스이자 디스크 물리 정렬을 동시에 결정한다고 했다.
그래서 **컬럼 순서**가 압축률과 쿼리 속도를 좌우한다.
공식 가이드의 권장 순서는 이렇다.

| 순위 | 기준 |
|---|---|
| ① | WHERE에서 자주, 많은 행을 걸러내는 컬럼을 앞에 |
| ② | 서로 상관 높은 컬럼을 인접 배치 (압축률↑) |
| ③ | 대체로 카디널리티 오름차순(저카디널리티 먼저) — 단 필터 효율과 균형 |

공식 예시에서 posts 테이블에 카디널리티 8짜리 `PostTypeId`를 먼저 두고 그다음 `toDate(CreationDate)`로 정렬했더니 데모상 쿼리가 약 3배 개선됐다.
4장에서 본 "저카디널리티 먼저" 원칙과 정확히 같은 이야기다.

### 비정규화 — 조인을 줄여라

마지막으로 큰 그림.
RDB는 정규화를 미덕으로 여기지만, ClickHouse는 **비정규화**를 권한다.
조인을 쿼리 시점에서 삽입 시점으로 옮기라는 것이다.

- 1:1이 아닌 관계는 복합 타입(Array/Map/Nested)으로 한 행에 담는다.
- 조인이 꼭 필요하면 Dictionary, Materialized View, Refreshable MV를 대안으로 쓴다(MV는 6장에서 본다).
- ClickHouse엔 **외래키 개념이 없다.** 참조 무결성은 애플리케이션 책임이다.

"왜 이렇게까지?"
ClickHouse는 분석 쿼리에서 수억 행을 스캔하는 게 일상이다.
그 와중에 매 쿼리마다 여러 테이블을 조인하면 비용이 폭증한다.
그래서 "쓸 때 한 번 고생해서 합쳐 두고, 읽을 때 편하게"라는 OLAP의 전형적 트레이드오프를 택한 것이다.

## 보너스: 압축 CODEC — 타입과 함께 가는 설계

타입을 좁게 잡는 것만큼 중요한 게, 컬럼별로 **압축 코덱(CODEC)** 을 지정하는 것이다.
컬럼 정의에 `CODEC(...)`으로 붙이며, 파이프라인으로 결합할 수 있다(예: `CODEC(Delta, ZSTD)`).
이때 압축 코덱은 암호화 코덱보다 **앞에** 와야 한다 — 암호화된 데이터는 사실상 더 압축되지 않기 때문이다.

코덱은 크게 두 부류다.
하나는 **전처리(인코딩) 코덱** — 데이터를 압축기에 넣기 전에 더 잘 줄어들게 변형한다.
다른 하나는 **범용 압축 코덱** — LZ4, ZSTD 같은 실제 압축기.

| 코덱 | 성격 | 잘 맞는 데이터 |
|---|---|---|
| Delta | 이웃 값의 차분 | 천천히 변하는 수열 |
| DoubleDelta | 델타의 델타 | 단조 증가열(타임스탬프 등) |
| Gorilla | 부동소수 XOR 기반 | 시계열 부동소수 게이지 |
| T64 | 정수의 미사용 상위 비트 제거 | 좁은 범위/희소 정수 |
| GCD | 최대공약수로 나눔 | 공약수가 있는 정수열 |
| LZ4 | 범용·빠른 해제 (기본) | 대부분 |
| ZSTD(1~22) | 범용·높은 압축률 | 대부분 (권장 기본) |

선택 가이드는 의외로 단순하다.
**대부분의 타입엔 `ZSTD(1)`을 기본으로** 두라(최고 수준 압축률에 해제도 일관되게 빠르다 — ClickHouse Cloud의 기본도 ZSTD(1)).
LZ4는 ZSTD와 압축률이 비슷할 때, 해제가 더 빠르고 CPU가 덜 드는 상황에서만 고른다.
그 위에 데이터 성격에 맞는 전처리 코덱을 얹는다.

- 타임스탬프·단조 증가 정수열 → `Delta` 또는 `DoubleDelta` + ZSTD ("Delta가 ZSTD를 개선한다"는 조합이 특히 효과적이다)
- 랜덤 스파이크가 섞인 부동소수 게이지 → `Gorilla`
- 값이 좁은 범위거나 희소한 정수 → `T64` (단, 랜덤 분포에는 부적합)
- 분포가 애매하면 Gorilla와 T64를 둘 다 테스트해 보라

이 모든 걸 한 테이블에 모아 보면 이렇게 된다.

```sql
CREATE TABLE events
(
    -- 좁은 정수폭: 0~255면 UInt8
    event_type      UInt8,
    -- 저카디널리티 문자열(~1만 미만): 딕셔너리 인코딩
    country         LowCardinality(String),
    -- 고정 길이: 통화코드 3바이트
    currency        FixedString(3),
    -- Nullable 대신 기본값 사용 (별도 null-mask 파일 회피)
    user_id         UInt64 DEFAULT 0,
    -- 금액: 정확도 위해 Decimal(정밀도 18, 소수 2자리 -> Decimal64)
    amount          Decimal64(2),
    -- 타임스탬프: 단조 증가 -> DoubleDelta + ZSTD
    ts              DateTime CODEC(DoubleDelta, ZSTD(1)),
    -- 천천히 변하는 부동소수 게이지 -> Gorilla
    temperature     Float32 CODEC(Gorilla, ZSTD(1)),
    -- 좁은 범위/희소 정수 -> T64
    seq             UInt32 CODEC(T64, ZSTD(1))
)
ENGINE = MergeTree
-- 저카디널리티(event_type) 먼저, 그다음 날짜, 마지막 고카디널리티
ORDER BY (event_type, toDate(ts), user_id);
```

이 한 장의 스키마에 이 장의 원칙이 다 들어 있다.
좁은 타입, LowCardinality, Nullable 회피, 성격에 맞는 CODEC, 그리고 저카디널리티 먼저인 ORDER BY.

## 정리하며

이 장의 한 줄 요약.
**ClickHouse에서 데이터타입 선택은 곧 성능 설계다.**

- Nullable은 별도 마스크 파일을 만들어 거의 항상 손해 → 기본값으로 회피.
- 반복 문자열은 LowCardinality(String) — 단 유니크 값이 ~1만 미만일 때 이득, ~10만 넘으면 역효과.
- 고정·안정 범주는 Enum, 길이 고정 값은 FixedString.
- 숫자·시간은 딱 맞게 좁게, 정확한 소수는 Decimal(단 128/256은 느림).
- 복합 타입 Map은 O(n) 스캔, Nested는 길이 같은 배열 묶음.
- ORDER BY는 저카디널리티·고선택성 먼저, 스키마는 비정규화 지향.
- CODEC은 ZSTD(1) 기본에 데이터 성격별 전처리 코덱(Delta/DoubleDelta/Gorilla/T64)을 얹기.

다음 6장에서는 이렇게 잘 설계한 테이블 위에서, 삽입과 동시에 집계를 미리 계산해 두는 **Materialized View** 를 본다.
"읽을 때 편하게"라는 OLAP 철학이 한 단계 더 나아가는 지점이다.
