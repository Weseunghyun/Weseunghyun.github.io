2장에서 클러스터의 뼈대를 봤다.
이제 그 위에서 우리가 실제로 다루는 대상으로 들어간다.

이 장은 의외의 사실 하나에서 시작한다.
**쿠버네티스에서 배포의 최소 단위는 "컨테이너"가 아니다.**
그럼 무엇인가?
— **Pod**다.

이게 무슨 뜻인지, 그리고 Pod 위에 얹히는 Deployment·StatefulSet 같은 것들이 무엇인지 풀어보자.

## 3.1 Pod — 배포의 최소 단위

쿠버네티스는 컨테이너를 직접 다루지 않는다.
컨테이너를 **Pod**라는 껍데기로 한 번 감싸서 다룬다.

> **Pod = 한 개(또는 몇 개)의 컨테이너 + 그들이 공유하는 자원(네트워크·스토리지)의 묶음.**

대부분의 경우 **Pod 하나에 컨테이너 하나**다.
그럼 "왜 굳이 한 겹 더 감쌌지?"라는 의문이 들 것이다.

이유는 가끔 **여러 컨테이너가 한 몸처럼 붙어 있어야** 할 때가 있기 때문이다.
예를 들어 메인 앱 옆에 로그를 수집하는 작은 보조 컨테이너를 딱 붙여두는 경우 (이런 보조를 **사이드카**라고 부른다).
이 둘은 늘 같은 곳에서, 같이 태어나고 같이 죽어야 한다.
Pod가 그 "한 몸"의 단위다.

🔴 한 Pod 안의 컨테이너들은 **같은 네트워크를 공유**한다.
같은 IP를 쓰고, 서로를 `localhost`로 부른다.
(Docker 2장에서 본 network namespace를, 쿠버네티스는 Pod 단위로 묶어준다고 생각하면 된다.)

## 3.2 🔴 Pod는 일회용이다 — 이게 모든 걸 바꾼다

Pod에 대해 가장 중요한 사실은 이것이다.

> **Pod는 일회용(ephemeral)이다.** 죽으면 되살아나지 않는다. 대신 완전히 새로운 Pod가, 새로운 IP로 태어난다.

이게 직관과 어긋난다.
서버는 보통 "고치고 다시 쓰는" 대상인데, Pod는 "버리고 새로 만드는" 대상이다.

비유하면 이렇다.
Pod는 애완동물이 아니라 **가축**이다.
애완동물은 아프면 정성껏 치료한다 (이름이 있고, 고유하다).
가축은 한 마리가 병들면 그 자리를 새 개체로 채운다 (개체보다 무리의 수가 중요하다).

이 "가축처럼 다룬다"는 철학이 쿠버네티스의 자가치유(1장)와 직결된다.
Pod 하나가 죽어도 시스템은 당황하지 않는다.
"원하는 개수"에 맞춰 새 Pod를 띄우면 그만이다.

하지만 여기서 문제가 하나 생긴다.
**Pod가 죽을 때마다 IP가 바뀐다면, 다른 앱은 그 Pod를 어떻게 찾아 통신하지?**
이 문제의 답이 4장의 Service다.
지금은 "Pod는 IP가 자꾸 바뀌는 일회용"이라는 것만 기억하자.

## 3.3 우리는 Pod를 직접 만들지 않는다

Pod가 일회용이라는 건, 우리가 Pod를 **직접 하나씩 만들지 않는다**는 뜻이다.
직접 만든 Pod는 죽으면 그걸로 끝이다 (아무도 다시 안 띄워준다).

대신 우리는 **"이 Pod를 이렇게 유지하라"고 관리해주는 상위 오브젝트**를 만든다.
이걸 **워크로드 컨트롤러**라고 부른다.
종류별로 정리하면 이렇다.

| 오브젝트 | 하는 일 | 언제 쓰나 |
|---|---|---|
| **Deployment** | Pod를 N개 유지 + 무중단 업데이트/롤백 | **무상태 앱**(웹·API) — 가장 흔함 |
| **StatefulSet** | 안정적 이름·순서·전용 저장소 | **상태 앱**(데이터베이스 등) |
| **DaemonSet** | 모든 node에 1개씩 | 로그 수집·모니터링 에이전트 |
| **Job / CronJob** | 1회 완료 작업 / 정기 작업 | 배치 처리·크론 작업 |

가장 많이 쓰는 둘, Deployment와 StatefulSet을 자세히 보자.

## 3.4 Deployment — 무상태 앱의 기본

**Deployment**는 가장 흔히 쓰는 워크로드다.
하는 일은 두 가지다.
1. "이 앱을 N개로 유지하라" (1장의 그 선언).
2. 새 버전 배포 시 **무중단으로 갈아끼우기**(롤링 업데이트)와 **되돌리기**(롤백).

여기서 "무상태(stateless)"라는 말이 중요하다.
무상태 앱이란, **각 Pod가 서로 구별될 필요가 없는** 앱이다.
웹 서버를 떠올려보자.
3개의 웹 서버 Pod는 완전히 똑같다.
어느 Pod가 요청을 처리하든 결과가 같다.
하나가 죽으면 아무거나 새로 띄워도 된다 (3.2의 "가축").

이런 앱에 Deployment가 딱이다.
무중단 배포도 쉽다 — 새 버전 Pod를 하나씩 늘리고 옛 버전을 하나씩 줄이면, 서비스가 멈추지 않는다 (7장에서 자세히).

## 3.5 🔴 StatefulSet — 상태 앱은 다르다

그런데 데이터베이스는 다르다.
DB를 3개 띄운다고 할 때, 이 셋은 **서로 구별되어야** 한다.
- 어느 게 주(primary)이고 어느 게 복제본인지.
- 각자 **자기만의 데이터**를 갖는다 (1번 DB의 데이터와 2번 DB의 데이터는 다르다).

이건 "가축"처럼 다룰 수 없다.
2번 DB가 죽었을 때 아무 새 Pod나 띄우면 안 되고, **"2번"이라는 정체성과 "2번의 데이터"를 그대로 이어받은** Pod가 떠야 한다.

이걸 위한 게 **StatefulSet**이다.
StatefulSet은 Pod에게 세 가지를 보장한다.
- **안정적인 이름**: `db-0`, `db-1`, `db-2` (랜덤이 아니라 고정).
- **순서 보장**: 0번부터 차례로 생성하고, 역순으로 삭제.
- **전용 저장소**: 각 Pod가 자기만의 볼륨에 항상 다시 연결됨 (5장의 PVC).

그래서 데이터베이스, 메시지 큐(Kafka 등) 같은 상태 앱에 StatefulSet을 쓴다.

🔴 다만 솔직히 말하면, **상태 앱을 쿠버네티스에 올리는 건 난도가 높다.**
데이터의 안전, 백업, 장애 복구가 까다롭기 때문이다.
그래서 실무에서는 DB는 차라리 클라우드의 관리형 서비스(RDS 등)를 쓰고, K8s에는 무상태 앱만 올리는 선택도 흔하다.

## 3.6 label과 selector — 느슨하게 엮는 법

여기서 쿠버네티스의 또 하나의 핵심 발상이 나온다.
오브젝트들을 어떻게 서로 연결할까?

쿠버네티스는 **고정된 ID로 직접 엮지 않는다.**
대신 **label(꼬리표)** 을 붙이고 **selector(고르개)** 로 고른다.

예를 들어 Pod에 `app: web`이라는 꼬리표를 붙인다.
- Deployment는 말한다: "`app: web` 꼬리표가 붙은 Pod를 3개 유지하라."
- Service는 말한다(4장): "`app: web` 꼬리표가 붙은 Pod들에게 트래픽을 보내라."

이게 왜 좋을까?
**느슨한 연결** 덕분이다.
Pod가 죽고 새로 떠도(IP가 바뀌어도), 새 Pod에 같은 꼬리표만 있으면 Deployment도 Service도 자동으로 그걸 포함한다.
이름이나 IP 같은 고정된 것에 묶이지 않으니, 가축처럼 다루는 철학과 완벽히 맞아떨어진다.

이 label-selector는 쿠버네티스 전체를 잇는 **접착제**다.
앞으로 계속 나온다.

## 3.7 매니페스트로 보기

지금까지를 YAML 하나로 보자.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
spec:
  replicas: 3                          # 3개 유지 (1장의 선언)
  selector:
    matchLabels:
      app: web                         # 이 꼬리표의 Pod를 관리
  template:                            # 만들 Pod의 틀
    metadata:
      labels:
        app: web                       # Pod에 이 꼬리표를 붙임
    spec:
      containers:
        - name: web
          image: myapp:1.2             # 버전 고정 권장 (Docker 4장)
          ports:
            - containerPort: 8080
```

읽어보면 이렇다.
"`web`이라는 Deployment를 만든다.
`app: web` 꼬리표가 붙은 Pod를 **3개** 유지하는데, 그 Pod는 `myapp:1.2` 이미지로 만든다."

`kind`가 오브젝트 종류, `spec`이 원하는 상태.
이걸 `kubectl apply`로 제출하면(7장), 2장에서 본 그 여정이 시작된다.

## 3.8 정리

이 장을 한 문장으로 줄이면 이렇다.

> 배포의 단위는 컨테이너가 아니라 **Pod**(컨테이너+공유 자원)이고, Pod는 **일회용(가축)** 이라 직접 만들지 않고 **Deployment**(무상태)·**StatefulSet**(상태) 같은 컨트롤러로 유지하며, 무엇을 묶을지는 **label·selector**로 느슨하게 연결한다.

이제 "Pod는 죽으면 새 IP로 다시 뜬다"는 사실에서 생긴 숙제가 남아 있다.
그렇게 자꾸 바뀌는 Pod를, 다른 앱은 어떻게 찾아서 통신할까?
4장에서 그 답, **Service와 네트워킹**으로 간다.
