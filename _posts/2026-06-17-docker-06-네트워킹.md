지금까지는 컨테이너 하나를 만들고 띄우는 이야기였다.
하지만 실제 서비스는 컨테이너 하나로 끝나지 않는다.
웹 서버 컨테이너가 데이터베이스 컨테이너에 접속해야 하고, 바깥세상에서 그 웹 서버에 들어와야 한다.

이 장의 질문은 둘이다.
**"컨테이너끼리 어떻게 통신하나?"** 그리고 **"바깥에서 컨테이너에 어떻게 들어오나?"**

## 6.1 컨테이너는 각자 자기만의 네트워크를 가진다

2장에서 NET namespace를 기억하는가.
각 컨테이너는 **자기만의 네트워크 세계**를 가진다 — 자기 IP 주소, 자기 네트워크 카드, 자기 포트.

그래서 컨테이너 안에서 보면 마치 독립된 컴퓨터처럼 보인다.
문제는 이 독립된 세계들을 어떻게 연결하느냐다.
Docker는 이걸 가상의 네트워크 장비로 푼다.

## 6.2 네트워크 드라이버 — 연결 방식 다섯 가지

Docker는 컨테이너를 연결하는 방식(드라이버)을 여러 개 제공한다.

| 드라이버 | 동작 | 언제 |
|---|---|---|
| **bridge** (기본) | 가상 다리에 컨테이너를 연결, 바깥은 주소 변환(NAT) | 한 서버 안 기본 |
| **host** | 호스트 네트워크를 그대로 사용 (격리 없음) | 성능 극대화, 포트 충돌 위험 |
| **none** | 네트워크 없음 | 완전 격리 |
| **overlay** | 여러 서버에 걸친 가상 네트워크 | 여러 대 묶을 때 |
| **macvlan** | 컨테이너에 실제 망의 주소를 직접 부여 | 특수·레거시 |

대부분의 경우 기본인 **bridge**를 쓴다.
이게 실제로 뭘 하는지 보자.

## 6.3 bridge가 실제로 하는 일

bridge 네트워크는 호스트 안에 **가상의 다리(브리지)** 를 하나 놓는다.
그리고 컨테이너마다 **가상 랜선(veth)** 을 하나씩 만들어, 한쪽 끝은 컨테이너 안에, 다른 쪽 끝은 그 다리에 꽂는다.

같은 다리에 연결된 컨테이너끼리는 서로 통신할 수 있다.
컨테이너가 바깥 인터넷으로 나갈 때는, 호스트가 주소를 자기 것으로 바꿔주는 **NAT**(주소 변환)를 거친다.
(집의 공유기가 여러 기기를 하나의 공인 IP로 내보내는 것과 같은 원리다.)

## 6.4 🔴 컨테이너 이름으로 통신하기 — 사용자 정의 네트워크

여기 초보자가 자주 막히는 함정이 있다.

Docker가 기본으로 주는 `bridge` 네트워크에서는, 컨테이너끼리 **이름으로 서로를 못 찾는다**.
IP 주소로만 통신할 수 있는데, 컨테이너 IP는 띄울 때마다 바뀌니 사실상 쓰기 어렵다.

해결책은 **사용자 정의 네트워크**를 직접 만드는 것이다.

```bash
docker network create mynet
docker run -d --name db --network mynet postgres
docker run -d --name api --network mynet myapp
```

이렇게 만든 네트워크에는 **내장 DNS**가 붙는다.
그래서 `api` 컨테이너가 `db`라는 **이름으로** 데이터베이스에 접속할 수 있다 (`postgres://db:5432`).
IP를 몰라도 이름만으로 찾아간다.

> 규칙: 여러 컨테이너를 연결할 거면 **항상 사용자 정의 네트워크**를 쓴다. 그래야 서비스 이름으로 통신된다.

다행히 8장에서 볼 Docker Compose는 이 사용자 정의 네트워크를 **자동으로 만들어준다**.
그래서 Compose에서는 서비스 이름으로 바로 통신이 된다.

## 6.5 포트 퍼블리시 — 바깥에서 들어오기

컨테이너끼리 통신은 됐다.
이제 바깥세상(내 브라우저, 외부 사용자)에서 컨테이너 안 웹 서버로 들어가야 한다.

기본적으로 컨테이너의 포트는 **바깥에서 안 보인다** (자기만의 네트워크 세계 안에 있으니까).
바깥으로 열려면 **포트 퍼블리시**를 해야 한다.

```bash
docker run -p 8080:80 nginx
```

`-p 8080:80`은 "호스트의 8080 포트로 들어온 요청을 컨테이너의 80 포트로 전달하라"는 뜻이다.
Docker가 이걸 위해 호스트의 방화벽 규칙(iptables)에 전달 규칙을 넣는다.

## 6.6 🔴 포트 퍼블리시의 함정 — 방화벽을 우회한다

여기 운영에서 사고를 부르는 중요한 함정이 있다.

방금 말한 대로 `-p`는 Docker가 호스트의 iptables를 **직접 조작**해서 작동한다.
문제는 이 규칙이 우리가 흔히 쓰는 방화벽(UFW 등)의 규칙보다 **먼저** 적용될 수 있다는 것이다.

무슨 뜻이냐면 — UFW로 "이 포트는 막아둠"이라고 설정해 놨어도, `-p`로 연 포트는 그 막음을 **건너뛰고 외부에 노출**될 수 있다.
데이터베이스를 `-p 5432:5432`로 무심코 열었다가 인터넷 전체에 DB가 노출되는 사고가 실제로 흔하다.

예방책은 이렇다.
- 바깥에 노출할 필요 없는 컨테이너는 **`-p`를 아예 안 쓴다** (같은 네트워크 안에서만 접근됨).
- 로컬에서만 접근하면 되면 `-p 127.0.0.1:5432:5432`처럼 **바인드 주소를 제한**한다 (이러면 외부에서 못 들어옴).

> 핵심: `-p`로 포트를 여는 것은 "방화벽에 구멍을 뚫는 것"이다. 정말 바깥에 열어야 하는 포트만 신중하게 연다.

## 6.7 정리

이 장을 한 문장으로 줄이면 이렇다.

> 컨테이너는 각자 NET namespace로 독립 네트워크를 갖고, **사용자 정의 bridge 네트워크**로 묶으면 서비스 이름으로 통신되며(내장 DNS), 바깥 노출은 **`-p` 포트 퍼블리시**로 하되 이게 방화벽을 우회할 수 있으니 노출 범위를 신중히 정해야 한다.

이제 통신은 됐다.
그런데 4장에서 미뤄둔 숙제가 있다 — "컨테이너를 지우면 데이터가 사라진다"는 문제.
데이터를 어떻게 살릴 것인가?
7장에서 볼륨으로 간다.
