## 6.1 운영

- **retention**: compactor가 작은 block을 병합하고 보관 기간 지난 block을 삭제한다(`block_retention`).
  기본값은 배포에 따라 다르니 환경에서 확인한다.
- **멀티테넌시**: `X-Scope-OrgID` 헤더로 테넌트를 가른다.
- **limit**: `max_bytes_per_trace`(단일 trace 최대)·ingestion rate 등으로 폭주를 막는다.
  3.0은 overrides 포맷이 바뀌었다(legacy 폐기).

## 6.2 터지는 지점들

1. **속성 검색이 느리다**: TraceQL 속성 검색은 시간범위 내 block을 스캔한다.
   볼륨이 크면 느리다 — 반면 **trace ID 직접 조회는 빠르다**(3장).
   자주 쓰는 속성은 dedicated column으로.
2. **스코프 생략 패널티**: `span.`·`resource.` 스코프를 빼면 여러 곳을 확인해 느려진다(4장).
3. **metrics-generator 카디널리티 폭발**: 동적 span 이름·고유 attribute를 메트릭 label로(5장).
4. **object storage 비용·요청 수**: 검색이 다수 block 객체를 읽어 S3/GCS 요청·전송 비용이 든다.
   3.0의 RF3→RF1이 저장 비용을 직접 줄인다.
5. **샘플링 오해**: Tempo는 샘플링 안 한다 — 너무 많이 들어오면 SDK/Collector에서 샘플링해야 한다(1장).

## 6.3 개념 체크리스트

- [ ] Tempo가 왜 싼가 (object storage 전용, 인덱스 DB 없음)
- [ ] "무인덱스"의 정확한 의미 (block 내 bloom+index는 있음)
- [ ] trace ID 조회(빠름) vs 속성 검색(스캔)
- [ ] 3.0 Project Rhythm 변화 (RF3→RF1, LiveStore/block-builder)
- [ ] TraceQL의 structural operator (`>>`·`>`·`~`)
- [ ] metrics-generator(RED·service graph) vs TraceQL metrics
- [ ] **샘플링은 Tempo가 아니라 SDK/Collector**

## 6.4 면접 Q&A

**Q.
Tempo는 왜 싼가요?**
인덱스 DB(ES·Cassandra) 없이 오브젝트 스토리지만으로 저장하기 때문입니다.
운영할 별도 DB가 없어 비용·복잡도가 낮습니다.
대신 속성 검색은 block 스캔이라 인덱스 백엔드보다 느릴 수 있어, 자주 쓰는 속성을 dedicated column으로 가속합니다.

**Q.
"Tempo는 인덱스가 없다"가 맞나요?**
외부 인덱스 DB가 필요 없다는 뜻이지, 인덱스가 0인 건 아닙니다.
block 안에 trace ID를 찾는 bloom filter와 index가 있어서 trace ID 조회는 빠릅니다.

**Q.
트레이스를 다 저장하면 비싸지 않나요?**
Tempo 자체는 샘플링하지 않고 받은 걸 다 저장합니다.
양을 줄이려면 SDK나 OTel Collector에서 head/tail 샘플링을 합니다.

**Q.
트레이스에서 메트릭을 어떻게 얻나요?**
metrics-generator가 span에서 RED 메트릭·service graph를 만들어 Prometheus/Mimir로 remote write하거나, TraceQL metrics로 쿼리 시점에 ad-hoc 시계열을 만듭니다.
둘 다 동적 값을 label로 노출하면 카디널리티가 폭발하니 주의합니다.

## 6.5 마치며

Tempo의 모든 것은 한 결정에서 나온다.
**인덱스 DB 없이 오브젝트 스토리지만 쓴다.**
그래서 싸고(인덱스 DB 운영비 0), trace ID 조회는 빠르고(block 내 bloom), 속성 검색은 스캔이라 dedicated column으로 보완한다.
2026년 3.0은 여기에 RF1으로 저장 비용을 한 번 더 줄였다.

이 한 줄을 기억하면 "왜 이 트레이스 검색이 느리지?" 같은 문제의 출발점이 잡힌다.
