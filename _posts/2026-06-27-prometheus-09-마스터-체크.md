## 9.1 터지는 지점들

Prometheus 사고는 거의 한 줄기에서 갈라진다.

> **카디널리티 → 메모리 → OOM.**

1. **카디널리티 폭발 → OOM → 관측 상실**: `user_id` 같은 무한 값을 레이블로 → 시계열 곱 폭발 → head block 메모리가 터져 Prometheus가 죽는다.
   장애 보려고 켠 게 장애의 원인이 된다(6장).
2. **WAL replay = 재시작 지연**: 시계열이 많으면 WAL이 커서 재시작이 몇 분씩 걸린다.
   그 사이가 사각지대 → HA pair로 메운다(4·8장).
3. **scrape 한도·타이밍**: `sample_limit`이 너무 낮으면 스크레이프 통째 실패, `scrape_timeout > scrape_interval`이면 겹친다(3장).
4. **PromQL 함정이 알림을 망친다**: sum 먼저 하면 counter reset을 놓쳐 거짓 알림, irate는 들쭉날쭉해 플래핑, histogram에서 `le` 손실로 분위수가 깨진다(5장).
5. **staleness 오해**: 대상이 사라지면 값이 "없음"이 된다(0이 아니라).
   Pushgateway에 push한 건 낡음 처리가 안 돼 죽은 잡이 정상처럼 보인다(3장).
6. **remote write backpressure**: 원격 장기저장이 느려지면 shard 큐가 차고 WAL 읽기를 막아 로컬이 압박받는다(8장).

핵심 교훈 한 줄: **레이블에 무한한 값을 넣지 마라.** 거의 모든 사고가 여기서 시작한다.

## 9.2 개념 체크리스트

스스로 설명할 수 있는지 점검해보자.

- [ ] Prometheus가 왜 **긁어오는(pull)** 방식인지, 그 장단점
- [ ] 시계열 = 이름 + 레이블, metric 4타입과 native histogram
- [ ] **histogram으로 분위수를 구하는 이유**(집계 가능) vs summary
- [ ] 서비스 디스커버리와 **relabeling 두 단계**(긁기 전/저장 전)
- [ ] TSDB가 왜 **메모리(head)+디스크(block)** 2계층인지, WAL의 역할
- [ ] PromQL: instant/range vector, **rate 먼저 sum 나중**, `histogram_quantile`+`by(le)`, vector matching
- [ ] **카디널리티**가 무엇이고 왜 OOM으로 이어지는지
- [ ] recording rule vs alerting rule, **`for`**의 역할
- [ ] HA pair·federation·remote write·agent mode가 각각 푸는 문제
- [ ] 장기저장 4종의 계보(Prometheus 확장 vs 별도 엔진)

## 9.3 면접 Q&A

**Q.
Prometheus는 왜 push가 아니라 pull인가요?**
긁으러 갔는데 응답이 없으면 대상이 죽었다는 걸 바로 알 수 있고(헬스체크 내장), 수집 주기를 중앙에서 통제하며, 누가 무엇을 긁는지 명확합니다.
단점은 단명 작업을 못 긁는 것인데 Pushgateway로 보완합니다.

**Q.
카디널리티가 왜 문제인가요?**
시계열 개수가 레이블 값들의 곱으로 정해지는데, `user_id` 같은 무한 값을 레이블로 넣으면 시계열이 폭발합니다.
Prometheus는 최근 데이터를 메모리(head block)에 두기 때문에 시계열 폭발이 곧 메모리 폭발(OOM)이고, 관측 시스템 자체가 죽습니다.
그래서 레이블은 값이 정해진 것만 쓰고, 무한한 차원은 로그나 트레이스로 보냅니다.

**Q.
rate를 먼저 하고 sum을 나중에 하는 이유는?**
sum을 먼저 하면 개별 타깃이 재시작해 카운터가 0으로 떨어진 걸 감지하지 못해 값이 망가집니다.
rate가 각자의 counter reset을 보정한 뒤 합쳐야 정확합니다.

**Q.
분위수(p99)는 어떻게 구하나요?**
histogram의 bucket에 `rate()`를 씌우고 `histogram_quantile()`로 계산합니다.
집계할 때 `le` 라벨을 반드시 남겨야 합니다.
summary가 아니라 histogram을 쓰는 이유는 여러 인스턴스의 분위수를 합쳐 전체 분위수를 낼 수 있기 때문입니다.

**Q.
Prometheus 한 대로 부족하면?**
가용성은 같은 설정 두 대(HA pair), 장기보존·전역쿼리는 remote write로 Thanos/Mimir/VictoriaMetrics에 위임합니다.
federation은 집계 메트릭 스냅샷을 끌어오는 용도지 장기저장 대체가 아닙니다.

## 9.4 마치며

Prometheus의 거의 모든 것은 두 결정에서 따라 나온다.
**긁어온다(pull)**, 그리고 **최근 데이터는 메모리에 둔다.**
전자가 헬스체크·서비스 디스커버리·Pushgateway를 설명하고, 후자가 빠른 쿼리·압축·그리고 카디널리티라는 아킬레스건을 설명한다.
