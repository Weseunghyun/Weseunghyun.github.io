LogQL은 두 얼굴을 가졌다.
로그를 필터·변환하는 **log query**(PromQL엔 없는 1급 기능)와, 로그를 숫자로 바꾸는 **metric query**(PromQL을 닮음)다.

## 4.1 🔴 log query 파이프라인

log query는 단계가 정해져 있다.

> 스트림 선택 `{}` → line filter → parser → label filter → line format

| 단계 | 문법 | 비고 |
|---|---|---|
| 스트림 선택 | `{app="api"}` | 색인된 레이블로 먼저 좁힌다 |
| line filter | `\|= "error"` · `!=` · `\|~`(regex) | "분산 grep" — 가능한 앞에 |
| parser | `\| json` · `\| logfmt` · `\| regexp` | 본문에서 필드 추출 |
| label filter | `\| status>=500` | 추출한 필드로 거르기 |
| line format | `\| line_format "{{.msg}}"` | 출력 가공 |

예: `{app="api"} |= "error" | json | status>=500`

가장 중요한 원칙 — **스트림 선택으로 최대한 좁히고, line filter는 앞쪽에.**
본문엔 색인이 없어서 line filter는 chunk를 풀어 훑기 때문이다(5·7장).

## 4.2 metric query

로그를 숫자로 바꿔 그래프·알림에 쓴다.

- 로그 라인에 직접: `rate({app="api"} |= "error" [5m])`, `count_over_time(...)`.
- 값을 추출해 집계: `| unwrap latency_ms`로 숫자를 꺼낸 뒤 `quantile_over_time(0.99, ...)`.
- `sum by (...)`·`topk` 같은 집계도 PromQL처럼 쓴다.

예: `sum by (route) (rate({app="api"} | json | status>=500 [5m]))`

## 4.3 PromQL과 닮은 점·다른 점

- **닮음**: metric query 문법(`rate`·`sum by`·`[5m]`)이 PromQL 차용.
  규칙(ruler)도 Prometheus 호환.
- **다름**: LogQL은 **log query**(파이프라인으로 raw 로그를 다루는 것)가 1급 기능이다.
  그리고 metric은 **로그를 런타임에 숫자로 바꾸는** 것이라, 미리 색인된 메트릭보다 비용이 크다.

## 4.4 정리

- **log query**: 스트림 선택 → line filter → parser → label filter.
  **먼저 좁히기**가 성능의 전부.
- **metric query**: `rate`·`unwrap`·`quantile_over_time`으로 로그를 숫자로.
- PromQL을 닮았지만 raw 로그를 다루는 log query가 LogQL의 고유함.
