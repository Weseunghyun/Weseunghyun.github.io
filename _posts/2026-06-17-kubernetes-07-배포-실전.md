여기까지 개념을 쌓았다.
이제 실제로 손을 움직이는 장이다.
**쿠버네티스를 실제로 어떻게 다루고, 어떻게 배포하고, 어떻게 무중단으로 새 버전을 올리는가?**

도구는 세 가지 — 명령 도구 **kubectl**, 그걸로 적용하는 **매니페스트**, 그리고 규모가 커질 때 쓰는 **Helm**.

## 7.1 kubectl — 클러스터와 대화하는 도구

쿠버네티스를 다루는 기본 명령 도구가 **kubectl**("큐브 컨트롤" 또는 "큐브 시티엘")이다.
2장에서 본 API 서버에 명령을 보내는 게 바로 이 도구다.

실무에서 가장 많이 쓰는 건 사실 배포가 아니라 **보기·진단**이다.

```bash
# 보기 · 진단 (운영의 대부분)
kubectl get pods -o wide          # Pod 목록 (어느 node·IP인지까지)
kubectl describe pod <이름>        # 그 Pod의 상세 + 이벤트 (왜 안 뜨는지 여기 다 나온다)
kubectl logs <pod> -f             # 로그 실시간 / --previous: 죽기 직전 로그
kubectl exec -it <pod> -- sh      # 컨테이너 안으로 들어가 셸 열기
kubectl get events --sort-by=.lastTimestamp   # 최근 무슨 일이 있었나
```

🔴 특히 `kubectl describe`와 `kubectl logs`는 문제 해결의 9할이다.
Pod가 안 뜰 때, `describe`의 맨 아래 **이벤트(Events)** 섹션에 이유가 거의 항상 적혀 있다 (자원 부족, 이미지 못 받음 등 — 8장).

## 7.2 선언형으로 배포하기 — `kubectl apply`

1장에서 쿠버네티스는 선언형이라고 했다.
배포도 선언형으로 한다.

원하는 상태를 YAML 매니페스트에 적고, 이 한 줄로 제출한다.

```bash
kubectl apply -f web.yaml
```

`apply`는 "이 상태가 되게 해줘"라는 뜻이다.
- 없으면 만든다.
- 있으면 차이나는 부분만 고친다.
- 이미 그 상태면 아무것도 안 한다 (멱등).

🔴 명령형 방식(`kubectl run`, `kubectl create`로 직접 띄우기)도 있지만, 그건 일회성·실험용이다.
운영은 반드시 **매니페스트를 git에 두고 `apply`** 하는 선언형으로 한다.
그래야 "현재 인프라 상태"가 코드로 남아 추적·복구가 된다.

## 7.3 한 서비스의 완전한 매니페스트

지금까지 배운 조각들(Deployment·Service·자원·probe·설정)을 다 합치면 이렇게 된다.
이 한 편이 실전 매니페스트의 표준 모습이다.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata: { name: web }
spec:
  replicas: 3
  selector:
    matchLabels: { app: web }
  template:
    metadata:
      labels: { app: web }
    spec:
      containers:
        - name: web
          image: myapp:1.2                 # 🔴 버전 고정 (latest 금지 — Docker 4장)
          ports:
            - containerPort: 8080
          resources:                        # 🔴 6장 — 스케줄링·OOM·HPA의 전제
            requests: { cpu: "100m", memory: "128Mi" }
            limits:   { cpu: "500m", memory: "256Mi" }
          readinessProbe:                   # 준비된 Pod에만 트래픽 (무중단의 핵심)
            httpGet: { path: /health, port: 8080 }
            initialDelaySeconds: 5
          livenessProbe:                    # 보수적으로 (오킬 방지 — 6장)
            httpGet: { path: /health, port: 8080 }
            initialDelaySeconds: 20
          envFrom:
            - configMapRef: { name: web-config }   # 설정 (5장)
            - secretRef:    { name: web-secret }
---
apiVersion: v1
kind: Service
metadata: { name: web }
spec:
  selector: { app: web }                    # 이 꼬리표 Pod로 라우팅 (4장)
  ports:
    - { port: 80, targetPort: 8080 }
```

읽으면 시리즈 전체가 한눈에 들어온다.
"web 앱을 3개(replicas) 유지하고, 자원을 정하고(requests/limits), 준비/생존을 검사하고(probe), 설정을 주입하고(configMap/secret), Service로 노출한다."

## 7.4 🔴 무중단 배포 — 롤링 업데이트

이제 이 시리즈에서 가장 실용적인 부분 중 하나.
**서비스를 멈추지 않고 새 버전으로 갈아끼우기.**

매니페스트의 이미지를 `myapp:1.2`에서 `myapp:1.3`으로 바꾸고 다시 `apply`하면, 쿠버네티스가 **롤링 업데이트**를 한다.

작동 방식은 이렇다.
1. 새 버전(1.3) Pod를 **하나 띄운다.**
2. 그 Pod가 **readiness probe를 통과**할 때까지 기다린다 (준비 완료).
3. 준비되면 옛 버전(1.2) Pod를 **하나 내린다.**
4. 이걸 **하나씩 반복**한다.

핵심은 항상 "준비된 Pod가 충분히 떠 있는 상태"가 유지된다는 것이다.
그래서 사용자는 서비스 중단을 느끼지 못한다.

![](/assets/img/posts/kubernetes-07-배포-실전/k8s-rolling-update.svg)

🔴 여기서 6장의 **readiness probe**가 왜 중요한지 다시 드러난다.
readiness가 없으면, 쿠버네티스는 새 Pod가 "떴다"는 것만 보고 바로 옛 Pod를 내린다.
하지만 새 Pod가 아직 준비 안 됐다면?
→ 그 사이 들어온 요청이 실패한다.
readiness가 있어야 "진짜 준비된 뒤에" 트래픽이 넘어가 무중단이 보장된다.

문제가 생기면 되돌리기도 한 줄이다.

```bash
kubectl rollout status deploy/web    # 배포 진행 상황 보기
kubectl rollout undo deploy/web      # 🔴 직전 버전으로 롤백
```

이 무중단 배포와 즉시 롤백이, 0장에서 말한 "쿠버네티스가 빛나는 순간"의 대표 사례다.

## 7.5 Helm — 매니페스트가 많아질 때

서비스가 늘면 매니페스트도 는다.
게다가 환경별로(개발·스테이징·운영) 값이 조금씩 다르다 (복제본 수, 자원 크기, 이미지 태그…).
같은 YAML을 환경마다 복사해 고치다 보면 관리가 엉킨다.

이걸 푸는 게 **Helm**이다.
Helm은 쿠버네티스의 **패키지 매니저**라고 불린다.
- 매니페스트를 **템플릿(차트)** 으로 만들고,
- 환경별로 다른 값만 `values.yaml`에 따로 둬서 주입한다.

덕분에 "개발용 values, 운영용 values"만 바꿔 같은 차트를 여러 환경에 배포한다.
또 공개된 차트로 흔한 앱(데이터베이스·모니터링 도구 등)을 한 줄로 설치할 수 있다.

```bash
helm install my-db bitnami/postgresql      # 공개 차트로 설치
helm upgrade my-app ./mychart -f prod.yaml # 운영 값으로 업그레이드
helm rollback my-app                        # 되돌리기
```

(템플릿 없이 "기본 매니페스트 + 환경별 패치"로 관리하는 **Kustomize**라는 대안도 있다.
취향·상황에 따라 선택한다.)

## 7.6 한 걸음 더 — GitOps

선언형의 자연스러운 귀결이 하나 있다.
매니페스트를 git에 두는 것까지 왔다면, 아예 **"git에 있는 상태 = 클러스터의 상태"** 로 자동 동기화하면 어떨까?

이게 **GitOps**다.
ArgoCD·Flux 같은 도구가 git 저장소를 지켜보다가, 거기 적힌 상태와 클러스터를 자동으로 맞춘다.
사람이 `kubectl apply`를 직접 치는 대신, **git에 커밋(merge)하는 것이 곧 배포**가 된다.

이는 1장의 선언형 철학을 배포 과정 전체로 확장한 것이다.
(개념만 짚고 넘어간다 — 깊이는 별도 주제.)

## 7.7 정리

이 장을 한 문장으로 줄이면 이렇다.

> 실무는 **kubectl**로 보고 진단하고, **매니페스트를 git에 두고 `apply`**(선언형)하며, 이미지만 바꿔 다시 적용하면 **readiness 기반 롤링 업데이트로 무중단 배포·즉시 롤백**이 되고, 규모가 커지면 **Helm**으로 환경별 값을 분리해 관리한다.

이제 만들고, 연결하고, 배포까지 다 했다.
하지만 현실은 늘 깨끗하지 않다.
다음 장은 **실제로 어디서, 왜 터지는지** — 쿠버네티스 운영의 대표적인 함정들을 모은다.
