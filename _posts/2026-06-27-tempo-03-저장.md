Tempo가 싼 이유의 핵심이 저장 구조에 있다.
오브젝트 스토리지에 컬럼형 포맷으로 저장하고, 외부 인덱스 DB를 두지 않는다.

## 3.1 object storage 전용

Tempo는 trace를 **오브젝트 스토리지(S3·GCS·Azure)** 에 저장한다(로컬 파일시스템은 개발용).
별도의 인덱스 데이터베이스가 없다.
이게 저비용·운영 단순의 핵심이다.

## 3.2 vParquet — 컬럼형 block

저장 단위는 **block**이고, 포맷은 **vParquet**(Apache Parquet 응용, 컬럼형)이다.

- **vParquet4**가 Tempo 3.0의 기본.
- **vParquet5**는 3.0에서 production-ready(아직 기본은 아님).
  자주 쓰는 속성을 담을 전용 컬럼이 더 많아졌다(string 10→20개 등).

컬럼형이라 "특정 속성만" 읽을 때 그 컬럼만 스캔하면 돼서 효율적이다.

## 3.3 🔴 "무인덱스"의 실체

![](/assets/img/posts/tempo-03-저장/tempo-block.svg)

"Tempo는 인덱스가 없다"는 정확하지 않다.
정확히는 **외부 인덱스 DB(엘라스틱서치·카산드라)가 필요 없다**는 뜻이고, block 안에는 인덱스가 있다.

block 하나에 들어있는 것:

1. **bloom filter**: "이 trace ID가 이 block에 있을 수도 있나?"를 빠르게 판별.
   없으면 그 block을 건너뛴다(I/O 절감).
2. **trace ID index**: block 내부에서 어느 부분(row group)에 그 trace가 있는지 좁힌다.
3. **컬럼형 Parquet**: 속성 검색용.

## 3.4 두 가지 조회

그래서 조회 방식이 둘로 갈린다.

| | trace ID 조회 | 속성 검색(TraceQL) |
|---|---|---|
| 방식 | bloom으로 block 후보 제거 → index로 좁힘 | 시간범위 내 block을 스캔 |
| 속도 | **빠름**(스캔 아님) | 상대적으로 느림 |
| 가속 | bloom + index | **dedicated columns** |

**trace ID를 알고 찾는 건 빠르고**, **속성으로 검색하는 건 스캔**이다.
그래서 자주 검색하는 속성을 **dedicated column**(전용 컬럼)으로 빼두면 그 스캔이 빨라진다.

## 3.5 정리

- trace를 **오브젝트 스토리지에 vParquet block**으로 저장(인덱스 DB 불필요).
- "무인덱스"는 과장 — block 안에 **bloom filter + trace ID index**가 있다.
- **trace ID 조회는 빠르고**, 속성 검색은 스캔(dedicated column으로 가속).

다음 장에서는 트레이스에 질문하는 언어, TraceQL이다.
