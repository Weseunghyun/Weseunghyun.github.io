4장에서 우리는 중요한 사실 하나를 마주쳤다.

> 컨테이너가 만든 변경은 맨 위 쓰기 판에 쌓이고, **컨테이너를 지우면 그 데이터도 함께 사라진다.**

웹 서버라면 괜찮다 (지우고 다시 띄우면 되니까).
하지만 데이터베이스라면?
사용자가 올린 파일이라면?
그게 사라지면 큰일이다.

이 장의 질문은 이것이다.
**"꼭 살려야 하는 데이터를 컨테이너 바깥에 어떻게 보관하나?"**

## 7.1 왜 데이터가 사라지는가 (복습)

컨테이너는 읽기 전용 레이어들 위에 **얇은 쓰기 판** 하나를 얹어 동작한다 (4장).
이 쓰기 판은 컨테이너의 일부다.
그래서 `docker rm`으로 컨테이너를 지우면 그 쓰기 판도 같이 사라진다.

게다가 이 쓰기 판은 copy-on-write 방식이라 무거운 쓰기 작업에는 느리다.
데이터베이스처럼 끊임없이 디스크에 쓰는 프로그램에는 더더욱 안 맞는다.

그래서 데이터는 컨테이너 바깥에 따로 둬야 한다.
Docker는 그 방법을 세 가지 준다.

![](/assets/img/posts/docker-07-데이터-볼륨/docker-volume-persistence.svg)

## 7.2 세 가지 데이터 보관 방법

| 종류 | 저장 위치 | 특징 |
|---|---|---|
| **볼륨(volume)** | Docker가 관리하는 영역 | 가장 권장. 백업·이전 쉽고 관리 깔끔 |
| **바인드 마운트** | 호스트의 특정 폴더 직접 연결 | 개발 중 코드 실시간 반영에 좋음 |
| **tmpfs** | 호스트의 메모리 | 디스크에 안 남음. 임시 데이터·비밀값 |

하나씩 보자.

## 7.3 볼륨 — 권장 방식

**볼륨**은 Docker가 알아서 관리하는 데이터 저장 공간이다.
컨테이너와 별개로 존재하므로, 컨테이너를 지워도 볼륨은 남는다.

```bash
docker volume create pgdata
docker run -v pgdata:/var/lib/postgresql/data postgres
```

`pgdata`라는 볼륨을 만들어, 데이터베이스가 데이터를 쓰는 경로(`/var/lib/postgresql/data`)에 연결했다.
이제 DB가 쓰는 데이터는 컨테이너가 아니라 볼륨에 쌓인다.
컨테이너를 지우고 새로 띄워도, 같은 볼륨을 연결하면 데이터가 그대로 살아 있다.

> 데이터베이스나 꼭 보존해야 할 상태는 **무조건 볼륨**에 둔다. 이게 가장 중요한 규칙이다.

볼륨은 Docker가 관리하니 백업·이전도 편하고, 원격 스토리지(NFS·클라우드)에 연결하는 드라이버도 있다.

## 7.4 바인드 마운트 — 호스트 폴더 직결

**바인드 마운트**는 호스트의 특정 폴더를 컨테이너 안에 그대로 연결한다.

```bash
docker run -v $(pwd)/src:/app/src node
```

내 현재 폴더의 `src`를 컨테이너 안 `/app/src`에 연결했다.
호스트에서 코드를 수정하면 컨테이너 안에 **즉시 반영**된다.
그래서 개발 중에 "코드 고치고 → 바로 확인"하는 데 유용하다.

다만 호스트의 폴더 구조에 의존하고, 권한 문제가 생기기 쉽다 (아래 함정).

## 7.5 tmpfs — 메모리에 잠깐

**tmpfs**는 데이터를 디스크가 아니라 **메모리**에 둔다.
컨테이너가 끝나면 흔적 없이 사라진다.
비밀번호 같은 민감한 임시 데이터나, 디스크에 남기면 안 되는 캐시에 쓴다.

## 7.6 🔴 자주 걸리는 함정들

**권한 문제 (바인드 마운트)**
컨테이너 안 사용자의 ID와 호스트 파일 주인의 ID가 안 맞으면 "permission denied"가 난다.
반대로 컨테이너가 root로 호스트 폴더에 파일을 만들면, 호스트에서 그 파일이 root 소유가 되어 지우기 곤란해진다.
`--user`로 사용자 ID를 맞추거나, 컨테이너를 비-root로 돌려 정렬한다 (5장).

**빈 볼륨이 덮어쓰는 문제**
이미지 안에 이미 데이터가 들어 있는 경로에 마운트할 때 주의가 필요하다.
연결 방식에 따라 컨테이너 안 원래 내용이 가려지거나 덮일 수 있다.

**무심코 데이터 날리기**
볼륨 없이 쓴 데이터는 `docker rm`과 함께 사라진다.
또 `docker compose down -v`의 `-v`는 볼륨까지 지우니, 운영에서 습관적으로 치면 데이터가 날아간다.

**디스크가 가득 차기**
안 쓰는 이미지, 떠다니는 볼륨, 빌드 캐시가 계속 쌓여 호스트 디스크를 가득 채우면 새 컨테이너가 안 뜬다.
`docker system df`로 사용량을 보고, `docker system prune`으로 안 쓰는 것을 정리한다 (단, 정말 안 쓰는 것만 지워지는지 확인하고).

## 7.7 정리

이 장을 한 문장으로 줄이면 이렇다.

> 컨테이너의 쓰기 판은 컨테이너와 함께 사라지므로, 보존할 데이터는 **볼륨**(권장)·바인드 마운트·tmpfs로 컨테이너 바깥에 둬야 하고, **데이터베이스·상태는 무조건 볼륨**이 정답이다.

여기까지 오면 컨테이너 하나를 잘 만들고, 연결하고, 데이터를 지키는 법을 다 안다.
그런데 실제 서비스는 "앱 + DB + 캐시" 처럼 여러 컨테이너의 묶음이다.
이걸 명령어 여러 줄로 일일이 띄우는 건 번거롭다.
8장에서 이 묶음을 한 파일로 다루는 **Docker Compose**를 본다.
