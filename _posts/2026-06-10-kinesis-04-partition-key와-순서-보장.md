3장에서 우리는 키네시스의 3층 구조를 봤다. 데이터 한 건은 **레코드**(봉투)에 담기고, 레코드는 **샤드**(컨베이어 레인)에 줄지어 쌓이고, 샤드 여러 개가 묶여 **스트림**(전체 설비)이 된다.

그런데 그 그림에는 아직 답하지 않은 질문이 두 개 숨어 있다.

1. 레인이 10개라면 — **방금 도착한 봉투는 누가, 무슨 기준으로 어느 레인에 올리나?** 아무 레인에나 던지면 안 되나?
2. 차량 호출 앱에서 차량 `car-7`의 위치 기록이 "강남 → 역삼 → 선릉" 순서로 들어왔는데, 읽는 쪽에서 "선릉 → 강남 → 역삼"으로 뒤섞여 나오면 어떻게 되나? 지도 위 차가 순간이동을 한다. **키네시스는 이 순서를 어떻게 지키나?**

놀랍게도 이 두 질문의 답은 하나로 이어진다. 그 연결 고리가 이 장의 주인공 **partition key**다. 
이 장을 다 읽으면 "partition key를 뭘로 정하느냐가 키네시스 설계의 절반"이라는 말이 무슨 뜻인지 알게 된다.

## 4.1 한 문단 복습 — 봉투에는 세 칸이 있다

3장에서 본 레코드(봉투)의 구성을 다시 꺼내자. 
![](/assets/img/posts/kinesis-04-partition-key와-순서-보장/Pasted-image-20260611083051.png)
봉투에는 세 칸이 있다. 
**data blob**(내용물 — 실제 데이터, 키네시스는 안을 들여다보지 않는다), 
**partition key**(주소 라벨 — 어느 샤드로 갈지 정하는 값, 보내는 쪽이 직접 정한다), 
**sequence number**(도착 순번 — 샤드에 들어갈 때 키네시스가 자동으로 붙여주는 일련번호). 
3장에서는 이 세 칸이 *있다*는 것까지만 봤다. 
O이번 장에서는 뒤의 두 칸 — 주소 라벨과 도착 순번 — 이 실제로 **어떻게 동작하는지**를 파고든다.

## 4.2 partition key — 내용물과 "분류 기준"을 분리한 이유

### 첫 원리부터: 왜 라벨이 따로 필요한가

이상하게 느껴질 수 있다. 데이터 안에 이미 `{"car":"car-7","lat":37.5,...}`처럼 차량 ID가 들어 있는데, 왜 봉투 겉에 또 `car-7`이라는 라벨을 붙이나?

이유는 키네시스가 **내용물을 절대 열어보지 않기** 때문이다. 키네시스 입장에서 data blob은 그냥 바이트 덩어리다 — JSON이든 이미지든 상관하지 않고, 해석도 하지 않는다. 그런데 분류는 해야 한다. 
봉투를 열어보지 않고 분류하려면? **겉면에 분류용 라벨을 따로 붙이는 수밖에 없다.**

> 비유: 우체국 소포를 떠올리자. 소포 안에 편지가 들어 있고 편지에 주소가 적혀 있어도, 우체국은 소포를 뜯어보지 않는다. 분류는 오직 **겉면에 적힌 주소 라벨**로만 한다. partition key가 바로 그 겉면 라벨이다.

이 분리에는 또 하나의 깊은 의미가 있다. `car-7`의 GPS 좌표는 매 초 바뀐다(내용물은 매번 다르다). 
하지만 "이건 car-7의 기록"이라는 **분류는 변하지 않는다.** 내용물과 분류 기준을 분리해 두었기 때문에, 
car-7의 기록은 — 내용이 어떻게 바뀌든 — 항상 같은 기준으로 같은 곳에 모일 수 있다. 
이게 뒤에서 볼 순서 보장의 토대가 된다.

### 메커니즘: 누가 정하고, 어떤 제약이 있나

- partition key는 **보내는 쪽(producer)이 직접 정한다.** 
  키네시스가 정해주지 않는다. 차량 ID, 사용자 ID, 주문 ID — 무엇이든 내가 고른 문자열이다.
- 길이는 **최대 256자**다(AWS 공식 문서 기준).
- "무엇을 key로 고르느냐"는 전적으로 설계자의 몫이고, 이 선택이 스트림의 성능과 순서 보장을 동시에 좌우한다(왜 그런지는 4.4와 4.5에서).

## 4.3 라벨이 레인 번호가 되기까지 — 해시 분배

자, 라벨은 붙었다. 그런데 `car-7`이라는 **문자열**이 어떻게 "Shard-3으로 가라"는 **레인 번호**가 되나? 
키네시스 어딘가에 "car-7 → Shard-3"이라고 적힌 거대한 명단이라도 있는 걸까?

명단은 없다. 대신 **계산**을 한다. 이게 이 장에서 가장 재미있는 부분이다.

### 1단계 — 문자열을 거대한 숫자로 바꾼다 (해시)

키네시스는 partition key를 **MD5 해시 함수**에 넣는다(AWS 공식 문서 기준). 
해시 함수가 처음이라면 이렇게 이해하면 된다.

> 비유: 해시 함수는 "글자를 넣으면 숫자가 나오는 믹서기"다. 특징이 두 가지다. (1) **같은 걸 넣으면 언제나 똑같은 숫자가 나온다** — 오늘 넣어도, 내일 넣어도, 어느 서버에서 넣어도. (2) 입력이 조금만 달라도 결과는 전혀 다른 숫자가 된다 — `car-7`과 `car-8`의 결과는 이웃 숫자가 아니라 완전히 동떨어진 값이다.

MD5는 어떤 문자열을 넣어도 **128비트 정수** — 0부터 2¹²⁸까지의 범위에 있는, 어마어마하게 큰 숫자 — 하나를 돌려준다. 즉 `"car-7"`이라는 글자가 `87539...` 같은 거대한 숫자 하나로 변환된다.

### 2단계 — 그 숫자가 떨어지는 구간의 샤드로 보낸다

0부터 2¹²⁸까지의 거대한 수직선이 있다고 상상하자. 키네시스는 이 수직선을 잘라서 **각 샤드에게 한 구간씩** 나눠준다. 이 구간을 **hash key range**라고 부른다. 봉투의 해시값이 어느 구간에 떨어지느냐 — 그게 곧 레인 배정이다.
![](/assets/img/posts/kinesis-04-partition-key와-순서-보장/Pasted-image-20260611083603.png)

> 비유: 택배 물류센터의 자동 분류기를 떠올리자. 라벨의 주소를 읽어 "이 우편번호 구간은 3번 레인"이라고 기계적으로 밀어 넣는다. 사람이 명단을 뒤지는 게 아니라, **규칙 하나로 모든 봉투가 자동 분류**된다.

### 이 방식이 영리한 이유 두 가지

**첫째, "같은 key = 항상 같은 샤드"가 공짜로 따라온다.** 해시 함수는 같은 입력에 항상 같은 출력을 주므로, `car-7`의 해시값은 영원히 `87539...`다. 따라서 car-7의 봉투는 1초 전 것도, 1시간 뒤 것도 **전부 같은 구간 → 같은 샤드**에 떨어진다. 명단 없이, 조회 없이, 순수 계산만으로. 이게 다음 절에서 볼 순서 보장의 핵심 재료다.

**둘째, key 종류가 많으면 봉투가 레인에 골고루 퍼진다.** 해시값은 입력이 조금만 달라도 수직선 위 전혀 다른 곳에 떨어지므로, 차량 ID가 수만 종류라면 봉투들은 자연스럽게 전 구간에 흩뿌려진다. 레인 4개가 비슷한 양으로 일하게 된다. (**단, "key 종류가 많으면"이라는 조건이 붙는다** — 이 조건이 깨지면 무슨 일이 생기는지는 4.5에서.)

## 4.4 순서 보장 — sequence number, 그리고 "샤드 안에서만"

이제 도입에서 던진 두 번째 질문으로 돌아가자. "강남 → 역삼 → 선릉"이 뒤섞이지 않는 이유.

### 샤드 안에서 벌어지는 일

봉투가 샤드에 들어가는 순간, 키네시스가 **sequence number**(도착 순번)를 자동으로 찍어준다. `49590...8732` 같은 긴 일련번호인데, 같은 partition key의 레코드들 사이에서는 **시간이 갈수록 커진다**(AWS 공식 문서 기준). 그리고 샤드는 한 번 들어온 레코드의 순서를 절대 바꾸지 않는 **불변 로그**다 — 새 봉투는 항상 줄의 맨 뒤에만 붙는다(append).

```
Shard-3 :  오래된 것 ─────────────────────────────────────▶ 최신
           [car-7 #49590...101] [car-2 #49590...102] [car-7 #49590...103] ← 새 봉투는 항상 맨 뒤에
            "강남"                                    "역삼"
              └ 한 번 줄을 서면 순서가 바뀌지 않는다(불변 로그)
              └ 읽는 쪽도 이 줄 순서 그대로 읽는다
```

이제 퍼즐이 맞춰진다.

1. `car-7`의 모든 기록은 — 해시 덕분에 — **항상 Shard-3**으로 간다. (4.3)
2. Shard-3 안에서는 들어온 순서대로 순번이 붙고, 그 줄은 절대 안 바뀐다.
3. 읽는 쪽은 그 줄을 순서대로 읽는다.

∴ **car-7의 위치 변화는 항상 "강남 → 역삼 → 선릉" 순서로 읽힌다.** "같은 key → 같은 샤드 → 샤드 안 순서 보장" — 이 세 단계 연결이 키네시스 순서 보장의 전부다.

### 그런데 — 보장 범위는 "샤드 안"까지다

여기서 정확히 짚어야 할 게 있다. 위에서 본 순서의 근거(sequence number)는 **샤드 안에서** 매겨지는 순번이다. 즉 키네시스가 보장하는 순서는 **한 샤드 안의 순서**다. 서로 다른 샤드 사이 — 예컨대 Shard-1의 `car-3` 기록과 Shard-3의 `car-7` 기록 중 무엇이 먼저냐 — 에는 이런 공통 순번이 없다.

이게 결함처럼 들릴 수 있지만, 사실 대부분의 경우 문제가 되지 않는다. 
우리가 지키고 싶은 순서는 보통 **"같은 대상 안에서의 순서"이기 때문이다. car-7의 이동 경로가 뒤섞이면 큰일이지만, car-3과 car-7 중 누가 먼저 위치를 보냈는지는 지도에 표시하는 데 중요하지 않다. 
그리고 "같은 대상"은 partition key 덕분에 이미 한 샤드에 모여 있다.

거꾸로 생각해보면 이 설계가 자연스럽다는 것도 보인다 — 만약 스트림 전체에 단 하나의 순번을 매기려면 모든 레인의 봉투가 한 곳을 거쳐 번호를 받아야 할 텐데, 그러면 레인을 여러 개 둔 의미(병렬 처리)가 사라진다. 
"순서는 레인 단위로, 병렬은 레인 개수로"라는 분담인 셈이다. (이 단락은 공식 문서의 설명이 아니라 동작 방식에서 따라 나오는 이해를 적은 것이다.)

그래서 실무 감각은 이렇게 정리된다: **"순서가 함께 지켜져야 하는 것들끼리 같은 partition key를 준다."** 
차량별 순서가 필요하면 key = 차량 ID, 사용자별 순서가 필요하면 key = 사용자 ID.

## 4.5 그 라벨, 아무거나 고르면 안 된다 — 쏠림 맛보기

partition key가 "어느 레인으로 갈지"를 전부 결정한다는 건, 
뒤집으면 **key를 잘못 고르면 레인 배분이 망가진다**는 뜻이다.

4.3에서 "key 종류가 많으면 골고루 퍼진다"고 했다. 그 반대 상황을 보자. 
어떤 글로벌 서비스가 partition key를 **나라 코드**(`KR`, `US`, `JP`…)로 정했다고 하자. 
key 종류가 200개 남짓밖에 안 되는 데다, 트래픽 대부분이 한국에서 나온다면?

```
나쁜 예: partition key = "나라 코드"  →  key 종류가 ~200개뿐
        ┌─────────┬─────────┬─────────┐
        │ Shard-1 │ Shard-2 │ Shard-3 │
        │ ███████ │ ▁       │ ▁       │   ← "KR" 봉투가 전부 한 레인으로 폭주
        └─────────┴─────────┴─────────┘      나머지 레인은 거의 놀고 있음

좋은 예: partition key = "차량 ID" / "user_id"  →  key 종류가 수만~수백만 개
        ┌─────────┬─────────┬─────────┐
        │ Shard-1 │ Shard-2 │ Shard-3 │
        │ ████    │ ████    │ ████    │   ← 골고루 분산
        └─────────┴─────────┴─────────┘
```

`KR`은 하나의 key이므로 — 4.3의 규칙대로 — **그 봉투 전부가 단 하나의 샤드**로 간다. 그런데 3장에서 봤듯 샤드 1개의 받아들이는 속도는 고정돼 있다(쓰기 1MB/s 또는 1,000건/s). 레인 하나에 전국 트래픽이 몰리면 그 레인만 한도에 부딪혀 막히고, 나머지 레인은 텅텅 빈다. 이렇게 한 샤드만 뜨거워지는 현상을 **hot shard**라고 부른다.

해법의 방향은 이미 나왔다: **종류가 충분히 많은(고유값이 많은) key** — 사용자 ID, 기기 ID, 차량 ID 같은 것 — 를 고르는 것이다. 이 문제를 한도·요금·자동 확장과 엮어 제대로 다루는 게 바로 다음 5장이다. 
지금은 "partition key 선택 = 분산의 품질"이라는 연결만 챙기자.

## 4.6 코드로 한 번 — 보내는 쪽 입장에서 보면

실제로 보내는 쪽 코드에서 partition key가 어디에 등장하는지 보자. 키네시스에 레코드 1건을 넣는 API가 `PutRecord`다. 의사코드(특정 언어가 아닌, 흐름만 보여주는 가짜 코드)로 쓰면 이렇다.

```python
# 차량 앱이 1초마다 실행하는 로직 (의사코드)
put_record(
    stream_name   = "vehicle-locations",          # 어느 스트림(통로)에
    partition_key = "car-7",                      # 주소 라벨 — 내가 정한다
    data          = '{"lat":37.50,"lng":127.03}'  # 내용물 — 키네시스는 안 본다
)
```

보내는 쪽이 지정하는 건 **스트림 이름 + partition key + 내용물**, 딱 셋이다. 
주목할 것은 **여기에 "샤드 번호"가 없다**는 점이다. 보내는 쪽은 샤드가 몇 개인지, car-7이 어느 샤드로 가는지 알 필요가 없다. 그건 키네시스가 해시로 알아서 정한다. 
sequence number도 마찬가지 — 보내는 쪽이 매기는 게 아니라, 샤드에 들어가는 순간 키네시스가 부여한다.

이 호출이 일어난 뒤의 전체 여정을 한 줄로 묶으면:

```
"car-7" ─① MD5 해시─▶ 87539... ─② hash range 매칭─▶ Shard-3 ─③ 맨 뒤에 append─▶ sequence number 부여
        (보내는 쪽이 지정)              (키네시스가 계산)                  (키네시스가 부여)
```

1. 데이터 안에 이미 차량 ID가 들어 있는데도 partition key를 봉투 겉에 따로 붙이는 이유는? 
   (힌트: 키네시스는 내용물을 열어보지 않는다)
2. `car-7`이라는 문자열이 특정 샤드로 배정되기까지의 과정을 두 단계(해시 → 구간 매칭)로 설명할 수 있는가? "어디에도 명단이 없는데 항상 같은 샤드로 가는" 이유는?
3. 키네시스의 순서 보장을 "같은 key → 같은 샤드 → 샤드 안 순서"의 3단 논리로 설명할 수 있는가? 그리고 그 보장이 미치는 범위는 어디까지인가?
4. partition key를 "나라 코드"로 정하면 어떤 문제가 생기나? 어떤 key가 좋은 key인가?
5. `PutRecord`를 호출할 때 보내는 쪽이 지정하는 것 셋과, 지정하지 *않는* 것(키네시스가 정하는 것) 둘은 무엇인가?

---

다음 **5장**에서는 이 장 끝에서 맛만 본 문제를 정면으로 다룬다 — 샤드 1개의 정확한 한도(쓰기/읽기 숫자), 한도를 넘으면 실제로 무슨 일이 벌어지는지(throttling), hot shard를 어떻게 발견하고 고치는지, 그리고 샤드 개수를 사람이 정하는 모드와 AWS에 맡기는 모드의 차이까지. "레인을 몇 줄 깔고, 막히면 어떻게 뚫나"의 장이다.
