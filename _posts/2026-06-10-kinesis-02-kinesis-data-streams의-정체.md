1장에서 우리는 "보내는 쪽과 받는 쪽 사이에 **중간 통로**를 두면 완충·분리·보관 문제가 한 번에 풀린다"는 것을 봤다.
그리고 그 중간 통로가 키네시스가 해주는 일이라고 했다.

그런데 막상 AWS에서 키네시스를 찾아보면 당황스러운 일이 벌어진다. 
"Kinesis"라는 이름 아래 제품이 **하나가 아니라 여러 개** 나온다. 
Data Streams, Firehose, Flink, Video Streams… 이름도 비슷비슷하다.
이 장에서 풀 질문은 세 가지다.

1. 이 시리즈가 다루는 **Kinesis Data Streams(KDS)는 정확히 뭘 하는 상자**인가?
2. 나머지 형제 제품들과는 **뭐가 다른가?** (입문자가 가장 헷갈리는 지점)
3. 그리고 — 의외로 중요한 질문 — **언제는 KDS를 안 쓰는 게 맞는가?**

## 2.1 Kinesis Data Streams — 정확히 "무엇을 하는 상자"인가

먼저 이름부터 정리하자.
**Kinesis Data Streams**, 줄여서 **KDS**.
"Kinesis 제품군의 코어"이고, 1장에서 말한 그 "중간 통로"를 실제로 구현한 AWS 서비스다.

KDS를 한 문장으로 줄이면 이렇다.

> **수많은 곳에서 1초에 수만 건씩 들어오는 데이터를, 잃어버리지 않고 순서대로 받아, 
> 여러 프로그램이 각자 자기 속도로 읽어가게 해주는 중간 저장 + 전달 장치.**

그림으로 보면 이렇다.
1장의 "중간 통로" 그림이 실제 AWS에서 어떤 모습인지를 구체화한 것이다.

![](/assets/img/posts/kinesis-02-kinesis-data-streams의-정체/스크린샷-2026-06-11-오전-7.14.41.png)

그림에 처음 보는 말이 몇 개 있으니 그 자리에서 풀고 가자.

- **생산자(producer)** = 데이터를 만들어 통로에 **넣는 쪽**.
  1장의 "보내는 쪽"이다.
  주문을 받는 웹서버, 클릭을 기록하는 앱, 측정값을 보내는 센서 같은 것들.
- **소비자(consumer)** = 통로에서 데이터를 **꺼내 읽는 쪽**.
  1장의 "받는 쪽".
  실시간 집계 프로그램, 사기 탐지 프로그램 등.
- **put / read** = 넣는 동작과 읽는 동작.
  생산자는 put 하고, 소비자는 read 한다.
- **레코드(record, 그림의 [rec])** = 통로 위를 흐르는 **데이터 한 건 한 건**.
  주문 1건, 클릭 1건이 각각 레코드 하나다.
- **샤드(shard)** = 벨트의 "줄".
  컨베이어 벨트가 한 줄이 아니라 여러 줄 나란히 놓인 모습을 떠올리면 된다.
  자세한 건 3장에서 다루니, 지금은 "벨트가 여러 줄이라 많은 양을 동시에 흘릴 수 있구나" 정도면 충분하다.
- **retention(보관 기간)** = 데이터를 통로에 남겨두는 기간.
  KDS는 **24시간에서 최대 365일**까지 보관할 수 있다(AWS 공식 문서 기준).
  1장에서 본 "보관" 기능이 바로 이것이다.

여기서 KDS의 성격을 규정하는 사실 두 가지를 짚자.

**첫째, 빠르다.** 데이터를 넣으면(put) **1초가 되기 전에** 읽을(get) 수 있다(AWS 공식 문서 기준).
"실시간 스트리밍"이라는 말이 과장이 아닌 이유다.

**둘째 — 이게 더 중요한데 — KDS 자체는 데이터를 가공하지 않는다.** KDS는 그냥 "안전하게 받아두고, 순서를 지켜주고, 원하는 만큼 읽어가게 해주는" **파이프 + 단기 저장소**다.
들어온 주문 데이터를 합산한다든지, 이상한 거래를 골라낸다든지 하는 일은 KDS가 하지 않는다.
그건 **읽어가는 소비자**(우리가 만드는 프로그램)가 할 일이다.

> 비유하자면, KDS는 **공장의 컨베이어 벨트 그 자체**다. 벨트는 물건을 나를 뿐, 조립하거나 포장하지 않는다. 
> 조립과 포장은 벨트 옆에 선 작업자(소비자)의 몫이다.

그리고 1장 끝에서 예고한 "AWS가 직접 운영해준다"의 의미가 여기 붙는다.
KDS는 **관리형(managed)** 서비스다 — 이 벨트를 돌리는 기계(서버)를 우리가 사거나 설치하거나 고치지 않는다.
AWS가 운영하고, 우리는 벨트를 **빌려 쓰기만** 한다.
"스트림을 하나 만들어줘"라고 요청하면 벨트가 생기고, 우리는 넣고 읽기만 하면 된다.

## 2.2 왜 필요한가 — 이게 없으면 무슨 고통이 있나

1장에서 일반론으로 본 "직접 연결의 세 가지 문제"를, 이번엔 실제 시나리오에 대입해 보자.
KDS가 풀어주는 고통이 구체적으로 보인다.

**시나리오**: 어떤 쇼핑몰이 주문이 들어올 때마다 세 가지 일을 **동시에** 하고 싶다.

1. 실시간 매출 대시보드 갱신 — "지금 이 순간 매출이 얼마지?"
2. 사기 탐지 — "이 주문, 도용된 카드 아닌가?"
3. 데이터 웨어하우스 적재 — 나중에 분석할 수 있게 창고에 차곡차곡 저장

KDS 없이 해보면 어떻게 될까.

**방법 1 — 그냥 DB에 쓰고 밤에 배치로 처리한다.**
대시보드는 늘 한 박자 늦는다.
밤에 배치를 돌려야 어제 매출이 보인다.
"실시간"은 포기.
더 나쁜 건, 주문이 폭주해서 DB나 앱서버가 터지면 **그 사이의 데이터가 유실**된다는 것이다.

**방법 2 — 세 소비자가 주문 서버에 직접 붙는다.**
이번엔 1장의 "분리" 문제가 터진다.
셋 중 하나(예: 웨어하우스 적재)가 느려지면, 그게 주문을 받는 앱서버까지 붙잡는다.
**창고 정리가 느리다고 계산대 줄이 멈추는 꼴**이다.
생산하는 쪽과 소비하는 쪽이 강하게 묶여 있어서 생기는 문제다.

**KDS를 끼우면** 이렇게 된다.
주문이 들어오는 즉시 스트림에 던지고(앱서버는 바로 응답하고 다음 주문을 받는다), 세 소비자는 **각자 자기 속도로** 같은 스트림을 독립적으로 읽는다.
대시보드는 빠르게, 웨어하우스 적재는 느긋하게 — 서로 영향을 주지 않는다.
생산과 소비가 **분리(decouple)** 된 것이다.

AWS 공식 문서도 KDS의 핵심 효용을 같은 맥락으로 설명한다 — 데이터를 서버에서 배치로 모았다가 보내는 게 아니라 **생기는 즉시 제출**하게 해서 초 단위 처리를 가능하게 하고, **앱서버가 죽어도 그 로그가 유실되지 않게** 한다는 것.

> 비유로 정리하면(어디까지나 비유다): 배치는 "우편물을 모아뒀다 저녁에 한 번 배달"하는 것이고, KDS는 "도착 즉시 컨베이어 벨트에 올려 여러 부서가 실시간으로 집어 처리"하는 것이다. 그리고 벨트에 올라간 편지는 retention 기간 동안 남아 있어서, 다른 부서가 나중에 와서 **다시 집어갈 수도** 있다.

## 2.3 "Kinesis"는 4형제다 — 제품군 한 번에 정리

이제 이 장의 두 번째 질문.
AWS 콘솔에서 마주치는 그 여러 "Kinesis"들의 정체다.

"Kinesis"는 단일 제품 이름이 아니라 **제품군(패밀리)의 이름**이다.
우리가 지금까지 본 KDS가 코어이고, 그 위에 형제 제품들이 얹혀 있다.
입문자가 **가장 헷갈리는 짝은 KDS vs Firehose**이니 그것부터 그림으로 보자.

![](/assets/img/posts/kinesis-02-kinesis-data-streams의-정체/Pasted-image-20260611080639.png)

차이가 보이는가?
**KDS는 벨트만 빌려주고, 벨트에서 물건을 집어 처리하는 작업자(소비자 프로그램)는 내가 만든다.** 그래서 무엇이든 할 수 있다 — 실시간 집계든 사기 탐지든, 내가 코드를 짜는 만큼 자유다.
한 번 읽었던 데이터를 처음부터 **다시 읽는 것(replay)** 도 된다.

반면 **Firehose는 "배달 서비스"다.** 데이터를 주면 정해진 목적지 — S3(AWS의 대용량 파일 창고), Redshift(분석용 데이터 창고), OpenSearch(검색 엔진) 같은 곳 — 에 **자동으로 적재**해준다. 
읽는 코드를 짤 필요가 없다.
대신 데이터를 일정량 모았다가(버퍼링) 묶어서 보내기 때문에, 즉시가 아니라 **준실시간(near-real-time)** 이다.

숫자로 비교하면 차이가 선명해진다(AWS 공식 문서 기준).

|             | KDS                        | Firehose               |
| ----------- | -------------------------- | ---------------------- |
| 데이터가 도착하기까지 | put → get **1초 미만**        | 버퍼링 **최소 60초** 후 묶음 전송 |
| 읽는 코드       | **내가 작성** (자유로운 가공·replay) | **불필요** (목적지에 자동 적재)   |
| 성격          | 저수준·범용 벨트                  | 목적지행 자동 배달             |

이제 4형제 전체를 표로 정리하자. 
이름이 최근에 바뀐 제품이 둘 있어서 옛 이름도 같이 적는다(검색하다 보면 옛 이름 자료가 많이 나온다).

| 제품 (현재 이름)                                                  | 한 줄 용도                                         | 결정적 차이                                        |
| ----------------------------------------------------------- | ---------------------------------------------- | --------------------------------------------- |
| **Kinesis Data Streams (KDS)**                              | 데이터를 실시간 수집·보관, **내가 만든 소비자**가 읽어 처리           | 저수준·범용. 소비 코드 직접 작성, replay 가능, put→get 1초 미만 |
| **Amazon Data Firehose** (구 Kinesis Data Firehose)          | 데이터를 **목적지(S3·Redshift·OpenSearch…)로 자동 적재**   | 코드·서버 불필요. 버퍼링(최소 60초) 후 묶음 전송 = 준실시간         |
| Managed Service for Apache Flink (구 Kinesis Data Analytics) | 흐르는 스트림을 **SQL/Java(Flink)로 실시간 분석**(집계·윈도잉 등) | "처리·분석" 계층. KDS나 MSK를 입력 소스로 받는다              |
| **Kinesis Video Streams**                                   | **영상·오디오 등 미디어** 스트리밍                          | 미디어 전용(WebRTC/RTSP 등). 데이터 스트림과는 다른 영역        |

> 모르는 단어 보충: **Flink**는 흐르는 데이터를 실시간으로 계산하는 오픈소스 분석 엔진이고, **윈도잉**은 
> "최근 5분간"처럼 흐름을 시간 단위로 잘라 집계하는 기법이다. 
> **MSK**는 AWS가 대신 운영해주는 카프카(Kafka)다 — 카프카는 Part 2에서 본격적으로 다룬다. 
> 지금 이 셋을 깊이 알 필요는 없다. "이런 형제가 있다" 지도만 그려두면 된다.

헷갈릴 때 쓰는 **한 줄 구분법**은 이것이다.

> **"내가 읽는 코드를 짜고, 자유롭게 가공하고, 필요하면 다시 읽고 싶다" → KDS. 
> "그런 거 없고 그냥 S3에 떨궈만 줘" → Firehose.**

참고로 둘은 경쟁 관계가 아니라 **보완 관계**이기도 하다.
Firehose는 KDS를 입력으로 받을 수 있어서,
"KDS로 실시간 처리를 하면서, 같은 스트림을 Firehose에 이어 붙여 S3에 원본도 자동 보관"하는 식으로 함께 쓸 수 있다.

이 시리즈는 4형제 중 코어인 **KDS**를 끝까지 판다. 
KDS를 이해하면 나머지 형제는 "KDS 주변의 편의 기능"으로 자연스럽게 보이게 된다.

## 2.4 언제 KDS를 안 쓰나 — 비슷해 보이는 서비스와의 경계선

기술을 배울 때 "언제 쓰나"만큼 중요한 게 **"언제 안 쓰나"** 다.
면접에서도 이게 더 좋은 질문이다.
"실시간으로 데이터가 오간다"고 다 KDS가 정답인 건 아니다.
목적이 다르면 맞는 서비스도 다르다.

AWS에는 "데이터/메시지를 전달한다"는 점에서 KDS와 비슷해 보이는 서비스가 몇 개 있다. 
경계선을 표로 그어보자.

| 상황                                         | 맞는 서비스                | 이유                                          |
| ------------------------------------------ | --------------------- | ------------------------------------------- |
| 작업을 줄 세워놓고 **한 소비자가 처리하면 지우는** "작업 큐"      | **SQS**               | 메시지 한 건 단위로 "처리 완료" 확인·재시도를 해주는 구조          |
| **"A가 일어나면 B를 호출해줘"** 식의 이벤트 라우팅·룰 기반 트리거  | **EventBridge / SNS** | 이벤트를 받아 조건에 맞는 곳으로 밀어주는(push) 구조            |
| **대량의 데이터를 순서대로, 여러 소비자가, 다시도** 읽어야 하는 스트림 | **KDS / MSK**         | 다수 소비자가 같은 데이터를 같은 순서로 반복해 읽을 수 있음          |
| 이미 **Apache Kafka 자산**이 있거나 세밀한 튜닝이 필요     | **MSK (Kafka)**       | 같은 스트리밍이라도 Kafka 생태계가 필요하면 그쪽 (Part 3에서 비교) |

처음 보는 이름들을 풀어보자.

**SQS** 는 "작업 대기줄" 서비스다.
비유하면 **은행 번호표**다 — 손님(작업)이 번호표를 뽑고 줄을 서면, 창구 직원(소비자)이 한 명씩 불러 처리하고, **처리가 끝난 번호표는 버린다.** 핵심이 이것이다: SQS의 메시지는 처리되면 사라지는 게 정상이다.
"여러 부서가 같은 데이터를 각자 읽는" KDS와는 목적 자체가 다르다.
이미지 리사이징 요청, 이메일 발송 요청처럼 "한 번 처리하고 끝"인 작업에 맞는다.

**EventBridge / SNS** 는 "이벤트 안내 방송" 쪽이다.
비유하면 **사내 방송 시스템** — "주문 취소 발생!"이라고 방송하면, 그 방송을 구독해둔 부서들에게 **밀어서(push)** 전달된다.
"어떤 일이 일어나면 누구를 호출한다"는 라우팅 규칙이 중심이고, KDS처럼 대량의 데이터를 순서대로 쌓아두고 당겨 읽는 용도가 아니다.

이 경계를 기억하는 **한 줄 정리**:

> **Pull(소비자가 당겨 읽기, 순서·replay가 중요) = SQS·KDS 
> Push(서비스가 밀어주기, 라우팅이 중요) = EventBridge·SNS.**
> 그중에서도 KDS는 **"여러 소비자가 같은 데이터를 순서대로, 필요하면 다시도 읽어야 할 때"** 쓴다.

거꾸로 말하면 — 소비자가 하나뿐이고 처리 후 데이터가 사라져도 된다면 SQS가 더 단순하고, 
"이벤트 한 건이 오면 함수 하나 호출"이 전부라면 EventBridge가 더 맞다. 
**굳이 벨트(스트림)를 깔 필요가 없는 일에 벨트를 깔지 않는 것**, 그게 이 절의 결론이다.

다음 **3장**에서는 드디어 KDS의 **내부**로 들어간다.
이 장의 그림에서 "벨트가 여러 줄"이라고 슬쩍 넘어간 그 **샤드(shard)**, 그리고 벨트 위를 흐르는 **레코드(record)** — 즉 **스트림 > 샤드 > 레코드**의 3층 구조를 뜯어본다.
