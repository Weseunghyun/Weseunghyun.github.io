분산추적(트레이스)은 "느린 요청이 어느 서비스에서 막혔나"를 보여주는 신호다.
관측가능성 시리즈에서 그 개념을 다뤘다.
이 시리즈는 그 트레이스를 **저장하는 백엔드**, 그중 **Grafana Tempo**를 깊게 본다.

## 0.1 트레이스 저장의 고민 — 비싸다

트레이스 데이터는 양이 많다.
한 요청이 수십 개 span을 만들고, 초당 수천·수만 요청이 들어오니까.
전통적인 트레이스 백엔드(Jaeger)는 이걸 빠르게 검색하려고 **엘라스틱서치나 카산드라 같은 인덱스 DB**에 넣었다.
강력하지만, 그 인덱스 DB를 운영하는 비용·복잡도가 크다.

## 0.2 Tempo의 발상 — object storage만

Tempo는 다르게 접근했다.

> 인덱스 DB 없이, **오브젝트 스토리지(S3·GCS·Azure)만으로** 저장한다.

별도의 인덱스 데이터베이스를 운영하지 않으니 저장 비용이 가장 낮고 운영이 단순하다.
질의는 **TraceQL**이라는 언어로 한다.

여기서 흔한 오해 하나를 미리 짚자.
"Tempo는 인덱스가 없다"는 말이 도는데, 정확하지 않다.
**외부 인덱스 DB가 필요 없을 뿐**, block 안에는 trace ID를 찾는 최소한의 인덱스(bloom filter 등)가 있다(3장).

## 0.3 2026년, 크게 바뀌었다

이 시리즈를 쓰는 시점에 Tempo는 **3.0("Project Rhythm")** 으로 아키텍처가 크게 바뀌었다.
데이터를 3벌 복제하던 걸 1벌로 줄이고(RF3→RF1), 읽기·쓰기 경로를 분리했다.
그래서 이 시리즈는 **2.x와 3.0을 구분**해 설명한다(2장).

## 0.4 이 시리즈가 답할 질문들

| 장 | 질문 |
|---|---|
| 1장 | Tempo는 정확히 무엇이고 왜 싼가 |
| 2장 | 컴포넌트는 무엇이고, 3.0에서 뭐가 바뀌었나 |
| 3장 | object storage에 어떻게 저장하고 어떻게 찾나 |
| 4장 | 트레이스에 어떻게 질문하나 (TraceQL) |
| 5장 | 트레이스에서 메트릭을 어떻게 뽑나 |
| 6장 | 어디서 터지나 + 면접 정리 |
