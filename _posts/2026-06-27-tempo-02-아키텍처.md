Tempo는 2026년 3.0에서 아키텍처가 크게 바뀌었다.
다수 운영 환경은 아직 2.x이니, 둘을 구분해 본다.

## 2.1 2.x — 대부분 배포의 기본

| 컴포넌트 | 역할 |
|---|---|
| **distributor** | span 수신·검증, **trace ID로 샤딩** |
| **ingester** | 메모리 배치 → object storage block flush, RF3 복제 |
| **compactor** | block 압축·dedup·retention |
| **querier** | block + ingester에서 span 조회 |
| **query-frontend** | blockID 공간을 샤딩해 querier에 병렬 분배 |
| **metrics-generator**(옵션) | span에서 메트릭 파생(5장) |

쓰기는 distributor → ingester → object storage, 읽기는 query-frontend → querier가 block과 ingester를 함께 본다.

## 2.2 🔴 3.0 — Project Rhythm

3.0은 세 가지를 노렸다 — 복제본 줄이기, 읽기·쓰기 분리, 비용 절감.

- **distributor**가 span을 받아 **Kafka 호환 버퍼**에 기록한다.
  Kafka가 확인한 뒤에야 쓰기 성공으로 친다(내구성 버퍼).
- **ingester가 둘로 갈렸다**: **LiveStore**(Kafka에서 최근 span을 소비, 최근 데이터 읽기 담당) + **block-builder**(장기 보관용 Parquet block을 만들어 object storage에).
- **compactor**는 **backend-scheduler/worker**로 대체됐다.
- 효과: **RF3 → RF1** — 데이터를 3벌이 아니라 1벌만 저장해 비용이 줄고, 읽기·쓰기를 따로 확장한다.

> 🔴 그래서 3.0 문서를 보면 "ingester/compactor가 대체됐다"고 나온다 — 2.x 사용자가 보는 컴포넌트와 다르니, 버전을 먼저 확인하자.

## 2.3 배포 모드

- **monolithic(single binary)**: 모든 컴포넌트 한 프로세스.
  소규모·개발.
- **microservices**: 컴포넌트별 독립.
  3.0의 Kafka·LiveStore·읽기/쓰기 분리는 microservices에서 적용된다.

마이그레이션 시에는 `tempo-cli`로 설정을 변환하고, 옛 block 포맷(vParquet3 이하)이면 먼저 vParquet4+로 올린다(3장).

## 2.4 정리

- **2.x**: distributor·ingester·compactor·querier·query-frontend.
- **3.0(Project Rhythm)**: Kafka 버퍼 + LiveStore(최근) + block-builder(장기) + backend-scheduler/worker, **RF3→RF1**.
- 배포는 monolithic·microservices(3.0 신구조는 microservices).
