여기까지 왔으면 키네시스가 **어떻게 동작하는지**는 안다.
샤드가 뭔지(3장), EFO가 왜 빠른지(7장), KCL이 lease로 뭘 하는지(8장)도 안다.
그런데 실무에서 키네시스를 다루다 보면 결국 이런 질문에 부딪힌다.

> "그래서 PutRecords 한 번에 몇 건까지 보낼 수 있지?", "iterator는 몇 분 만에 만료되지?", "KCL의 failover 시간 기본값이 몇 초였지?"

개념은 머리에 남지만 **숫자는 휘발된다.** 그래서 이 장은 시리즈 전체의 숫자와 설정을 한 곳에 모은 **레퍼런스 장**이다.
처음부터 끝까지 정독하는 장이라기보다, 북마크해 두고 필요할 때 펴 보는 장이다.
다만 표만 던지지는 않는다 — 각 표 앞뒤에 **"왜 이 숫자가 중요한가"**를 짧게 붙여서, 숫자가 개념과 연결되게 했다.

이 장의 모든 수치는 AWS 공식 문서와 API Reference에서 원문 그대로 확인한 값이다(2026-06 기준, KCL은 3.x, KPL은 최신 소스 기준).
공식 문서에서 확인 못 한 값은 솔직하게 "확인 못 함"이라고 적었다.

## 12.1 시작하기 전에 — 공식 문서끼리도 숫자가 다르다

레퍼런스 장의 첫 절이 "문서를 의심하라"인 게 이상해 보일 수 있다.
그런데 실제로 **같은 AWS 공식 문서 사이트 안에서 숫자가 서로 충돌하는 곳**이 있다.
키네시스 한도를 인용할 일이 있다면(블로그, 사내 위키, 면접 답변까지) 어느 페이지의 값인지 알고 말해야 한다.

대표적인 충돌 세 곳:

| 항목 | 한쪽 페이지 | 다른쪽 페이지 | 어느 쪽을 믿을까 |
|---|---|---|---|
| **레코드 최대 크기** | 한도 페이지·`PutRecord` API Reference: **10 MiB**(10485760 바이트) | 핵심 개념(key-concepts) 페이지: "up to **1 MB**" | **10 MiB.** API Reference의 길이 제약(Length Constraint)이 실제 검증 기준이고, 개념 페이지는 갱신이 늦은(stale) 것으로 판단 |
| **PutRecords 요청 한도** | API Reference: 레코드당 10 MiB·요청 전체 10 MiB | 프로듀서 개발 가이드 본문: 레코드당 1 MB·요청 전체 5 MB | API Reference 쪽이 우선으로 보이지만, 이를 못 박는 단일 권위 문장은 확인하지 못했다 — 단정하지 않는다 |
| **샤드 지속 쓰기 총량** | `PutRecord` 페이지: "10 MiB/sec" | 그 외 모든 페이지: **1 MB/s** | **1 MB/s**로 보는 게 합리적이다. PutRecord 페이지의 10 MiB/sec는 문서 오류로 보이지만, 이 역시 확정된 판단은 아니다 |

> 참고로 "레코드 10 MiB"가 언제부터 적용된 한도인지(발효일)는 공식 문서 페이지 본문에서 날짜를 확인하지 못했다. 비교적 최근에 올라간 한도라는 점만 알아두자.

왜 이 절이 먼저 오느냐 — 이 장에서 앞으로 나올 표들이 "어느 페이지 기준인지"를 명시하는 이유가 바로 이것이기 때문이다.
**레퍼런스를 다룰 땐 출처의 출처까지 본다.** 이것이 마스터 레벨의 습관이다.

## 12.2 서비스 한도 총표 — 샤드·스트림·계정

한도(quota)란 AWS가 "여기까지는 보장하고, 넘으면 거절한다"고 선을 그어둔 값이다.
비유하자면 **수도 계약**과 같다 — 계약 수압(1 MB/s)을 넘게 틀면 물이 안 나오는 게 아니라, "지금은 안 됩니다"라는 거절(throttling)이 돌아온다.
이 한도들의 *의미*와 hot shard 문제는 5장에서 충분히 다뤘으므로, 여기서는 숫자만 한 장에 모은다.

### 샤드 1개당 처리량 (provisioned, 고정 단위)

| 방향 | 한도 |
|---|---|
| **쓰기** | **1 MB/s** 또는 **1,000 records/s** — 둘 중 먼저 닿는 쪽 (partition key 크기 포함) |
| **읽기 (shared/GetRecords)** | **2 MB/s** 또는 **5 transactions/s** |
| **GetRecords 1회** | 최대 **10 MiB** · 최대 **10,000 records** |
| **10 MiB 쿨다운** | 한 번의 GetRecords가 10 MiB를 반환하면 → 이후 **5초** 안의 호출은 예외. throughput이 부족했던 직후 → **1초** 안의 호출은 `ProvisionedThroughputExceededException` |

쿨다운 규칙은 5장 §5.1과 7장 §7.3에서 본 그 규칙이다 — "한 번에 왕창 읽으면 잠시 쉬어야 한다"는 뜻이다.
컨슈머 폴링 루프를 직접 짤 때(7장) 이 5초/1초를 모르면 "왜 갑자기 예외가 터지지?"가 된다.

### 스트림·계정·레코드 한도

| 항목 | 값 |
|---|---|
| 샤드 기본 쿼터/계정 | 특정 리전 **20,000** / 그 외 리전 **1,000 또는 6,000** |
| 스트림 수 (provisioned) | 계정당 **상한 없음** |
| 스트림 수 (on-demand) | 기본 **50** (서포트 티켓으로 증설 가능, 상한 없음) |
| retention (보관 기간) | 기본 **24시간**, 범위 **24h ~ 8,760h(365일)** |
| partition key | 유니코드 최대 **256자** (→ MD5 해시 → 128-bit 정수) |
| 레코드 데이터 페이로드 | base64 인코딩 전 최대 **10 MiB** (1~10 MiB 구간은 burst capacity로 처리) |
| sequence number | 샤드 내 partition-key별 고유 · 쓰기 후 키네시스가 할당 · 같은 key는 시간순 증가 |

### On-demand 모드 처리량

| 항목 | 값 |
|---|---|
| 시작 용량 | 쓰기 **4 MB/s** · 읽기 **8 MB/s** |
| 최대 (us-east-1 · us-west-2 · eu-west-1) | 쓰기 **10 GB/s** · 읽기 **20 GB/s** |
| 최대 (그 외 리전) | 쓰기 **200 MB/s** · 읽기 **400 MB/s** (티켓으로 10/20 GB/s까지 증설 가능) |
| 용량 모드 전환 (provisioned ↔ on-demand) | 스트림당 **24시간에 2회** |
| EFO 등록 컨슈머 수/스트림 | on-demand **Advantage 50** / Standard·provisioned **20** |

여기서 기억할 한 줄: **provisioned의 모든 숫자는 "샤드 1개당"이고, on-demand는 "스트림 전체"다.** 샤드 10개면 쓰기 10 MB/s — 이 곱셈이 키네시스 용량 설계의 전부였다(5장).

## 12.3 데이터를 쓰는 API — PutRecord vs PutRecords

이제 API로 들어간다.
API란 키네시스에게 말을 거는 **정해진 문장 양식**이다 — 우체국 창구에 양식 없이 소포를 던질 수 없듯, 키네시스에도 정해진 형식으로 요청해야 한다.
키네시스의 API는 크게 둘로 나뉜다.

![](/assets/img/posts/kinesis-12-설정-api-한도-총정리/Pasted-image-20260611153410.png)

데이터가 다니는 길(data-plane)과 관리 명령이 다니는 길(control-plane)이 분리돼 있다 — 고속도로 본선과 관리사무소 진입로가 다른 것과 같다(비유).
이 절은 본선의 쓰기 쪽이다.

### 모든 API의 공통 입력 규칙

- 스트림 지정은 **StreamName**(1~128자, `[a-zA-Z0-9_.-]+`) 또는 **StreamARN**(1~2048자) 중 하나 — AWS는 **StreamARN 쪽을 권장**한다.
- 모든 API에 `StreamId`라는 필드가 보이는데, 공식 문서상 **"Not Implemented (Reserved for future use)"** — 미래를 위해 자리만 잡아둔 미구현 필드다.
  쓰지 말 것.

### PutRecord와 PutRecords 비교표

| 항목 | PutRecord (1건) | PutRecords (배치) |
|---|---|---|
| 호출 한도 | 샤드당 **1,000 TPS** | 호출당 최대 **500 records** |
| Data (본문 blob) | 0 ~ **10 MiB**, 필수 | 동일 (레코드당) |
| PartitionKey | 1~**256자**, 필수 | 동일 |
| ExplicitHashKey | 선택 — 해시를 건너뛰고 샤드 직접 지정 | 동일 |
| SequenceNumberForOrdering | 선택 — 같은 클라이언트·키의 순번 단조증가 보장 | **없음** |
| 응답 | `SequenceNumber` + `ShardId` + `EncryptionType` | `FailedRecordCount` + `Records[]` + `EncryptionType` |
| 부분 실패 | — | 성공 entry는 `SequenceNumber`+`ShardId`, 실패 entry는 `ErrorCode`+`ErrorMessage` |

> TPS는 "transactions per second", 초당 호출 횟수다.

이 표에서 시험에 나올(=장애에 나올) 부분은 세 가지다.

**첫째, PutRecords는 순서를 보장하지 않는다.** 배치 안의 한 레코드가 실패해도 뒤 레코드는 그대로 진행된다.
공식 문서의 처방은 명확하다 — *"쓴 순서대로 읽어야 하면 PutRecord로 같은 샤드에 써라."* 11장에서 본 Disney+의 증언과도 일치하는 부분이다.
순서가 필요한 데이터(예: 계좌 입출금)에 PutRecords를 쓰면 4장에서 공들여 이해한 순서 보장이 무너진다.

**둘째, 부분 실패는 호출 실패가 아니다.** PutRecords는 500건 중 30건이 실패해도 HTTP 응답 자체는 성공으로 돌아온다.
`FailedRecordCount`를 확인하고 실패 entry만 골라 재전송하는 건 **호출자의 책임**이다(재처리 패턴은 §12.10에서).

**셋째, 실패 entry의 `ErrorCode`는 단 두 값이다** — `ProvisionedThroughputExceededException`(한도 초과) 또는 `InternalFailure`(키네시스 내부 오류).
한도 초과일 때의 `ErrorMessage`에는 계정 ID·스트림 이름·샤드 ID가 들어 있어서, **어느 샤드가 뜨거운지**(hot shard, 5장)를 메시지만으로 짚을 수 있다.

쓰기 API가 던질 수 있는 예외: `ProvisionedThroughputExceededException`(400), `InvalidArgumentException`(400), `ResourceNotFoundException`(400), `AccessDeniedException`(400), `InternalFailureException`(500), 그리고 암호화 관련 KMS 계열 6종(모두 400).

## 12.4 데이터를 읽는 API — iterator·GetRecords·SubscribeToShard

읽기 모델의 *원리*는 7장에서 다뤘다(iterator라는 책갈피, 폴링 vs push).
여기서는 그 원리에 달린 **숫자들**을 모은다.
이 숫자들이 중요한 이유: 직접 컨슈머를 짜든 KCL을 쓰든, 타임아웃·재시도·지연의 디폴트 거동이 전부 이 숫자에서 나온다.

### GetShardIterator / GetRecords (shared 폴링 읽기)

| 항목 | 값 |
|---|---|
| GetShardIterator 호출 한도 | 계정·열린 샤드당 **5 TPS** (초과 → throughput 예외) |
| **iterator 만료** | 발급 후 **5분** → `ExpiredIteratorException` |
| ShardIteratorType | `AT_SEQUENCE_NUMBER` \| `AFTER_SEQUENCE_NUMBER` \| `TRIM_HORIZON` \| `LATEST` \| `AT_TIMESTAMP` |
| iterator 문자열 길이 | 1 ~ **512자** |
| 닫힌 샤드 | Split/Merge로 닫힌 샤드도 마지막 sequence number의 유효 iterator를 반환 |
| GetRecords `Limit` | 기본 **10,000**, 범위 1~10,000 (초과 시 `InvalidArgumentException`) |
| GetRecords 호출 한도 | 샤드당 **5 TPS**, 1회 최대 **10 MiB** |
| 공식 권장 | throttle을 만나면 GetRecords 사이 **1초 대기** — 1초를 넘겨도 예외가 날 수 있음 |

iterator 5분 만료를 책갈피 비유로 다시 보면 — 이 책갈피는 **5분짜리 모래시계가 달린 책갈피**다.
받아두고 5분 안에 안 쓰면 무효가 되고, 새로 발급받아야 한다.
폴링 루프가 어떤 이유로든(GC 멈춤, 네트워크 등) 5분 이상 멈췄다 재개하면 `ExpiredIteratorException`부터 만난다는 뜻이다.

GetRecords 응답에서 눈여겨볼 필드:

- `NextShardIterator` — **null이면 그 샤드는 닫혔고 끝까지 읽었다는 신호**다.
  리샤딩 후 부모 샤드를 다 읽었는지 판단하는 기준.
- `MillisBehindLatest` — 지금 읽은 지점이 스트림 끝에서 몇 ms 뒤처져 있는지.
  **0이면 따라잡은 것.**
- `ChildShards` — 샤드 끝에 도달했을 때만 채워지는, 자식 샤드 정보.
- 레코드 객체에는 `ApproximateArrivalTimestamp`(도착 시각), `Data`, `EncryptionType`, `PartitionKey`, `SequenceNumber`가 들어 있다.
  단 **ArrivalTimestamp는 단조 증가가 보장되지 않으므로 이걸로 정렬하면 안 된다**(7장에서 본 그 함정).

### SubscribeToShard (EFO push 읽기)

| 항목 | 값 |
|---|---|
| 전제 | `RegisterStreamConsumer`로 컨슈머 등록 후 |
| 필수 입력 | `ConsumerARN` + `ShardId` + `StartingPosition` |
| 호출 한도 | **등록 컨슈머·샤드 조합당 초당 1회** (예: 샤드 4,000개 + 컨슈머 2개 → 1초 안에 전부 구독 가능) |
| **구독 수명** | 최대 **5분** → 받은 `ContinuationSequenceNumber`로 재구독 |
| **5초 takeover 규칙** | 같은 (컨슈머, 샤드)에 **5초 이내** 재구독 → `ResourceInUseException` 거절. **5초 이후** → 두 번째 구독이 기존 연결을 **빼앗고** 기존 연결은 만료 |
| 이벤트 페이로드 | `Records[]`, `ContinuationSequenceNumber`, `MillisBehindLatest`, `ChildShards` |

여기도 5분이 나온다 — EFO 구독은 **5분짜리 회원권**이다(비유).
끊기는 게 아니라, 5분마다 `ContinuationSequenceNumber`(이어 읽을 지점)를 들고 재구독하는 게 정상 동작이다.
KCL을 쓰면 이 재구독을 알아서 해준다.

"5초 takeover"는 장애 복구를 위한 설계다.
컨슈머 A가 죽고 같은 이름의 컨슈머 A'가 떴을 때, 5초만 지나면 A'가 구독을 가져갈 수 있다.
거꾸로 5초 안의 중복 구독은 거절되므로, 실수로 같은 컨슈머를 두 번 띄워도 즉시 둘 다 읽는 사고는 막힌다.

또 하나, **EFO는 샤드당 2 MiB/s를 전용으로 받는다 — 컨슈머가 몇 개든 각자.** shared 모드의 "2 MB/s를 모두가 나눠 쓰기"와의 차이가 6장의 핵심이었다.

## 12.5 Control-plane API — 관리 명령의 TPS와 제약

control-plane API들의 호출 한도는 data-plane보다 훨씬 빡빡하다.
왜냐 — 데이터는 초당 수천 건 흘러도, "스트림을 만들어줘" 같은 관리 명령이 초당 수천 번 필요할 일은 없기 때문이다.
그런데 이 빡빡함을 모르면 **자동화 스크립트가 스스로 throttle을 맞는** 사고가 난다(예: 모니터링이 매초 DescribeStream을 때리는 경우).

| API | 호출 한도 | 핵심 제약 |
|---|---|---|
| CreateStream | 5 TPS / 계정 | CREATING 상태 스트림 5개 초과 또는 샤드 권한 초과 → `LimitExceededException` |
| DeleteStream | 5 TPS / 계정 | |
| DescribeStream | 10 TPS / 계정 | |
| **DescribeStreamSummary** | **20 TPS / 계정** | 샤드 목록 없는 요약(`OpenShardCount`, `ConsumerCount` 등) — 모니터링엔 이쪽 |
| DescribeStreamConsumer | 20 TPS / 스트림 | `ConsumerStatus` = CREATING/DELETING/ACTIVE |
| **ListShards** | **1,000 TPS / 스트림** | `MaxResults` 기본 1,000 · `NextToken`은 **300초** 만료 |
| ListStreams | 5 TPS / 계정 | |
| RegisterStreamConsumer | 5 TPS / 스트림 | EFO 컨슈머 스트림당 **20(Advantage 50)** · 동시 CREATING **5개**까지 |
| MergeShards / SplitShard | 각 5 TPS / 스트림 | **provisioned 전용** |
| UpdateShardCount | 10 TPS(아래 제약 참고) / 스트림 | **provisioned 전용**(on-demand에서 호출 시 `ValidationException`) · `UNIFORM_SCALING`만 |
| UpdateStreamMode | / 스트림 | 모드 전환 **24시간에 2회** |
| Increase/DecreaseStreamRetentionPeriod | 각 5 TPS / 스트림 | 최대 8,760h / 최소 24h |
| Start/StopStreamEncryption | / 스트림 | 24시간 롤링당 각 **25회** |
| AddTagsToStream | 5 TPS / 계정 | 스트림당 태그 **50개** |
| DescribeLimits | 1 TPS / 계정 | |

표에서 실무 팁 둘만 뽑으면 — **스트림 상태 조회는 DescribeStream(10 TPS)보다 DescribeStreamSummary(20 TPS)**, **샤드 목록은 ListShards(1,000 TPS)**가 여유롭다.
AWS가 한도를 다르게 둔 것 자체가 "이쪽을 쓰라"는 신호다.

### UpdateShardCount의 기본 스케일 제약 — 전부 외울 가치가 있다

UpdateShardCount(샤드 수를 한 번에 바꾸는 API)는 기본 설정에서 다음을 **할 수 없다**:

- 24시간 롤링 기준 **10회 초과** 스케일 변경
- 현재 샤드 수의 **2배 초과** 증설 / **절반 미만** 감축
- **10,000 샤드 초과** — 이 API의 기본 제약.
  §12.2의 계정당 샤드 쿼터(특정 리전 20,000 등)와는 **별개 한도**다
- 이미 10,000을 넘는 스트림을, 결과가 10,000 이상이 되도록 감축
- 계정 샤드 한도 초과
- **10 TPS 초과** 호출 (→ `LimitExceededException`)

그리고 공식 권장: **target은 현재의 25% 배수(25/50/75/100%)로.** 배수가 아니면 잠깐 생겼다 사라지는 단명 샤드가 생겨 작업이 느려지고, 그 단명 샤드도 **총 샤드 한도에 가산**된다.

"2배/절반" 제약이 왜 중요하냐면 — 트래픽이 갑자기 4배가 되면 **UpdateShardCount 두 번**(2배 → 또 2배)이 필요하다는 뜻이고, 그래서 11장에서 본 Disney+처럼 이 제약을 우회하는 커스텀 오토스케일러를 만드는 회사가 나온다.
한도는 그냥 숫자가 아니라 **아키텍처 결정을 강제하는 벽**이다.

## 12.6 KCL 3.x 설정 기본값 — 디폴트가 곧 거동이다

KCL(8장)의 동작 — lease 뺏기, 죽음 판정, checkpoint — 은 전부 설정값의 함수다.
**디폴트를 모르면 "KCL이 왜 10초나 걸려 복구하지?"라는 질문에 답할 수 없다**(답: failover 판정 기본값이 10초라서).
아래 값들은 KCL 공식 설정 문서 기준이고, 일부는 공식 문서 표에 없어 소스 코드에서 확인한 값이다(해당 항목에 "(소스 기준)" 표기).

### Lease 관리 (lease = 샤드 점유권, 8장)

| 설정 | 기본값 | 의미 |
|---|---|---|
| `failoverTimeMillis` | **10,000** (10초) | lease 갱신이 없을 때 "죽었다" 판정까지의 시간 |
| `shardSyncIntervalMillis` | **60,000** (60초) | 샤드 목록 동기화 주기 |
| `cleanupLeasesUponShardCompletion` | **true** | 자식 샤드 처리 시작 시 부모 lease 제거 |
| `maxLeasesForWorker` | **Integer.MAX_VALUE** (사실상 무제한) | 워커 하나가 가질 수 있는 lease 수 |
| `maxLeaseRenewalThreads` | **20** | lease 갱신 스레드 풀 크기 |
| `billingMode` | **PAY_PER_REQUEST** | lease 테이블(DynamoDB) 과금 방식 — 기본이 on-demand 과금 |
| `initialLeaseTableRead/WriteCapacity` | **10 / 10** | (provisioned 테이블로 만들 때만) |
| `initialPositionInStreamExtended` | **TRIM_HORIZON** | 최초 lease 생성 시 읽기 시작 위치 — **기본이 "맨 처음부터"** |
| `isGracefulLeaseHandoffEnabled` | **true** (3.x 신규) | lease 이관 전 처리 마무리 시간을 줌 |
| `gracefulLeaseHandoffTimeoutMillis` | **30,000** (30초, 3.x 신규) | 강제 이관 전 대기 |
| `leaseAssignmentIntervalMillis` | **20,000** (= failover의 2배, 소스 기준) | lease 배정 실행 주기 |

세 가지를 짚는다.

1. **`initialPositionInStreamExtended`의 기본이 TRIM_HORIZON** — 새 KCL 앱을 retention 7일짜리 스트림에 붙이면 **7일치 과거를 전부 읽는 것부터 시작**한다.
   "최신만 보면 되는데 왜 이렇게 느리지?"의 단골 원인.
2. **graceful handoff(3.x 신규)** — lease를 넘기기 전 30초의 마무리 시간을 주되, 앱이 `shutdownRequested()`에서 checkpoint를 해줘야 의미가 있다.
3. **3.x에서 폐기된 설정**: `maxLeasesToStealAtOneTime`, `enablePriorityLeaseAssignment` — 설정해도 무시된다(2.x→3.x 마이그레이션 호환 목적으로만 존재).

### 3.x의 워커 부하 균등 분배 (WorkerUtilizationAwareAssignment)

KCL 3.x는 워커별 CPU 사용률을 보고 lease를 재배치한다(8장).
그 민감도가 아래 값들이다.

| 설정 | 기본값 | 의미 |
|---|---|---|
| `reBalanceThresholdPercentage` | **10** | 재배치를 트리거하는 불균형 임계(%) |
| `dampeningPercentage` | **60** | 과부하 워커에서 한 번에 옮기는 비율 댐핑 |
| `allowThroughputOvershoot` | **true** | 약간 넘치더라도 lease 회수 허용 |
| `disableWorkerMetrics` | **false** | true면 CPU 메트릭 무시(throughput 균등 분배로 후퇴) |
| `maxThroughputPerHostKBps` | **Double.MAX_VALUE** (무제한) | 워커당 최대 할당 throughput |
| `workerMetricsReporterFreqInMillis` | **30,000** (30초, 소스 기준) | 워커 메트릭 보고 주기 |
| `workerMetricsEMAAlpha` | **0.5** (소스 기준) | 메트릭 지수이동평균 평활 계수 |

### 읽기(Retrieval) / 처리·체크포인트·메트릭

| 설정 | 기본값 | 의미 |
|---|---|---|
| 기본 retrieval 모드 | **FanOut (EFO)** (소스 기준) | ConfigsBuilder에서 POLLING을 명시하지 않으면 **기본이 EFO** |
| `listShardsBackoffTimeInMillis` | **1,500** | ListShards 실패 간 대기 |
| `maxListShardsRetryAttempts` | **50** | ListShards 재시도 한도 |
| `PollingConfig.maxRecords` | **10,000** | GetRecords 호출당 최대 (= API 한도와 동일) |
| `PollingConfig.idleTimeBetweenReadsInMillis` | **1,500** (하한 200) | 폴링 사이 대기 |
| `PollingConfig.kinesisRequestTimeout` | **30초** (소스 기준) | 폴링 요청 타임아웃 |
| `callProcessRecordsEvenForEmptyRecordList` | **false** | 빈 응답에도 프로세서를 부를지 |
| `parentShardPollIntervalMillis` | **10,000** | 부모 샤드 완료 확인 주기 (리샤딩 시 순서 강제) |
| `shardPrioritization` | **NoOpShardPrioritization** | 샤드 우선순위 전략 없음 |
| `clientVersionConfig` | **CLIENT_VERSION_CONFIG_3X** | 마이그레이션 중엔 `..._COMPATIBLE_WITH_2X` |
| `taskBackoffTimeMillis` | **500** | 실패한 task 재시도 대기 |
| `metricsBufferTimeMillis` / `metricsMaxQueueSize` | **10,000 / 10,000** | CloudWatch 발행 전 버퍼 |
| `metricsLevel` | **DETAILED** | NONE/SUMMARY/DETAILED — **기본이 가장 비싼 DETAILED** |
| `checkpointFactory` | **DynamoDBCheckpointFactory** (소스 기준) | checkpoint 저장소 = DynamoDB |

기본 retrieval이 **EFO**라는 점은 비용과 직결된다 — EFO는 추가 과금이 있는 모드다(10장).
"그냥 기본으로 띄웠더니 EFO 요금이 나온다"가 가능하다는 뜻이다.
또 `metricsLevel` 기본이 DETAILED라는 것도 같은 결의 이야기다.
**KCL의 디폴트는 '성능·관측 우선'이지 '비용 우선'이 아니다.**

KCL 3.x는 DynamoDB 테이블을 **3개** 쓴다(lease 테이블 + 워커 메트릭 + coordinator 상태) — 내부 구조는 8장 참고.

## 12.7 KPL 설정 기본값 — aggregation과 collection

KPL(Kinesis Producer Library)은 프로듀서 쪽 도우미 라이브러리다.
핵심 아이디어는 **"작은 레코드를 묶어서 보낸다"** — 두 단계로.

![](/assets/img/posts/kinesis-12-설정-api-한도-총정리/Pasted-image-20260611153433.png)

왜 묶나 — 샤드 쓰기 한도가 "1 MB/s **또는** 1,000 records/s"라서, 레코드가 작으면(1 KB 미만) 용량보다 **건수(RPS) 한도가 먼저 막힌다.** 공식 문서의 예가 위 그림 그대로다: 512 B짜리 1,000개를 그대로 보내면 RPS 1,000으로 한도 꽉, 50 KB 레코드 10개로 묶으면 RPS 10.
같은 데이터량인데 한도 소모는 1/100이다.

여기서 aggregation으로 묶인 레코드 안의 각 user record를 컨슈머가 다시 풀 때(de-aggregation) 개별 식별자가 필요한데, 소스 코드 기준으로는 `subSequenceNumber`라는 필드가 그 역할을 한다 — 다만 공식 KPL 개념 문서에는 이 단어가 등장하지 않아, 명칭 수준에서는 소스로만 확인된 내용이다(8장 ExtendedSequenceNumber 참고).

### 주요 기본값 (KPL `default_config`, 소스 기준 — docs로 교차된 항목은 본문에 명시)

| 설정 | 기본값 | 의미 |
|---|---|---|
| `RecordMaxBufferedTime` | **100 ms** | 버퍼에 머무는 최대 시간 — 클수록 잘 묶이지만 지연 증가 |
| `RecordTtl` | **30,000 ms** | 이 시간 안에 전송 못 하면 실패 처리 |
| `AggregationEnabled` | **true** | 1단계 on (공식 문서도 "둘 다 on"이 기본이라 명시) |
| `AggregationMaxCount` | **4,294,967,295** | 묶음당 user record 수 — 사실상 무제한, 실제 한계는 크기 |
| `AggregationMaxSize` | **51,200 B (50 KB)** | 묶음 레코드 최대 크기 — 1 MB보다 작게 둬서 collection 여지를 남김 |
| `CollectionMaxCount` | **500** | PutRecords 1회 record 수 (= API 한도) |
| `CollectionMaxSize` | **5,242,880 B (5 MB)** | PutRecords 요청 총 바이트 |
| `MaxConnections` / `MinConnections` | **24 / 1** | 백엔드 동시 연결 |
| `RequestTimeout` / `ConnectTimeout` | **6,000 / 6,000 ms** | HTTP·연결 타임아웃 |
| `RateLimit` | **150 (%)** | 샤드 한도의 150%까지 순간 초과 허용(버스트 흡수) |
| `MaxUserRecordSize` | **1,048,576 B (1 MB)** | 단일 user record 최대 |
| `ThreadingModel` | **PER_REQUEST** | (대안: POOLED) |
| `FailIfThrottled` | **false** | true면 throttle 시 재시도 없이 즉시 실패 |
| `MetricsLevel` / `MetricsGranularity` | **detailed / shard** | KPL → CloudWatch 메트릭 |
| `MetricsNamespace` / `MetricsUploadDelay` | **KinesisProducerLibrary / 60,000 ms** | |
| `LogLevel` | **info** | |

해설 한 단락 — KPL 튜닝은 결국 **`RecordMaxBufferedTime` 하나의 트레이드오프**로 수렴한다.
100 ms를 키우면 더 잘 묶여 한도 효율이 오르지만, 그만큼 레코드가 키네시스에 늦게 도착한다(지연 +최대 그 값만큼).
"실시간"의 기준이 초 단위인 서비스라면 키워도 되고, 수십 ms가 중요하면 못 키운다.
그리고 `RateLimit` 150%는 "잠깐의 버스트는 봐준다"는 뜻이지 지속 초과 허용이 아니다 — 지속 트래픽이 한도를 넘으면 결국 §12.10의 throttle로 돌아온다.

## 12.8 CloudWatch 지표 — 무엇을 보고 무엇에 알람 거나

모니터링 *전략*은 10장에서 다뤘다.
여기서는 지표 이름과 단위의 레퍼런스다.
기본 사실: 네임스페이스 **`AWS/Kinesis`**, **1분 주기** 수집, **2주** 보관.

### Basic (stream-level, 자동·무료, 차원 `StreamName`)

| 지표 | 단위 | 의미 |
|---|---|---|
| **`GetRecords.IteratorAgeMilliseconds`** | ms | 🔑 마지막으로 읽은 레코드의 나이. 0이면 따라잡음. **가장 중요한 지표** |
| `GetRecords.Bytes` / `.Records` | Bytes / Count | 읽은 양/건수 |
| `GetRecords.Latency` / `.Success` | ms / Count | 호출 응답시간 / 성공 수 |
| `IncomingBytes` / `IncomingRecords` | Bytes / Count | 쓰기 성공량 (Put 합산) |
| `PutRecord.Bytes/.Latency/.Success` | — | 단건 쓰기 (Success의 Average = 성공률) |
| `PutRecords.Bytes/.Latency/.Success` | — | 배치 쓰기 (Success = 1건 이상 성공한 호출) |
| `PutRecords.TotalRecords/.SuccessfulRecords/.FailedRecords/.ThrottledRecords` | Count | 배치 내 총/성공/내부실패/throttle 거부 |
| **`ReadProvisionedThroughputExceeded`** | Count | 읽기 throttle (Average 권장) |
| **`WriteProvisionedThroughputExceeded`** | Count | 쓰기 throttle (Average 권장) |
| `SubscribeToShard.RateExceeded/.Success` | — | EFO 구독 실패/성공 (차원 +`ConsumerName`) |
| `SubscribeToShardEvent.Bytes/.Records/.MillisBehindLatest/.Success` | — | EFO 이벤트 (차원 +`ConsumerName`) |

> deprecated 주의: `GetRecords.IteratorAge`(→ ms 버전으로 대체), `PutRecords.Records`(→ `.SuccessfulRecords`). 옛 블로그 글이나 오래된 대시보드 템플릿엔 아직 옛 이름이 남아 있다.

### Enhanced (shard-level, opt-in·유료, 차원 +`ShardId`)

`EnableEnhancedMonitoring`으로 켜면 `IncomingBytes`, `IncomingRecords`, `IteratorAgeMilliseconds`, `OutgoingBytes`, `OutgoingRecords`, Read/Write `ProvisionedThroughputExceeded`가 **샤드별로** 나온다 — hot shard를 찍어내는 유일한 방법이다.

단, 비용 구조를 알고 켜야 한다: 차원당 메트릭 1개 ≈ **월 43,200 PutMetricData 호출**에 해당하고, **샤드 수 × 메트릭 수 × 월** 단위로 과금된다.
샤드가 수백 개면 요금이 곱셈으로 불어난다.
그래서 실무 관행은 **평상시 끄고, 문제가 보일 때 한시적으로 켜는 것**(10장).

### 권장 알람 — 두 개만 기억

1. **`IteratorAgeMilliseconds`는 Maximum 통계로 알람.** Average로 걸면 멀쩡한 샤드들이 hot shard 하나를 평균 속에 숨겨버린다.
   그리고 age가 **retention의 50%를 넘으면 데이터 손실 위험 구간** — 컨슈머가 따라잡기 전에 오래된 데이터가 retention 만료로 지워질 수 있다는 뜻이다.
2. **`PutRecord.Success` / `PutRecords.Success`는 Average 알람.** KPL을 쓰면 PutRecords 쪽을 본다(§12.7 — KPL은 PutRecords로 보내니까).

## 12.9 보안 설정 레퍼런스 — IAM·KMS·VPC

보안 3계층(누가 호출하나 / 데이터 암호화 / 네트워크 경로)의 개념은 10장에서 다뤘다.
여기는 정책을 실제로 쓸 때 필요한 식별자·조건 키 모음이다.

### IAM (액션 접두사 `kinesis:`)

- **stream-level 액션** — Resource는 `arn:aws:kinesis:{region}:{account}:stream/{name}`: `PutRecord`, `PutRecords`, `GetRecords`, `GetShardIterator`, `DescribeStreamSummary`, `ListShards`, `CreateStream` 등.
- **consumer-level 액션** — Resource는 `.../stream/{name}/consumer/{name}:{ts}`: `SubscribeToShard`, `DescribeStreamConsumer` (EFO 전용 리소스가 따로 있다는 점이 포인트).
- `ListStreams`는 **account-level** — Resource를 `*`로만 쓸 수 있고 특정 스트림으로 한정할 수 없다.
- **resource policy**(스트림 쪽에 붙이는 정책): `PutResourcePolicy` / `GetResourcePolicy` / `DeleteResourcePolicy`.
  제약 — 와일드카드 principal·서비스/연합 principal·canonical user **미지원**, 정책 최대 **20 KB**.
- **KCL은 Kinesis 권한만으로 부족하다** — lease/checkpoint용 **DynamoDB** + 메트릭용 **CloudWatch** 권한이 필수(§12.6에서 본 그 테이블 3개 때문).
  "KCL 앱이 권한 에러로 안 뜬다"의 90%가 여기다.
- EFO를 교차 계정으로 공유하려면 **stream ARN과 consumer ARN 양쪽**에 정책이 필요하다.

### KMS 서버측 암호화 (SSE)

- 쓰기 전 암호화, 읽기 후 복호화 — **클라이언트 코드 변경 0**(투명).
  단 **소급 적용은 안 된다**: 켜기 전에 들어간 데이터는 평문으로 남는다.
- 사용자 관리 키(CMK) 사용 시 필요한 권한: 프로듀서 `kms:GenerateDataKey`, 컨슈머 `kms:Decrypt`(Flink·Lambda 실행 역할에도), 관리자 `kms:List*`·`kms:DescribeKey*`.
  AWS 관리 기본 키면 커스텀 KMS 권한이 불필요하지만, **교차 계정 공유에는 CMK가 필수**다.
- encryptionContext는 `{"aws:kinesis:arn": "<stream ARN>"}` — CloudTrail에 평문으로 남고, 조건 키 `kms:EncryptionContext:aws:kinesis:arn`으로 "이 키는 이 스트림에만" 식의 한정이 가능하다.

### VPC endpoint (interface형, PrivateLink)

- VPC ↔ 키네시스 트래픽을 AWS 네트워크 내부로만 흐르게 한다(인터넷 게이트웨이/NAT/VPN 불필요).
  스트림·프로듀서·컨슈머 쪽 설정 변경도 불필요.
- 조건 키 **`aws:sourceVpce`** 로 "이 endpoint를 거치지 않은 호출은 Deny" 정책을 만들 수 있다 — 예: `"StringNotEquals": {"aws:sourceVpce": "vpce-..."}` → Deny.

## 12.10 에러·throttling·재시도 — 그리고 키네시스의 경계선

### `ProvisionedThroughputExceededException`이 터지는 조건 전부

이 시리즈 내내 등장한 PTEE의 트리거를 한 줄로 모으면:

- 쓰기: 샤드당 1 MB/s 또는 1,000 records/s 초과
- 읽기: 샤드당 2 MB/s 또는 5 TPS 초과
- GetRecords가 10 MiB 반환 후 **5초** 이내 재호출 / throughput 부족 직후 **1초** 이내 재호출
- GetShardIterator 과다 호출 (계정·샤드당 5 TPS 초과)

### 공통 에러 코드

`ThrottlingException`(400, SDK가 자동 재시도), `InternalFailure`(500), `ServiceUnavailable`(503), `AccessDeniedException`(403), `ValidationError`(400), `ExpiredTokenException`(403), `RequestTimeoutException`(408).

재시도 전략에 대해 한 가지 솔직하게 적어둘 것 — **구체적인 backoff 수치(초기 지연·배수·jitter)는 키네시스 문서에 없다.** AWS SDK 공통의 "Retry behavior" 가이드로 위임돼 있고, 키네시스 전용 수치는 공개 문서에서 확인하지 못했다.
"키네시스 권장 backoff는 X ms"라고 단정하는 글을 보면 출처를 의심하자.

### PutRecords 부분 실패 재처리 — 표준 루프

§12.3에서 본 부분 실패의 재처리는 패턴이 정해져 있다: 응답의 `FailedRecordCount > 0`이면, `ErrorCode != null`인 entry만 골라 다음 요청에 재제출하는 루프를 돈다.
응답 배열은 **항상 요청과 같은 개수·같은 순서(natural ordering)** 로 오기 때문에, 인덱스로 요청-응답을 짝지을 수 있다 — 이 보장이 없으면 어떤 레코드가 실패했는지 알 길이 없다.

재시도는 곧 중복 가능성이다(9장).
공식 문서가 안내하는 멱등 처리 패턴 세 가지: ① 레코드에 primary key를 심어 최종 저장소에서 중복 제거, ② 목적지가 중복을 흡수(OpenSearch의 versioning + unique ID), ③ S3에 결정적 파일명(`prefix-shardid-FirstSeqNum`)으로 쓰고 마지막 sequence number를 checkpoint.

### 경계선 — Firehose와 Managed Flink는 다른 물건이다

레퍼런스 장을 닫기 전에, 이름이 비슷해서 혼동되는 두 서비스와의 경계를 그어두자(2장의 제품군 지도 복습).

- **Amazon Data Firehose** — 완전관리형 **"전송"** 서비스.
  목적지(S3/Redshift/OpenSearch/Splunk/Iceberg/HTTP 등)로 **버퍼링 후 배달**해주는 택배 서비스다(비유).
  레코드 최대 **1,000 KB**이고, **샤드도 replay(다시 읽기) 개념도 없다.** 이 시리즈에서 배운 샤드·iterator·EFO가 전부 해당 없다는 뜻이다.
  대신 KDS를 **소스로** 연결할 수 있다(KDS → Firehose → 목적지).
- **Managed Service for Apache Flink** — 스트림 **"처리·분석"** 서비스.
  Java/Scala/Python/SQL로 스트림 위에서 연산을 돌리고, 프로비저닝·AZ failover·오토스케일·checkpoint를 AWS가 관리한다.
  KDS를 source/sink로 쓴다.

즉 **KDS = 통로, Firehose = 통로에서 창고로의 배달, Flink = 통로 위의 계산기**다(비유).
"Kinesis 쓴다"고 말할 때 셋 중 무엇인지 구분하는 것 — 이것도 마스터 레벨의 기본기다.
